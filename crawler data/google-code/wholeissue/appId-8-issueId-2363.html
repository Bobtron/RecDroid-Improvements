I guess I don't see the issue with a fake signature permission: It is just another permission that the app would need to ask for. mar-v-in Dec 16, 2016

@robsee to restrict it to apps that are signed with a key only the rom builder holds, you can easily change the permission protection level to signature, if you want to restrict it to apps residing in the /system/priv-app you can set protection level to system (or privileged in newer android versions). The relevant line in the patch is this one. It's really that simple, you don't have to consider something else (the patch is a small one anyway)
I doubt that any user would want that (because it means that they can't simply install the microG build from microg.org) and it would also render your users unable to contribute to microG because they won't be able to test their own compilations. But if you want to do it for yourself and have no problem with compiling microG yourself every now and then, this might be useful nonetheless.

mar-v-in Dec 17, 2016

Member

If an app can access data by using signature spoofing, this is only because another third-party app uses an unusual way to protect data. Using signatures directly is not intended by AOSP, as mentioned in the developer reference and also a warning is shown while compiling the app if it does. A bad app can not use the private data storage of the genuine app itself, only data stored by a third party for the genuine app might become accessible.

You might want to read this blog post for more details.
