emembrives Jan 16, 2016

I looked into it a bit.

  • db in withTransaction (inside model/src/main/java/de/azapps/mirakel/model/provider/MirakelInternalContentProvider.java) is null because getWritableDatabase() returns null.
  • getWritableDatabase() returns null because dbHelper is null (database is also null, but would get filled-in if dbHelper was fine). This is suggesting that onCreate is not called.
  • DatabaseHelper.getDatabaseHelper() needs a Context, which is unavailable in the getWritableDatabase() static method. However, as DatabaseHelper is a singleton, the Context is only needed to create the object the first time. I added a method to get an existing DatabaseHelper to the DatabaseHelper class. It turns out that there is already existing DatabaseHelper, so we can get one and create the database object.
  • However, the code now fails to acquire a lock on the database, so syncing still fails.

Now entering the realm of speculation:

  • In the Mirakel app, it seems that DatabaseHelper is created twice: one in de.azapps.mirakelandroid in MirakelContentProvider.java and one in de.azapps.mirakelandroid:sync in Mirakel.java.
  • It would seem that these are two different processes used by the app.
  • However, my understanding is that if one process accesses the database and locks it, the other won't be able to access it. Hence the issue.

weiznich Jan 16, 2016

Member

@emembrives You are near the real problem.
In android a sync adapter and a process are running in two different processes. This would be no problem if database transactions are build with the help of sqlite. But they are implemented on top of sqlite, so there is no way to synchronize this across multiple processes.
As I understood this at the beginning of the year a started to rewrite how transactions are handled in mirakel. This means rewriting some of the central model code of mirakel. To verify that there is no new bug introduced by the rewrite, a decide to fix the tests first.
Because of this not all test are executed, so I could not verify everything is working again.
For anyone is interested in this: I've uploaded the current version of the patch, but be warned that there may some new bugs in mirakel itself.

Looks like I just re-traced @emembrives's steps (wasn't aware of this ticket).
While the big fix is in the works, the patch that fixed the crash on sync for me is the following three lines. Perhaps this introduces the possibility of data loss if the update sequence that was supposed to be in the transaction gets interrupted, but it seems to at least work.

diff --git a/model/src/main/java/de/azapps/mirakel/model/provider/MirakelInternalContentProvider.java b/model/src/main/java/de/azapps/mirakel/model/provider/MirakelInternalContentProvider.java
index 3b99272..5de7de9 100644
--- a/model/src/main/java/de/azapps/mirakel/model/provider/MirakelInternalContentProvider.java
+++ b/model/src/main/java/de/azapps/mirakel/model/provider/MirakelInternalContentProvider.java
@@ -449,6 +452,11 @@ public class MirakelInternalContentProvider extends ContentProvider implements
     public static void withTransaction(final Context ctx, final DBTransaction what) {
         if (what != null) {
             final SQLiteDatabase db = getWritableDatabase();
+            if (db == null) {
+                what.exec();
+                return;
+            }
+
             if (!db.inTransaction()) {
                 db.beginTransaction();
                 try {

Note that calling DatabaseHelper.getDatabaseHelper from withTransaction (even having passed a context object to it from all call sites), does not work due to a deadlock (probably the same one that was mentioned earlier in the ticket).
