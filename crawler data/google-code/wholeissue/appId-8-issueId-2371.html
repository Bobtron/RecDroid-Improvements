voidstarstar Jan 30, 2018

This seems to be the same root cause as #444 and I suspect it affects many more apps too. If this bug also affects an official Google app, it seems logical to me to either ignore the permission (by passing null to sendOrderedBroadcast as @ghost mentioned) or loosen the permission naming requirements.

From the documentation, the manifest of an app should include:

An applicationPackage + ".permission.C2D_MESSAGE" permission to prevent other Android applications from registering and receiving the Android application's messages. The permission name must exactly match this pattern—otherwise the Android application will not receive the messages.

@mar-v-in

We could work-around this, but I wonder why this works with original Play Services

In the original Play Services, it might only be required that each app have a unique prefix to the permission name, not that it matches this format exactly. If true, then they probably just tell developers that the package name is required since it's the most obvious naming convention to prevent a collision. Since apps that do not follow the documentation still work with the official implementation, it could be Google's way of adhering to the robustness principle by being liberal in what they accept. I would suggest that microG does the same.

Would it be possible to just look for a permission that matches the pattern *.permission.C2D_MESSAGE? For example, when an app registers, microG can look for this pattern and save the permission name. Later when a message is received, it could be used in place of msg.category here. I'm not very familiar with the microG code, so this probably isn't the most elegant way, but hopefully the basic idea is clear.
