mar-v-in Mar 3, 2017

Member

I won't say that microG is more or less secure than original GAPPS in general. It depends a lot on how you define security (there is no proper defintion - e.g. some people say that having the possibility to turn off a certain security system already is a security issue, others say it's just important that it is only possible with user consent).

In some portions of microG we had to open "backdoors" for Google apps because Google Play Services has them as well. For example, some Google apps are able to read your contact list even though they don't have the contacts permission. I would call this backdoor a security issue, but Play Services has it as well (in microG it's just more visible).

It might also be interesting to know that Google is not granting rewards via their android bug bounty program for reporting security issues in Play Services as long as it does not extend to their servers (e.g. unauthorized access to data from your Google account).

Regarding signature spoofing: It's not a technical problem to further restrict the signature spoofing permission.

The patch I built is designed for people that are aware of permissions and thus decided that it should be up to the user to decide if they want to grant this permission. I also see low risk of misuses of this feature, given that apps should normally not use this signature directly (and in fact developers see a warning in android studio and during compilation if they do nonetheless). I thus think that the patch is a good compromise on giving the user powers that they should have nonetheless (decide to replace apps with other apps) and possible security issues.

However, if you want to further restrict the permission, there are several options:

  1. Restrict the permission to apps in /system/priv-app. This requires to ship microG or two placeholder apps with the ROM (or to allow modification of /system, which is a far worse option). Modifying the permission to be for system apps only is changing one word in the patch (from "dangerous" to "system"), creating a placeholder apps is about writing ~10 lines of XML code and thus could be done in a few minutes.
  2. Only activate the permission after further consent, e.g. a hidden checkbox in the developer settings or similar locations.This was already realized by crDroid. It is slightly more work the option 1 as it involves creating new UI code, but an experienced developer should be able to do this within less than 10 minutes.
  3. Restrict the permission to only be used to a) spoof Google's signature and/or b) by the two apps of microG requiring it. This would be a work of adding one if statement for each of these restrictions to the patch.

All of this is easy to realize, especially for a android os developer that deals with the security of the system. I already announced that I'd willing to help developing such restrictions if they will be used in the end.

@ale5000-git CopperheadOS verifies the /system partition to be unchanged, so all signature spoofing patch tools/mods we currently have won't work on CopperheadOS.

@cswag CopperheadOS devs made a clear statement they are against embedding microG or adding specific features for it. I think that the responsible devs are well aware of everything I described above, because they know a lot about the OS.
Remember that CopperheadOS is a commercial product and the devs have good reasons to not do anything that could be seen as an action against Google, because this could cost them certification required for a CopperheadOS based system that ships with GAPPS (which they mentioned in the past could be a requirement for some of their business partners).
If @mikeperry-tor or anyone else seeks for help regarding microG integration/support, I am happy to help out. But please excuse me that i will no longer actively try to convince people it is a good idea or that it is not a huge security issue - it took me a lot of wasted time in the past that would be better used in actually developing microG.

mar-v-in Jul 22, 2017

Member

@unlmtd I think matrix is not designed to be a push system, but actually promotes using third-party push systems (including GCM). And my short experience with matrix/riot when not using GCM was that it ate my battery in no time.

I once discussed with @iNPUTmice (dev of the popular XMPP client Conversations) the possibilities to use XMPP and the Conversations client for push messaging. GCM is actually based on XMPP and Conversations has proven to deliver messages on time without GCM and without consuming the battery. The idea is to do this in a GCM-compatible fashion, so that app developers can use the same API for XMPP-based push messaging and GCM push messaging. But this is all still in the beginnings, so don't expect it to be there in the next months (not even this year).
