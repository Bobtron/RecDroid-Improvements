Comment #2

Posted on Feb 15, 2012 by Quick Hippo

It's usually due to a misconfiguration or to the fact the sip provider refuse to dial this way.

If you can collect logs (see HowToCollectLogs wiki page). I can tell exactly what the sip provider returns as error when you try to make the call.

The case that are often encountered are : * list of codec does not match any of the codec supported by the sip provider (usually in this case you can't receive call either) * The dialing plan of the sip provider is different than the normal dialing plan you use in your country. Many sip provider requires you to add some prefix or to have complete phone number. Few support + sign in the sip uri too. Usually you can fix that by dialing what is expected by the sip provider + csipsimple provides a feature to automatically rewrite phone numbers when it comes from stock contacts / dialer app. (when you dial from within csipsimple it's assumed that you force the call to a specific number so rewriting rules does not apply).

Comment #3

Posted on Feb 15, 2012 by Quick Hippo

@bruno : I've just see your logs :)

Well in your case seems to be because your UDP packets are too big.

Try to enable the "compact mode" in settings or reduce number of codecs activated.

In fact, some network routers does not allow big UDP packets. Since many sip providers only supports UDP, csipsimple try to use UDP anyway to send the invite.

But if packet is considered as too big by the router, it will be dropped.

So to workaround (if the sip server does not support udp), you can try to use compact mode (in csipsimple network settings) that can decrease packet size. It has to be supported by the sip server, but usually it is. Reducing codec number can also help (because the app try to announce codecs supported in the invite).
