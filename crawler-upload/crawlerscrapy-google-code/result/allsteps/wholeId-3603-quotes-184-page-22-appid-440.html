csipsimple - issue #1229

Tapping END does not end call when internet suddenly is disconnected during call

Posted on Aug 23, 2011 by Swift Cat

What steps will reproduce the problem? 1. Both parties connected and talking well

  1. Internet connection (on mobile running this app) is lost (disconnected from ISP either on wifi or 3G) and no more audio is heard from the other party.

  2. The END button/function (on screen) does not end the call. Instead it keeps the call screen ON and I have to manually close this program from Task Manager.

What is the expected output? What do you see instead? Tapping END should end the call, which is already disconnected by disruption in internet connection. or when Internet connection is lost, call should automatically END.

What version of the product are you using? On what operating system? version 0.02-03

Please provide any additional information below.

In this case, that I am refering, only the availability of internet is lost, wifi and 3G connections are still turned on. If somehow, internet connection is established in the meantime, the call can be ended by tapping END.

Comment #2

Posted on Aug 23, 2011 by Quick Hippo

Not necessarily.

Just guess.

Wifi goes off and then go on again... Behaving like that, the connection will not be stopped. Besides, it is actually NOT stopped. Since the remote server/part does not have the information about the fact the call has been ended.

Besides, SIP already has timers about these kind of situation, and when the sip stack will not receive media data for a while, it will automatically stop your communication (and obviously the screen will go off).

So, behaving like you advise, would lead in situation where things are not in a consistant state. So that's not a good idea. BTW, if you decide to press hangup and then just press the home button of your phone, it will behaves as you expect. The sip stack will try to send the end call command to the remote side (and will actually do if the wifi comes up again), and will just timeout else.

Waiting for the timeout while on background is not a big problem I think and besides, I would advise, to not kill the application while running cause obviously it will not clean up wifi locks, screen locks and audio layer and may lead to weird state of your phone.

Comment #4

Posted on Aug 23, 2011 by Quick Hippo

Yes, there is definitions of timeouts here :

http://www.ietf.org/rfc/rfc3261.txt

(p265 : table 4). By default pjsip (the sip stack on which csipsimple rely), use the same timeout values (see http://www.pjsip.org/pjsip/docs/html/group__PJSIP__CONFIG.htm#ga20034511855d73140e26847c2220273f )

CSipSimple directly use this very reliable sip stack and does not add clever things on the top of that. So even if I would like to changes things it would be very complicated to do and would break the robustness of this sip stack (that is used on many other devices : iphone, pc, mac, .... and that is done by sip experts). All I could do would be just awful hack.

About the use case of ending call, and then pressing home button, anyway when the phone is in your pocket, if the connection is established back (so within the next 32 seconds), the BYE command will be send to your remote party or if not the remote party will consider (because these timers are normalized in SIP RFC), that the call is ended. So anyway, he will not be in communication with your pocket ;).
