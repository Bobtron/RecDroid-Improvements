Comment #1

Posted on Jun 15, 2012 by Quick Hippo

Maybe the caller id is not what you think it is used for. Sip providers doesn't always name things the same way.

The latest nightly builds : http://nightlies.csipsimple.com/trunk/ adds a field in advanced wizard for username and authentication name. I guess that what you think should be used as caller id, is the username. And the username is the authentication name. Caller ID is in fact the sip display name. In sip packets it becomes something like that : "Caller ID"

And authentication name is used when server challenge for an authentication.

So the nightly build advanced wizard could help.

Alternatively, if you want to keep the market version you can also do that by transforming your advanced account into an expert one (long press the account row, and choose wizard and choose expert). Here you can set the AoR to exactly what you want. It's more complicated but gives a real access on what is sent by the underlying sip stack ;).

Comment #3

Posted on Jun 15, 2012 by Quick Hippo

With no registrar uri the application doesn't really register. In this kind of configuration the account is for outgoing only and no registration step is made. So the app report green light and registered because the account is in theory valid for outgoing but no check with server has been done yet ;). So the problem you have with registration just appears later for call.

I don't know if it's the solution but you talk about a caller id required in your first report. Many sip provider use this label for the user name. so maybe you could try to change sip id into "...."

But I'm not sure it will help .In such case clean sip server report a user not found error.

So if it doesn't help, could you try to collect logs? See HowToCollectLogs wiki page. Also ensure that the setting use compact mode is disabled ( a sip server that doesn't support that would report the same error than you get). And last things if you get things working with another sip client I could translate from the working settings label into advanced or expert config in CSipSimple .

Comment #4

Posted on Jun 16, 2012 by Quick Hippo

Ok, just seen the logs you sent. I missed was about this issue ;).

I think the problem is the use of the "use compact mode" option (in settings > network). Your sip provider seems to not support that correctly; so if you activate the option they think it's a bad request. It will probably go better once the "Use compact mode" will be turned off.

BTW, you should report a bug to this SIP provider : For the technical information you can invoke that the compact header mode is defined here : Section 7.3.3 of RFC 3261 , and that it would be great if they support this mode that helps to reduce packet size which is important to reduce mobile network usage.

Comment #8

Posted on Jun 18, 2012 by Quick Hippo

I think that I got the problem. I spent some time trying to connect the sip provider with fake credential and always got the 400 bad request... until .... I changed the user agent ! Now I get the normal unauthorized error that I expect since I have no valid credentials for this server.

So to fix the problem I think that this will work : Go in settings, press menu key to switch expert mode (if you didn't already switched to this expert global setting mode).

Then in network section edit the User agent field. Change it for example to Twinkle/1.2.4

Else, for the account setting, the first thing you did was absolutely correct :).

If this fixes the problem, there is a bug on server side ! They shouldn't reject CSipSimple user agent. I guess that's because it contains spaces : when default to CSipSimple, it becomes "CSipSimple / rxxxx / DEVICE-ID" while when setup to something different from csipsimple it's just the value you set. But as far as I know this is supported by sip RFC.

Let me know how it goes. :)

Comment #9

Posted on Jun 18, 2012 by Quick Hippo

BTW, I found a user agent that doesn't make crash this sip server (and match some sip rfc subsection (the product ABNF) -- even if previous was apparently valid, and if I refer to existing sip user agents existing around the world, this sip server is not valid for many of them).

I'll push in next update something like "CSipSimple_DEVICE-ID/rxxxx" so for your server you don't need to change the user agent in next build. It also reduces number of characters used so good for network usage ;).

For the record : http://www.voip-info.org/wiki/view/SIP+user+agent+identification

All sip user agent here with more than one "/" breaks sip server you are using ;). So if you encounter other sip client with same problem on this server.... the culprit is probably not the sip client but the fact its user-agent header is not properly managed by the server ;)
