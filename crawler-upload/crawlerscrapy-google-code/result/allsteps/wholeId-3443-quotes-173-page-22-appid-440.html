csipsimple - issue #1346

Unnecessary round of INVITE messages

Posted on Oct 25, 2011 by Quick Giraffe

What steps will reproduce the problem? 1. Make a SIP call from CSS 2. Answer the call by a SIP client 3. Watch for the second round of INVITE messages immediately after the call is answered

What is the expected output? What do you see instead? Expected: There is no second round of INVITE messages triggered by CSS Instead: CSS triggers unnecessary round of INVITE messages, confirming the same call

What version of the product are you using? On what operating system? r1064 (and earlier versions too), Android 2.1

Please provide any additional information below. I'm watching this problem looking at log generated by FreeSWITCH (UAS). Calls are made by CSS via UAS (FS) to another SIP client (UAC).

Making a a call, CSS sends initial INVITE message. After the call is answered by UAC, CSS gets "OK/INVITE" message (as expected). But then suddenly it sends the same INVITE message again, causing the new round of INVITE messages: 1. CSS - UAS: INVITE 2. UAS - CSS: Trying 3. UAS - UAC: INVITE 4. UAC - UAS: OK/INVITE 5. UAS - UAC: ACK 6. UAS - CSS: OK/INVITE

How to avoid unnecessary second round of messages actually repeating the initial call?

Comment #1

Posted on Nov 9, 2011 by Quick Giraffe

Is it only me, who see the second round of INVITE messages coming, when call is already established?

I've installed the latest build r1094 and it still exhibit that weird problem. I watch SIP packets with FreeSWITCH: CSS makes a call. Other UAC accepts the call and, when CSS gets OK for its initial INVITE request, it sends to FreeSWITCH the very same INVITE message again. The second round of all messages is fast and don't disrupt the already established new call, but it creates completely unnecessary traffic between all parties (CSS, FreeSWITCH and remote UAC)...

Can we get rid of it?

Comment #2

Posted on Nov 9, 2011 by Quick Hippo

Can you try to collect logs or wireshark trace from the exchange. I suspect the app is actually finalizing codec negociation. pjsip supports only one audio codec at a time, but sometimes during neg, remote side leave several codecs. In this case pjsip will re-send an invite the an SDP with ONLY 1 codec prefered to communicate.

With full logs, I'll be able to tell you what actually happen.

Comment #3

Posted on Nov 9, 2011 by Quick Giraffe

Regis, you're absolutely right. I looked at logs and that's exactly what's going on. CSS sends initial set of supported codecs. After negotiation with SIP server and remote UAC it gets OK/INVITE message with reduced subset of codecs, that both ends (CSS and remote UAC) do support. Then it picks up the first codec from the list and starts the second round of negotiation, now providing only one codec. Of cause the second round succeeds, because the codec was already approved by both parties in the initial round...

Now the logical question arises - why CSS needs the second round of codec negotiation? The codec to use is already picked up by CSS from the approved list of codecs, that both parties can use. CSS should just start immediately using it... The second round is completely pointless and unnecessary. Can we get rid of that?

Comment #4

Posted on Nov 10, 2011 by Quick Hippo

Well, actually that's not really relevant of csipsimple but rather of pjsip (the sip stack used by csipsimple). So if a fix has to be done, it should be done on the sip stack, and it is unfortunately not relevant of csipsimple project. Sometimes I do some hack on the sip stack, but this point would imply a hudge refactoring.

I can tell you what I understood from my readings on pjsip project about this issue. Cause it is actually really annoying when the sip server ignores (because buggy) the re-invite / update and keep using one random entry on the codec list instead of the one choosen by pjsip.

First, the tickets on pjsip project that talk about codec lock process : http://trac.pjsip.org/repos/ticket/476 (and maybe this one too ) : http://trac.pjsip.org/repos/ticket/1242

Well, the point is that pjsip does not support to receive multiple codecs at a time. I guess that the first reason why that they didn't planned to have this feature by their first design. I think that another reason is that support multiple codec at the same time mean mounting different the codecs immplementation at the same time which increase memory footiprint of the stack, and pjsip guys really take care about using as less memory as possible. However I would have expected at least an option to choose... but there's no option for now in pjsip :(. The only way csipsimple can force pjsip to avoid this scenario is to select only one valid codec from the first SDP.

pjsip guys knows about this problem but for now that's not a high priority on their side cause the lock codec process is completely RFC compliant. So if you have other arguments to ask them to support multiple codecs at a time you can make pressure on pjsip mailing list. The only argument I had was about buggy sip server, but that's not really an argument... SIP servers must handle correctly the update/re-invite with codec locked. Network usage could be an argument, but not sure that's really significant. On CSipSimple side, it is not a good idea to make a local fix : first reason is that it will take me a lot of time to dive into pjsip media process, second reason is that it will not become easy to maintain, and last reason is that by fixing that on pjsip level, it will not only benefit csipsimple but all sip client based on pjsip (those on windows, linux, symbian, iphone, and the others on android, even the closed source one like 3cx).

So, I tag the issue as WontFix, cause I will not fix that in csipsimple. Obviously if it is fixed in pjsip, it will automatically be fixed in csipsimple too.
