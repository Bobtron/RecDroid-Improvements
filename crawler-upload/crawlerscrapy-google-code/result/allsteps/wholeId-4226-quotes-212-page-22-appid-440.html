Comment #19

Posted on Jan 12, 2011 by Quick Hippo

Ok, logs give some interesting clues. I think that the interesting one I get is the one of #4 : STUN disabled, UDP. The first register get a response from your server. Then the sip stack automatically try to update the registration to provide SIP server your public IP instead of your IP in the sub-network area. This is not done when using STUN cause with STUN the first register is already sent with the public IP address.

And then when registering with the public IP address, we get no response from the server. I think that your router get the response but do not transmit it. The firewall probably block re-entrant sockets.

So I've search over the web and found a really interesting hint : http://www.voip-info.org/wiki/view/Linksys+WRT54G [quote] : """ NOTE: There is a known SIP registration problem with the WRT54G v5 & v6 with firmware versions below 1.00.6~~: A bug in the Linksys/VxWorks firmware version 1.00.6 on the WRT54G v5 and v6 can cause SIP registrations to fail regardless of NAT settings in Asterisk or on the phone/ATA. This has been verified with a Cisco 7940 running SIP 6.3 and with X-Lite. Updating to the latest firmware will fix the problem. """ [/quote]

Can you check what is the firmware version of your wrt54g? It sounds a really good candidate :)
