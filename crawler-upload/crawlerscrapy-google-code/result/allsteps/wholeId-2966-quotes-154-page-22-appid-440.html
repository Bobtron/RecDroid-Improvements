csipsimple - issue #1867

3G: TCP requires STUN, but UDP does not

Posted on Aug 8, 2012 by Massive Bear

What steps will reproduce the problem?

  1. Make calls through 3G network.

  2. UDP calls work fine -- somehow, the SDP "c=" has the public IP address by the time the INVITE reaches the SIP Server. But from the CSipSimple logs, it appears that it had a 10.x.x.x private IP address when it left the application. On UDP, there is audio in both directions.

  3. TCP calls have no audio. The SDP "c=" has the private IP address 10.x.x.x both leaving the application, and when the INVITE reaches the SIP Server. There is no audio. If STUN is enabled, this resolves the public IP address, and audio is OK.

What is the expected output? What do you see instead? Why is there a difference in TCP and UDP behavior (i.e. why does TCP require STUN and UDP not)? Is this being modified by the carrier between CSS and the SIP Server? Or is CSS modifying it on the way out?

Since there is an automatic way to determine the public IP address (contact-rewrite), can we not use this for the SDP "c=" address as well (instead of putting the private IP address).

What version of the product are you using? On what operating system? r1594, Android 4.0.3 ICS

Please provide any additional information below.

Comment #1

Posted on Aug 12, 2012 by Quick Hippo

We can't use the contact-rewrite resolved address because it's not enough : To reach a listening socket you need both IP AND port of the media plan.

The allow contact rewrite will resolve public IP/port of the SIP control plan. Not the port of the media plan ! So this value must not be used to announce the SDP because we don't know the public port of the media socket and can't guess it. So it's good to announce the private one if was not resolved by another mean (STUN for example), because with luck something in the middle (a router or the sip server) will be able to find us back if they are able to solve NATed situations.

About why it's correctly rewritten when you use SIP UDP... I'm not 100% sure. But I guess that pjsip/CSipSimple send the same SDP in both case and that something in the middle (a router capable of SIP ALG for example) is able to rewrite it properly. And probably this SIP ALG router is only capable of inspect UDP packets and not TCP ones.

If you are in a NATed situation, it's recommended to use STUN (at least for media -- with latest nightly build you can select if you'd like to use stun for media only or for media and sip in expert setting mode). Besides, STUN is only able to resolve UDP sockets (so only for SIP+UDP and media channel RTP/RTCP).
