csipsimple - issue #1176

TLS and SRTP and Wireshark

Posted on Jul 27, 2011 by Massive Giraffe

What steps will reproduce the problem? 1. enable SRTP in general network settings and the account to be used 2. enable TLS in general network settings and the account 3. provide the client certificate, key and CA certificate in the TLS settings

What is the expected output? A completely encrypted communication, meaning that the control channel for the SIP communication is encrypted with TLSv1 and the media channel for the media transportation is encrypted with SRTP.

What do you see instead? If used sepeartely (either TLS or SRTP) both work fine. But if they are used together only TLS is working and the media transport is over UDP (not even RTP, which I don't understand).

What version of the product are you using? On what operating system? The newest nightly build on android 2.1update1 and 2.3.1

Please provide any additional information below. In my own application where I only use the libraries and the wrapper from CSipSimple I experience the same problem. But so far I couldn't find anything useful on the internet. I really don't understand the problem, since those two channels should be independent and it works perfectly fine if not used in combination. I use Kamailio as the proxy server.

Comment #3

Posted on Jul 28, 2011 by Massive Giraffe

Ok it seems as if I found the root of the problem for this thanks to a hint from Régis.

The problem is that Wireshark, which I use to trace my package, did not recognize the correct protocol of the package. It cathegorized the media channel as a UDP instead of a SRTP. Well this is not even that wrong since SRTP has a UDP header (cf. http://www.networksorcery.com/enp/protocol/srtp.htm).

But how do we then know that the media channel is encrypted? Well, Wireshark has a very neat feature which lets you replay captured VoIP sessions. If the communication is unencrypted we will be able to hear the communication between the two phones. If it is encrypted we only hear some noise. It is important that you use the G.711 codec for that (I tested it with PCMA, but should also work with PCMU). So far this is the only codec wireshark supports.

If the control channel is encrypted with TLS then it is not recognized as a SIP communication and we have to do a little trick. So following are two short descriptions how to make sure the media channel is encrypted (since we're not able to "read" audio communication)

without TLS enabled (with or without SRTP): -capture the communication -go to Telephony->VoIP Calls (depending on the capture sice it may take a few seconds to find the calls) -select the conversation and press "Player" and in the next window on decode -set the cursor at the position you want to and press "Play". -if you used SRTP you should only be able to hear noise -if you didn't use SRTP you should be able to hear your communication normaly

with TLS enabled (with or without SRTP): -capture communication -go to a UDP package which is for the media communication -right click it and select "Decode As..." -it will automatically select the port on which the package was. If the source and destination port are not the same you have to choose "both" from the drop down menu. -select "RTP" from the list on the right side (also if the communication was SRTP) -click OK and you'll see that all the packages changed the protocol from UDP to RTP -click on Telephony->RTP->Show All Streams -you should see 2 streams per captured communication -select one and click on "Analyze" -now click on "Player" (make sure to have a newer version of Wireshark since in older ones the player is not implemented in this stream analysis tool) -click on "Decode" -if you used SRTP you should only be able to hear noise -if you didn't use SRTP you should be able to hear your communication normaly -in order to "undo" this little change in the protocols you either have to close Wireshark or make a right click on a package again, click "Decode As" and click on "Don't decode" and then OK.

I hope this clarifies some things for those who experienced the same problem as I did.

Thanks to Régis again for the help. I think you can close the issue.
