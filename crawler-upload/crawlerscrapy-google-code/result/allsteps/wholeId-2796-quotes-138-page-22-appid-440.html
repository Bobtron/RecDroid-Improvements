Comment #1

Posted on Oct 12, 2012 by Quick Hippo

Mp4 is a file format container. It's not a video codec. mp4 usually use h264 (aka MPEG-4 Part 10) as codec but not always.

So, I don't understand your request.

Video codecs that are planned to be supported in the future are webm (the one that is usually contained by vp8) and theora (the one that is usually contained by ogg containers).

And, actually these are not requests for csipsimple but rather for pjsip (the underlying sip stack). I'll not track progress on that here so, and it will maybe be contributed by other guys on pjsip side. The main blocking point is RTP packetisation/depacketisation glue with pjsip as both theora and webm are supported by ffmpeg and can be built on android. (Note, the rtp packetisation/depacketisation of ffmpeg can't be used in pjsip and is usually very basic -- for example the webm one only packetise 1 frame/packet).

Comment #4

Posted on Oct 12, 2012 by Quick Hippo

Ok, it's "MPEG-4 Part 2" (http://en.wikipedia.org/wiki/MPEG-4_Visual) It seems very similar to H263. Would be possible to support pretty easily once packetisation/depacketisation is done. Adding the ffmpeg parameter takes 5 seconds ;) but will not add it as a supported codec as missing sdp parsing and rtp packetistation/depacketisation. After a quick review of rfc 3016 seems packetisation of this codec is pretty obvious (one frame per packet without split).

However seems a pretty legacy codec. So for me ok to try to see to integrate it once webm and theora will be there, but takes the third priority as far as I'm concerned. However, maybe on pjsip will land sooner as they already list it in code (http://svn.pjsip.org/repos/pjproject/trunk/pjmedia/src/pjmedia-codec/ffmpeg_vid_codecs.c) as a codec description possibility (PJMEDIA_FORMAT_MPEG4) but not treated elsewhere and has not the packetizer/depacketizer implementation.
