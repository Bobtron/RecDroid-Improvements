Comment #1

Posted on May 9, 2013 by Quick Hippo

Not necessarily. Caller ID is something that is linked to how your sip provider gateways on the landline network. They can have settings on their web interface or interpret some of the sip parameters the app pass when calling/when registering.

That's not consistent from one sip provider to another. Some has only settings on web interface, some take into account what SIP standard calls the AoR username, some others takes into account what SIP standard calls the "Display name".

In wizards that are known to support Caller-ID feature by AoR username or by display name the wizard indicates one specific "caller-id" field is configurable in the account settings. You also have, in global settings, the possibility to setup a global "caller-id" but it's in fact a global 'display name' (what between sip client we use for caller id). But if "display name" is not taken into account to be used as caller id on pstn network, it will have no impact for your landline calls. In expert wizard, you have access to most things that sip protocol allows (and so it's possible to do every specific configuration one could wish). Note that you can convert an existing account into expert wizard at any time by long pressing the account row and "choose wizard".

If your sip provider have a way to configure something in their web interface I doubt they interpret one of the sip parameter passed by the app for caller id (else the setting in the web interface would be useless).

Comment #3

Posted on May 13, 2013 by Quick Hippo

Hi, Thanks for the details. Do you have a sip trunk with voipms? It looks like it uses a trunk + features that comes from IMS world + your server is identified by voipms as a trusted one. In this case, the way that could help is probably the one defined in RFC 3325 and is tracked by this issue : https://code.google.com/p/csipsimple/issues/detail?id=2167

However, depending on how voipms behaves when it's a client that send a "P-Preferred-Identity" and not another trusted server that send a "P-Asserted-Identity", it might not be so useful.

Well, I guess I could pretty quickly re-introduce the "P-Preferred-Identity" in next nightly build so you can test it quickly and let me know if helps and if I should definitely push this feature with higher priority. I'll continue on issue 2167.
