csipsimple - issue #612

What steps will reproduce the problem? 1. Setup standard Gizmo5 account 2. Connect Google Voice to Gizmo5 3. Receive incoming call on Google Voice Number 4. Answer in cSipSimple (i.e. incoming caller -> GV -> Gizmo5 -> cSipSimple)

Comment #1

Posted on Jan 22, 2011 by Quick Hippo

Yes, there is a missing feature in CSipSimple.

In fact priority is an indication given by csipsimple in the SDP.

In fact it is the way SIP works : When you "invite" somebody you send to him your own codec priority list. Then the other compare with what he support and choose the first supported codec in your priority list.

As consequence for outgoing calls you send the invite so the server on the other side take the first supported codec in your list. On the other side when you receive a call CSipSimple acts the same standard behavior. It take the first supported codec in the top of the priority list of the other side.

So what you observe is absolutely logic and it is not a bug.

However the missing feature is to say to have an option to "force" codec choice according to our priority list. I have not done that yet cause it should be done in the SIP stack (CSipSimple use pjsip as sip stack), and the sip stack does not support that AFAIK.

So accepted as enhancement (I had already that on my todo list but good to have an issue to track that).

However meanwhile, as you know exactly what you server support you can simply disable the PCMU/A codec. It will ensure that it will not be choosen by CSipSimple for incoming (long press on codec row to disable codec). However the bad side of that is that if somebody from another server call you directly (without using the media gateway feature of your sip server the call may fail if the other side does not support remaining codecs).

Comment #4

Posted on Jan 22, 2011 by Quick Hippo

For codec efficiency it depends on many many things : the CPU power and arch (support floating point or not), bandwidth available, options such as voice detection and echo cancellation, it may also depend on the audio driver available on your phone.

So a little bit hard to give advise in the absolute :). There is some intersting charts over the web. For example : http://www.broadcom.com/support/broadvoice/codec_comparison.php But there is a lot of others. Some focusing on voice quality, others on bandwidth... (If bandwidth is important and your network does not support such a bandwidth quality rapidly decrease).

Most of these charts does not include the recently released codec from Skype (named SILK) that is now included in CSipSimple (nightly builds). It's an interesting codec cause support low bandwidth when 8kHz and high quality when 24kHz.

Last point about g729, remember that's a codec that use is restricted to a proprietary license. I do not distribute this license (I distribute only an opensource application). I included g729 cause of pressure of users that need this codec cause their SIP provider already pay a license for them to use that. But I'd advise to use iLBC instead cause it's a true opensource codec.

Just last point about your initial issue. After replying I search a little bit more in pjsip source if there were no quick way to provide such an option and found something interesting. There is a static option (I mean that user can not change in the application but that can be changed when I build the application), that should allow to prefer our codec priorities instead of remote codec priorities.

So in a first time could be possible to activate that.

That would make more sense for users than the current default behavior. Then I'll try to add that as an option that can be changed by users.
