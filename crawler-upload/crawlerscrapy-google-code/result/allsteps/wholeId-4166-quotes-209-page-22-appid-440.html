Comment #11

Posted on Jan 31, 2011 by Quick Hippo

Codecs list in combination with an old account (configured previously to this version) are a good candidate.

Let me try to explain what may happen. In previous version transport for account was set to "Auto" which mean that the sip stack decide whether to send packets using UDP or TCP according to the packet size. In new version I decided to change the default to "force UDP" cause many sip servers does not support TCP. As the new version support more codecs it send bigger packets. As consequence if your account is set to mode 'auto' (an old account not saved a new time with the new interface), you didn't observe any problem with previous cause packets size was not so big and that the stack always decided to use UDP, but now that there is more codecs it try to use TCP and after tcp timeout try to use UDP and then succeed.

So to solve the problem : try to save again your account (edit account and press save). At least it will enforce UDP. And maybe too, worth to disable some codecs (in settings > media > codecs , long click to disable). CHeck with your providers what are supported codecs. They probably do not support all new codecs (specifically silk codec).

Comment #15

Posted on Feb 1, 2011 by Massive Lion

Just a comment from experience with similar problems: The best solution is to keep the UDP packet size as small as possible :-)

For example, Ethernet supports a maximum packet size of 1500 bytes, now reduce this by IP and UDP overhead and you get the maximum size fo SIP (approx 1450 bytes). If the SIP packets are bigger than this maximum size IP splits (fragments) the packet - and this is usually bad for several reasons.

Some firewalls, depending on their configuration, even throw away fragmented packets completely and thus may block SIP INVITE at all.

Regards, Werner

Comment #16

Posted on Feb 1, 2011 by Quick Hippo

Yes absolutely. There is an build option in pjsip to allow short tag naming for SIP packets. I was wondering if I should not enable it. But I fear that some SIP provider do not support it. As it's not a dynamic option (that the user can set), it's probably risky... (but I do not what is the state of the art regarding the support for that).

For now in the last update on the market I've disabled SILK codec by default (as it is supported only by few sip provider not worth to enable it). It should reduce packet size, but I advise users to activate only codecs supported by their SIP provider.

If the sip provider support TCP I'd also advise to activate mode "Auto" instead of "Force UDP" cause it will automatically fallback to TCP if packet size is too big. But as it seems to not be supported by a majority of sip provider I've set force udp by default to avoid problems.
