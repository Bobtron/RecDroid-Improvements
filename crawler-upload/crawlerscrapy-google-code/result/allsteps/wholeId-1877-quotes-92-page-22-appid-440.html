Comment #4

Posted on Aug 12, 2014 by Quick Hippo

Hi, There is not all the logs of the sip exchanges, but it sounds to me a problem that I also observed with TCP. As this is managed by the underlying opensource sip stack (pjsip), I asked on the mailing list, and sounds the problem is not with the behavior of the sip stack but ... with what the server replies. You can read my investigations + conclusions here : http://lists.pjsip.org/pipermail/pjsip_lists.pjsip.org/2010-December/012231.html (I was very new to SIP when sending that... was the beggining of csipsimple project ;) ).

So, in short, it appeared that what is done by pjsip is totally valid. And actually few sip stack actually follow this part of the RFC and other stack does that wrong ! The big problem is that some servers (such as pbxes.org) and default settings for many opensource sip servers, does not provide a clean configuration regarding this part of the RFC, and they actually asks the SIP application to change the transport to reply the ACKs. As other SIP stack are buggy following this point, the problem is often reported to SIP applications based on pjsip. For example, when configuring freeswitch and try to get TCP working, you can get something working quickly but that will require the client to switch to UDP. To get something that actually follows the RFC to asks client to keep TCP it requires some extra conf.
