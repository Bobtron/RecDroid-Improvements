csipsimple - issue #2512

disabled codecs still used by css

Posted on Oct 11, 2013 by Happy Giraffe

What steps will reproduce the problem? 1. Compile from source and install, say, gsm codec with both the Core and Codecs packages. 2. Configure sip account for a provider who accepts a certain codec (say gsm), then go to the codecs section (I don't use slow/fast bandwidth separation of codecs and "lump" them all in one place) and activate the "usual" codecs, while disable the gsm codec. 3. If I then get a call through this provider which uses one of the disabled codecs (say gsm), that codec is used. The only way I could truly disable that codec is not to compile it at all.

What is the expected output? What do you see instead? Codec not to be accepted (either drop the call if no enabled codec is present or use another codec that is enabled).

What version of the product are you using? On what device / operating system? Nightly from the source repo.

Please provide any additional information below. As I already pointed out, I stumbled upon this with the gsm codec. Since this codec was compiled, BUT disabled in my Codecs package, I did not expect this codec to have been used.

It is also worth pointing out that I also had two other codecs enabled: 711-u/a. My gsm codec was present in the list of codecs, but it was disabled. When I received a call, to my surprise, the gsm codec was used (I verified that when I pressed the "i" button during call and looked at the codec used).

The only way I got rid of that problem is to recompile the 2 packages and explicitly disable compilation of the gsm codec and then reinstall the 2 packages. Whn I did that, it obviously worked and the 711-u codec was used.
