Comment #8

Posted on Aug 26, 2012 by Quick Hippo

Hi

Thanks for the logs and the explanation. I misunderstood first the problem.

In fact the problem starts from the answer of the application. It sends a 200 OK to the invite but never get any reply (ACK) from the server. So it never arrives to the confirmed state.

In this case the call will automatically ends after 30 sec of retry. Normally in this case it should behaves slightly differently than your description (it's what I observe in logs too)-- on the device that receive the call --. The interface should display "Connecting…" in the blue line. Normally there is also the button to terminate the call. In your logs I see that when you terminate the call the server reply with "481 Call leg/transaction does not exist" and the call ends immediately.

However possible that if your server on the middle actually doesn't takes a role of sip proxy the device that initiate the call is in the state you describe.

Usually this case happens when the SIP server is not a sip proxy (or manage proxying a weird way). If I don't mistake FreePBX is based on Asterisk which is not really a SIP proxy (it's more a gateway to pstn network). As your are on your LAN a SIP proxy is probably not mandatory in your architecture, so it could be fine. However, by default csipsimple when you configure server with the basic wizard consider that as both registrar and proxy (because in mainstream user case and sip provider case there is very often a real sip proxy in front of the topology). So you should try to disable in csipsimple account configuration the sip proxy. To do so, switch your account to expert wizard mode (http://code.google.com/p/csipsimple/wiki/ExpertSettingMode?wl=en#Account_settings ) and edit the "Proxy" field and remove contents of it (set it blank). Do that on both devices (caller and callee).

If it doesn't help and if you have an hand on the sip server it would be very interesting to have logs of the server and/or a tcpdump trace (~# sudo tcpdump -s 1500 -w outfile.pcap). It would confirm that the sip server receive correctly sip packets from the client and what/where it replies if it replies.
