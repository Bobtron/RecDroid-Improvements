csipsimple - issue #1782

Unable to hang up if call is not established

Posted on Jun 13, 2012 by Swift Lion

What steps will reproduce the problem? 1. Dial a number and press the call button. 2. The called phone rings. 3. Tap the Hang up button to end the call so as not to establish a call.

What is the expected output? What do you see instead?

Would expect the call to be terminated as the caller chose to end it before it was established (i.e. dialled a wrong number and realised after tapping the call button and wants to end the call before it is answered). The called phone should see a missed call notification.

The call does not disconnect. The called phone continues to ring even after the calling device has been shut down.

Have seen the FAQ for "I can't end calls" and other bug reports where selection of UDP for transport has been suggested. In this case, however, it was verified by going into the expert mode, that the UDP is set by default on the transport for the Localphone profile.

What version of the product are you using? On what operating system? CSipSimple version 0.03-01 r1108 Motorola Xoom MZ604 Android version 3.2

Please provide any additional information below.

Provider: Localphone

An interesting observation though .. if the called phone is picked up, (after the call has been established) it is possible to end the call by tapping the end call button from the calling handset.

Shall be happy to provide more details if you need. Feel free to ask. Thanks for your efforts.

Comment #1

Posted on Jun 14, 2012 by Quick Hippo

I think that it's different from the FAQ entry, because in your case it's about cancelling a call, not terminating an ongoing one.

When you press hangup the application send a CANCEL to the server. It expect the server to reply to this cancel to consider that the call is actually cancelled correctly. For obvious robustness reason it's good behaves like that because if cancel packet get lost and other side answer then the state of the application should be consistent with the real SIP state.

I guess that in your case, the sip server doesn't handle properly the CANCEL request. If you can collect logs (see HowToCollectLogs wiki page), it could help to be sure of where it goes wrong.

As on other providers I tested there is no problem with cancel, and that this is entirely managed by the underlying sip stack which is a reliable opensource project used in many products, my first thought is about a problem on server side. But I could be wrong, that's why logs will be helpful. Maybe also it could be something with configuration. The Localphone wizard has already some specificities, maybe there is missing configuration to have everything working properly.
