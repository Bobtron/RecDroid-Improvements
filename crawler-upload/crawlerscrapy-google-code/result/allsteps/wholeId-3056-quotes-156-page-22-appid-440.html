csipsimple - issue #1768

Add codecs ptime parameter

Posted on Jun 1, 2012 by Swift Panda

What steps will reproduce the problem? 1. Select G.729 as only codec 2. Set ptime to 40 3. Make a call

What is the expected output? What do you see instead?

The expected output would be packets containing 4 10ms frames. But independant of ptime packet always consists of 2 10ms frames. This increases the overhead by at least 30%, which is a lot when using low bandidth links.

The same holds for iLBC (setting ptime to 60 results in 1 30ms frame per packet).

What version of the product are you using? On what operating system?

Latest trunk.

Comment #1

Posted on Jun 1, 2012 by Quick Hippo

Normal, it's not the same ptime.

You think it's the ptime of the codec, while it's actually the ptime of the audio device layer.

The option to change ptime from CSipSimple is not yet in the user interface. I have to add it. It will be per codec, and actually it's not ptime that you set in this case, but rather number of frames per packet.

Since you get codec_ptime = frame_per_packet * frame_length_ms.

So such an option could not be to set ptime, because it would lead to mistakes if users set for example 15ms while using g729 that has 10ms frame size.... just impossible value ! So it must be a setting that allow to set frames per packet instead.

I have to add this option. It's pretty hard to present in user iterface because it's a per codec option. BTW, it's not the only one to add. For example for video you'll be able to set profile. For opus you may also be able to changed target bandwidth/clock rate, the same for silk and so on...

So it's absolutely not a bug, but an feature request that is already in my todo list, but good to have a issue to track that :). It's already supported by the sip stack, and it's just about user interface problem.

Comment #8

Posted on Apr 13, 2015 by Helpful Camel

I've also noticed setting a custom "frames per packet" (G729, iLBC) "seems to do anything".

Well, after tracing, the codec's default frm_per_pkt is correctly set, but while negotiating SDP, if remote peer sends a ptime attribute, PJSIP overwrites our default with the received one (get_audio_codec_info_param function in pjmedia/src/pjmedia/stream_info.c): /* Get the remote ptime for our encoder. */ attr = pjmedia_sdp_attr_find2(rem_m->attr_count, rem_m->attr, "ptime", NULL); if (attr) { pj_str_t tmp_val = attr->value; unsigned frm_per_pkt;

    pj_strltrim(&tmp_val);

    /* Round up ptime when the specified is not multiple of frm_ptime */
    frm_per_pkt = (pj_strtoul(&tmp_val) +
                  si->param->info.frm_ptime/2) /
                  si->param->info.frm_ptime;
    if (frm_per_pkt != 0) {
        si->param->setting.frm_per_pkt = (pj_uint8_t)frm_per_pkt;
    }
}
