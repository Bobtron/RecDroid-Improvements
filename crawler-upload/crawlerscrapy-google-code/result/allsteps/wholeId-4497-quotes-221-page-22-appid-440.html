Comment #4

Posted on Nov 1, 2010 by Quick Hippo

Ok, understood.

I'll try to make things more robust.

Indeed, when you are in accounts activity I assume that sip stack has been started (not necessarily registered, but at least that the native stack has created some native objects). Then as creating account requires to create other native objects it waits for native stack to be started. But if stack has not been started you can observe what you observed. And indeed, could be due to something that is listening on the same port on your device. Cause sip try to create tcp/udp transport when starting ... if it stay at this point trying to create transport, it can lock all other calls to native stack.
