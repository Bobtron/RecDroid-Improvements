Yep I'm aware of this change but it should apply only to apps targeting API 19. It's not the case of CSipSimple (unless you are running your own build and modified something in the manifest). The doc states : """ Applications whose targetSdkVersion is earlier than API 19 will continue to see the previous behavior in which all alarms are delivered exactly when requested. """ http://developer.android.com/reference/android/app/AlarmManager.html

Of course, when we will upgrade to api 19 as target we will take that into account. But it's not yet done. (Android always did this way which is very good for api evolution). But from your description (and if you are using a build I provide), the statement they announce in the doc (and that was expected as they always do this way) is wrong. And they did something that hard break existing apps. That's why I'm pretty afraid of this problem and why I mentioned the break in the timer api.

For reference, the other thing they broke and required me to do an upgrade was about a permission they removed. On this topic (permissions), they are doing transition less seamlessly and I was not so surprised to have a hotfix to do.

Comment #11

Posted on Dec 11, 2013 by Grumpy Monkey

Apparently this might be an issue with IPV6.

https://code.google.com/p/csipsimple/issues/detail?id=2551

https://code.google.com/p/csipsimple/issues/detail?id=2549&can=1&q=320044

You can solve this by forcing IPV 4.

These instructions are for the Android OS, not within CSipsimple. 1. Go to settings 2. Wireless & Network > More 3. Mobile networks 4. Access Point Names 5. Select T-Mobile GPRS (fast.t-mobile.com) or your selected carrier. 6. Click on APN Protocol (by default it's set to use IPv6) 7. Select IPv4/Ipv6 8. Exit out of settings.

However, this is still a bug with the UA not being able to handle IPV6. This may be merged into issue 2549
