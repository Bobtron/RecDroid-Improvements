Comment #25

Posted on Feb 17, 2013 by Quick Hippo

On update on the issue : I spent all my day trying to debug it as was able to reproduce. In fact the key point to reproduce is to use a server with TCP on port 5060 and to try invites with big packets : for example a lot of codecs, ice enabled, video enabled, sip mode compact disabled - the combination of these allows to generate big sdp.

As soon as the packet is bigger than a value, the app get totally blacklisted for this socket and any new sip packets sent (even lower than the limit) will not be transmitted.

So, it appears that it's something that happens when packet size is bigger than the MTU.

I found some very crappy workaround to fake the socket layer (basically changing the MSS etc), but all workarounds I found had limits and can't be introduced in stable.

So after these debugs, I started to search with correct keywords on google and found some very interesting topics :

http://bugzilla.netfilter.org/show_bug.cgi?id=760 and finally : http://code.google.com/p/android/issues/detail?id=32736

And the best workaround that is described in these issues ... and that works ! ... is to have : - remote server listening on a different port than 5060 (I tested with 6060) - local app not bound to 5060 port (should be fine by default as default is 0 = random free port).

So if you can try this workaround on your side and tell me how it goes. BTW, I'm very surprised there is such a rule on netfilter. SIP uses TCP to solve the MTU limit... so they are absolutely silly to block a sip app trying to send packets bigger than mtu limit ! And apparently it's not platform specific... so I fear the guy that introduced this limit will break a lot of apps !

Another workaround is to try to reduce SIP packet size (to do so, having less codecs activated, sip compact mode enabled, disabling ICE can help).
