Comment #2

Posted on Apr 5, 2011 by Quick Hippo

Thanks for logs.

I don't think that RTP recv errors are linked to the dtmf problem. (but that's just a guess). Usually that just cause the server is talking with a codec that is not the one chosen by the SDP negociation. That's cause pjsip only support one codec at a time but that's not hurting since they finally force the codec to use if the server announce multiple codecs. But anyway telephony RTP packets are treated as separate flow. So should not be a problem. At least that's errors I usually get but that are just here at the beggining of the come while the server is not yet aware about the fact the client has made a choose on the codec to use for audio. (unless you get one RTP recv() error each time you send a dtmf tone that would mean server is trying to reply us using non 101 payload type, but I don't think so)

rfc 2833 (aka RTP events mode) should be a good choice. It is supported by pjsip (the sip stack csipsimple rely on). Logs seems to send it correctly and usually when it does not try to fallback to tone generation (in band generation) it means that the other part is fine with RTP packet.

However sometimes, when there is multiple media gateways between you and the final machine that should receive the RTP payloads, one gateway can drop RTP telephony events. I guess other sip phones works with your configuration, but usually most sip phones use by default tone generation. It's the most robust way (less clean but more robust). So it could be interesting in CSipSimple to force the "In band generation" mode. I think that asterisk is able to manage it too.

Comment #7

Posted on Jul 12, 2011 by Helpful Kangaroo

I am also using Xoom with 3.1: the only mode that works is SIP INFO, and even then all keys register as zero.

The only way to get tones to work with Xoom is to disable Voice Audio Detection, and force Dialing Tone Feedback. I use GVoice + Sipgate, so all my calls are incoming. After receiving the call from GVoice, you MUST open the keypad and dial a number so that the Tone Feedback starts. If you try dialing after the call is connected, the tone feedback WILL NOT WORK. Also, if you change any volume settings tone feedback will not work for the remainder of the call. I've had calls where I get halfway through an automated menu and accidentally change the volume, which means I have to restart the call!

So! If you can successfully get dial tone feedback in-call using the keypad (which is sometimes impossible for certain numbers), you can navigate automated menus. This works because the microphone and speaker are right next to each other on the Xoom, so the tone coming out from the speaker is recorded by the mic. This works MUCH better if you hold your figer over the left speaker as you dial (right speaker when viewed from the back).

It's kind of a silly hack, but the only way I've been able to get 'DTMF' to work. Oh yeah, using external tone generators doesn't work either because CSipSimple locks the audio stream. I've tried many different settings with no success in allowing tone generators to output audio.

Please look into resolving DTMF issues for Xoom. Even without DTMF, menu navigation would be much easier if Dial Tone Feedback worked properly. I'm surprised that the software doesn't generate the tone directly into the input stream, like generating an audio sequence into the mic instead of the speaker. ;)

Comment #14

Posted on Mar 28, 2012 by Quick Hippo

Well, I just pushed some workaround based on the deduction I do from the fact it always send char "." It's available as r 1356 here : http://nightlies.csipsimple.com/trunk/

If my analysis is correct, it is a potential bug in Xoom that doesn't correctly convert key code to char. Normally there is some Keymap tool in android that should allow to transform key code to char. This is the clean way to do that and normally allow to treat the same way virtual keyboard and a real keyboard. But apparently, if all keys you press produce the same signal, this is probably the root cause of the problem. So I just did a fix by coding my own mapping for 0-9, # and * chars (others chars will fallback to clean keymap method, but since the ui doesn't allow to type others should not be a problem for the xoom).

If one of you can test the nightly build and tell me if it helps.
