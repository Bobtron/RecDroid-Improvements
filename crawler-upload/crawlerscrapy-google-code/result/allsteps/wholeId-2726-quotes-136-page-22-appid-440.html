csipsimple - issue #2069

Call log doesn't keep DTMF, pause and wait info

Posted on Nov 10, 2012 by Quick Giraffe

What steps will reproduce the problem? 1. Dial a number with pause ',' and/or wait ';' symbols inside 2. Watch the call succeeding and DTMF tones sent correctly 3. Go to log and try to re-dial using record left in the log. It dials correctly, but fails to send any DTMF tones

What is the expected output? What do you see instead? If CSS supports re-dialing using its log records, I expect it should use them correctly and without any modification.

Instead or that, when dial string is recorded in log, DTMF tones after pause/wait are somehow removed. So, what is recorded in the log is not what was actually dialed.

What version of the product are you using? On what operating system? Android v2.2, HTC Aria

Please provide any additional information below. I've tried to test the new feature, implemented in r2003: http://code.google.com/p/csipsimple/issues/detail?id=2061

I've entered correct DTMF sequence after ';' and made successful call. Then I tried to re-use the record left in the log and repeat the call. But this time, when call was answered, the DTMF sequence was not sent. Reason? Log record is modified and DTMF tones are removed from the dialing string.

BTW, in order to enter pause ',' or wait ';' symbols I have to switch from 123 to txt mode. If then I switch (on purpose or not) back to 123 mode, the entered dial string disappears... This is one more reason to fix the issue: http://code.google.com/p/csipsimple/issues/detail?id=1916&thanks=1916&ts=1346125315

Comment #1

Posted on Nov 10, 2012 by Quick Hippo

Indeed,

However, tt's not redialing feature, it's call log which is a little bit different. In the code design the call log is added at the end of a call. So what's stored is the sip uri of the remote part. Not the dtmf, nor the number before rewriting rules. It's by software design. Log is not "modified" it just contains the actual SIP uri. Keep in mind that supporting "," and ";" is not some SIP feature, it's something implemented in the application on the top of SIP.

Also, just by curiosity I just did a test using GSM stock application on my SGS 3. And guess what... it also ignores the pause and wait symbols in the call logs when I use the stock GSM application. So their approach is probably identical to mine and they store the remote phone number without extra stuff not related to how to reach the callee. I don't know how it goes with stock android phone/contact app but I guess it's the same.

So for parity with gsm app and mainly because modification to add this enhancement would need a complete rewrite of how the app manages call logs. Marked as WontFix.

Comment #2

Posted on Nov 10, 2012 by Quick Giraffe

I guess, I mistakenly thought, that CSipSimple is a SIP phone, and is not designed to reprint the old GSM technology. It's obvious to me, that there is a lot of things, that old GSM can't implement or implements differently for what SIP can offer now... E.g. GSM can't dial URI... and my old GSM phone uses completely different symbols for setting timeouts - 'p' and 'w' too.

If the goal is to make CSS design similar to GSM, then I don't have any proposals, or frankly, even any desire to improve it... WontFix seems appropriate in such case.
