Posted on Jan 29, 2012 by Quick Hippo

It usually due to a problem of network topology + configuration combination.

In fact the application respect sip rfc. If the sip transaction is not considered as finished it will not close the user interface. Many sip client anticipate the close while the call is still in a state that is not really terminated.

In fact what I guess that happens is that csipsimple send the terminate (bye or cancel depending on the case), but the remote side never reply a ACK. As consequence the dialog is not considered as terminated for the very good reason that it's defined like that in SIP protocol. It will just wait for timeout (that is also defined in sip specifications). This timeout is there to be able to manage slow network and retransmition of the packet.

So the problem is not to know why the app does not close the view (because it has good reason to do so), but why the remote side never replies (or why the application does not send the bye/cancel to the correct remote party). As I said it's usually because of a bad configuration on the app regarding the server, or on the server. To find a clue about what's going wrong, could you follow step of HowToCollectLogs wiki page. I'll try to investigate in your logs why there is no reply from remote side. Thanks in advance
