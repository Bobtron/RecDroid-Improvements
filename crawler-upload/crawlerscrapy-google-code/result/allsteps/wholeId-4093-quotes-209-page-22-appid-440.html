csipsimple - issue #691

Use of ";ob" in Contact: header not supported by all proxies

Posted on Feb 7, 2011 by Massive Ox

What steps will reproduce the problem?

The only way I can reproduce this issue is when registering with XNET in New Zealand. Registration is successful, but it appears that addresses of the form "who@where:5060;ob" cannot be resolved when the xnet proxy attempts to connect to my phone for incoming calls.

What version of the product are you using? On what operating system?

Version 0.00-16 r410. Running on a Google/Huawai Ideos with Android 2.2.

Please provide any additional information below.

It seems the ";ob" extension has only appeared in a draft RFC (tools.ietf.org/html/draft-ietf-sip-outbound) and not in any standards-track RFC. There should perhaps be a flag or a switch to turn on/off non-standard options?

Posted on Feb 8, 2011 by Quick Hippo

Ok, I can report the pjsip option to user interface for this setting.

However, you should report the bug to your sip provider too. I think that they will be interested in it. Cause even if they do not support the option they should not misbehaves regarding extensions, it should just ignore it... and I think that is in RFCs.

By default I'll keep PJSUA_REG_USE_OUTBOUND_PROXY + PJSUA_REG_USE_ACC_PROXY option cause if supported in mobile environment, outbound proxy provide a lot of advantage. On most sip server even if ob is not supported the sip server just ignore it and there is no problem with that.

As consequence, if XNET misbehaves with that option, what I could do (and what I do usually in such case when sip server need some legacy option), is to create a wizard for it that automatically set this new option to the correct value.

So, in the same time I'll add the option in expert account wizard for removing ob support, a good idea would be to create a wizard for XNET to ease things for XNET users. To do so, if possible, could you send me a 48x48 px icon for XNET (if not I'll search on the web), and most important (that's more hard to find usually), the default configuration that I should hard code for XNET (sip registrar, domain, if username and auth name are different etc).

If you can get support from XNET and ask them to fix their server to ignore the unsupported extension, let me know, I'll remove the legacy option (not the wizard ;), if you use it I guess other users too, so worth to have the wizard anyway )

Comment #3

Posted on Feb 8, 2011 by Quick Hippo

mmmh... However, regarding pjsip code even if I disable rfc5626, it will continue to send server ;ob. I guess that make sense cause server should never fail just cause of an extension in contact URI..... So if we disable rfc5626 support it only remove the treatment done after not what is sent.

As consequence I can't fix it. It could be done by pjsip guys, but I really think that they are doing absolutely right regarding RFC + the fact a sip proxy should not fail cause of that regarding other SIP rfc. (so bug is clearly on server side here).

Workaround for disabling this rfc in pjsip does not plan to workaround server that misbehaves... so ... impossible for me to provide you a workaround for that. If the sip proxy actually fail cause of that (worth to check however, that's actually the problem); you should ask XNET to fix it. Cause here it fails with pjsip, but if so it will fail with all pjsip based clients (there is others on android such as 3cxphones, but also on iphone, on linux, mac, windows based on pjsip).. and I guess it will also fail soon on other sip clients that use this RFC (or potentially wants to support another extension...)

So as for this issue => won't fix. Cause up to the sip server to fix that and the only legacy workaround planned by pjsip does not provide a way to disable ;ob but the treatment done after...
