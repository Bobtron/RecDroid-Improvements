csipsimple - issue #2199

[enhancement] give the choice on the code of hangup

Posted on Jan 26, 2013 by Quick Giraffe

What steps will reproduce the problem? 1. Reject incoming call with the red slider

What is the expected output? What do you see instead?

"486/Busy Here"; Calling party should hear a busy tone. Instead, displayed is "603/Decline"; Calling party (GSM phone) hears a "Connection error" tone and displays "Call not allowed".

What version of the product are you using? On what operating system?

0.04-04 r1916 on a Samsung Galaxy Glide SGH-I927 running Android 4.0.4

Please provide any additional information below.

When a call is being rejected with the red slider, it is expected that the calling party will hear a busy tone (and optionally gets forwarded to the voice mailbox or somesuch), for which to happen, CSipSimple should send a 486 when the red slider is slid.

Also, according to http://en.wikipedia.org/wiki/List_of_SIP_response_codes

  • 486 Busy Here - Callee is busy.

    This is what one is when one can not receive a call (OK, one might not want to receive a call, but there's no real provision for that...)

  • 603 Decline - The destination does not wish to participate in the call, or cannot do so, and additionally the client knows there are no alternative destinations (such as a voicemail server) willing to accept the call.

    This is most positively not true, and what makes it not true is the "additionally" clause. CSipSimple does not currently know if there are or there are not any alternative destinations (no setting for that). Moreover, even if it does for a fact know that there are alternative destinations by means of the user having set voice mail stuff in an account's settings, it still rejects the call with 603. If voice mail stuff is set for an account, I think it's a pretty positive indication of there being a voice mail facility for said account. Even if such a thing is not explicitly set for an account, CSipSimple can't really know if the provider/account has additional provisions for a rejected/not answered call (or unreachable client or whatever).

My suggested resolution is to unconditionally switch to sending 486 the call is being rejected by the user as I feel 603 is flat out the wrong thing to do here; however, if for any reason this is not possible or in your opinion a reasonable thing to do, I'd like to ask for an option where the call reject code can be specified.
