csipsimple - issue #1490

Entering digits in TXT mode

Posted on Jan 11, 2012 by Quick Giraffe

What steps will reproduce the problem? 1. set dialer to use TXT mode 2. change keyboard to '12#' and enter a digit 3. after each digit the keyboard returns back to 'abc' mode

What is the expected output? What do you see instead? Switching between '12#' to 'abc' should be done by user and not automatically after entering every new digit.

What version of the product are you using? On what operating system? Latest nightly build - r1184. Android 2.1

Please provide any additional information below. With the new keyboard setting it's very difficult to enter series of digits (e.g. <sip:1234567@domain.com:5080>). Keyboard switches back to character mode ('abc') after entering every new digit... I need to manage it manually. When I want to enter text, I use 'abc' mode, when I need to enter numbers, I expect keyboard to be in '12#' mode until I'm done with the number.

Comment #5

Posted on Jan 12, 2012 by Quick Hippo

About file size in fact, it's due to the fact that nightly builds are now built in debug mode (which deeply increase size) and about the fact it includes many stuff for video (there is ffmpeg inside that is not necessarily stripped down yet).

So don't worry about final size. The first point is that nightly builds will be soon back to be done in release mode. (I do that for now in order to have a complete feedback when it crashes). The second point is that video feature will be available as a separate plugin. There is already all technical points OK to do that. It's now just about integration issue (create a new package and add it to build process). It will land soon, and according to my first analysis it will almost come back to previous size.

There is also another way to reduce again the size of the app, it's to do a custom build just for your target CPU architecture. Since android support 2 CPU archs officially for now (armeabi and armeabi-v7a), for now I include the 2 versions of the library. So the size of the native part is almost twice bigger than what it should actually be. Previously I did automatically builds for only armeabi or armeabi-v7a to distribute optimal size package. I may also add soon to my automated build system this again.

So don't worry size is still a big matter for me. And that's known drawback of recent builds to be heavy and things are under control to keep size not so huge.

About your second point can you try to collect me logs for this use case? Also, the upgrade test is done automatically each day at launch of the application. If no updates are available, no popup will be shown. If you go in help > update nightly, it will force a check. But if no updates are available nothing will happen.
