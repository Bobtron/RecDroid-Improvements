Posted on Nov 26, 2010 by Quick Hippo

I've done an update that could help to debug the problem :

Could you try to install http://code.google.com/p/csipsimple/downloads/detail?name=CSipSimple_0.00-15-16.apk .

It include 3 hidden settings. To activate it : Go in Settings > press menu and switch to "Expert mode". Then in Settings > Media scroll to the bottom. The 3 settings are : Use Routing API => It enforce to use the old API that is deprecated in android 2.0. Some ROM however are based on old audio drivers but are > to 2.0. In this case activating this setting can help Use Mode API => It enforce to use something that is actually not an API but that is considered by some manufacturer to be the way to route audio... It's a very bad idea to use that on some device cause it use mode_in_call that directly plug GSM chipset to audio hardware. But sometimes on some devices that's the only way to get routing goes through earpiece. In Call Mode => (Useless if you use "Mode API"), Force to use one audio mode. On most device should be set to Normal. But sometimes could be interesting to test others.

Comment #7

Posted on Nov 28, 2010 by Quick Hippo

Ok maybe my test and my developer experience on the galaxy S may help you to understand what is happening on your phone. I was able to test on a galaxy S i9000 of a colleague. Here were my conclusions : * If I set audio mode to IN_CALL (which is done by "use Mode API" when trying to use earpiece), on the samsung driver : - They route the audio output to earpiece - They route micro directly to the GSM chipset (so that android application doesn't get any audio stream from micro). As consequence choosing IN_CALL mode or Mode API is not a good idea for galaxy S (i9000). * If I set audio mode to NORMAL, on the i9000 : - They don't change the routing for output - They route properly micro. - As soon as I try to use routing APIs or new API to set speakerphone mode, it route to speakerphone and it's impossible to get the routing back to the earpiece only unless I set again mode to IN_CALL.

As consequence my workaround that actually works for galaxy S is to first set mode to IN_CALL and then set to NORMAL. So on Galaxy S I get something working with Use routing api disabled, Use mode api disabled and in call Mode "NORMAL".

What is a little bit annoying is that on your phone that is also a Samsung it doesn't behaves the same. I think that setting back to normal re-route audio to rear-speaker.... I so my workaround will not work, and I fear that nobody using the android audio api will succeed in doing something that works on this device. If so, users should cry to Samsung and say them that's a petty to do a device that doesn't respect android API ! I've already done a lot of workaround to get things works on their driver. (Even in the core glue with the driver there is very strange and not useful things done is this driver, for example the audio client must ensure that it doesn't ask for frames to the driver at a rate higher than the frequency and if so, the galaxy S driver just behaves a really crappy way).

The first opened issue was about the fact it was not possible to route through speaker. So I guess that the first opening issue was just that it was not possible to route through rear speaker (but that it actually worked through earpiece).
