Comment #2

Posted on Mar 26, 2011 by Quick Hippo

No that's not a bug. It's something missing in your configuration.

I'll try to explain you what is happening and give you keys to solve the problem.

  • You are probably trying to use expert account to configure csipsimple and have probably removed "proxy" field content.
  • Your sip server does not support TCP transport.

As consequence what is happening : when you dial sip:x@x.com it use standard SIP to contact remote side. As there is not the proxy field with ;transport=udp which is the case when there is a proxy field with transport set to UDP, as per SIP RFC pjsip/csipsimple use UDP by default, excepted if the UDP packet size is too big ! (it do that cause routers does not like heavy UDP packets). And unfortunately here is the problem in your case UDP packet size becomes too big. As consequence it fallback to UDP and your sip server probably does not manage this very well.

I think that when you tried to remove one of the two PCMa/u codec you have reduced packet size just enough to get pjsip always choosing UDP. You could have removed any other codec I think that you'll get exactly the same result.

So now as for solutions... * Well I bet your sip server does not manage all codecs that CSipSimple can handle. For example silk is probably not supported by your server (or if so your are very lucky ;) ). As it's not an Asterisk with the bug of speex multifrequency, you can also choose only one frequency for speex (depending on your bandwidth). All this codec choose will reduce packet size. * In nightly builds there is a new coool option.. In settings > network > use compact form. It will activate light form for sip headers and SDP announce and will drastically reduce packet size. I guess that's supported by OpenSER (your server). I do know yet how widely this option could be supported by sip providers but it's in SIP RFC. I'll probably do a pool to see if I should not activate it by default. * Last point you could use your sip provider as proxy (if they support it... usually they do either on the same server than the registrar or on another server).

Hope it will solve your issue. Let me know how it goes but regarding logs I'm almost sure ;).

Comment #7

Posted on Apr 29, 2011 by Happy Panda

@callum: I just had the exact same problem as you and now it's working.

I don't know what exactly made it work, among all the settings I changed in CSIPSimple, so you may want to try all of these: * Create a new account using the "Basic" wizard. * Then change the wizard that account to "Advanced". Set the Caller Id and the proxy (proxy.localphone.com). * Go to "Media" -> "Codecs" and move "PCMA" above "PCMU" (in both "Wideband" and "Narrowband"). This is the default order in SIPdroid.

Also note that I have STUN enabled (using "sip:stun.ekiga.net") but you might not need it.

I found it very hard to get CSIPsimple to work with Localphone, compared to other desktop and mobile SIP clients I've tried, but I have to say it seems like the best client for Android. :)
