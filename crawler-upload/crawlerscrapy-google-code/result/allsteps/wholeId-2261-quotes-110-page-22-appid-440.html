Comment #1

Posted on Dec 2, 2013 by Quick Hippo

This something that happens very often. The reason behind that is not the app nor the server. It's the network / configuration.

Let me explain why : First thing to know is that many network routers limits the size of UDP packet they transmit. By default, SIP uses UDP. When an INVITE is made to the server, the app has to announce everything you request it to announce. This includes the list of codecs you selected and the ice candidates resolved using STUN or TURN.

This limit is unfortunately not something we can predict, but when it's reached, the visible result for user is a 408 timeout. This means that the app sent something but never got any reply from server (in this case because the packet never reached your server as was dropped by some network router in the middle).

There is 2 possible solutions to solve this problem : * The first possible one, is to use TCP transport (your sip server has to support TCP). To do so, convert your account config to advanced wizard by long pressing account row + choose wizard. * The second possible solution if you want to keep UDP is to try to reduce packet size. To do so, there is mainly 2 things you can do for that : > Use "Sip Compact mode" (in settings > network); > Reduce number of codecs announced (csipsimple supports a lot of codecs, but usually, once your server is configured you'll always use the same, so it's not necessary to announce all)

Thank you after more digging you was totally correct about UDP and packet size.

At first my asterisk logs give me the feeling it was receiving the first register request normally but after sniffing tcpdump I found second packet of register is not full and asterisk was never showing this packet in the logs.
