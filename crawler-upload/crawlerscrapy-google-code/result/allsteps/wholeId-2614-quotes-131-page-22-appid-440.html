csipsimple - issue #2238

SILK Sampling rate should be changed in real-time.

Posted on Feb 10, 2013 by Grumpy Bird

What steps will reproduce the problem? 1. If SILK 24,16,12,8 are enabled on both phones. 2. Start the connection with SILK 24 if the bit rate slows down the sampling rate should be changed to 16,12 or 8 but this is not the case.

The SILK draft states http://tools.ietf.org/html/draft-vos-silk-02

"Regardless of the mode it operates in, the encoder accepts input signals sampled at 8, 12, 16 or 24 kHz and allows the input sampling rate to be changed in real-time. Internally, the encoder resamples the input signal to a lower sampling rate if the mode does not support the input sampling rate or if the bitrate is so low that reducing the sampling rate improves overall fidelity."

What is the expected output? What do you see instead? Sampling rate should be changed in real-time if the bitrate is so low.

What version of the product are you using? On what operating system? Galaxy Nexus 4.2

Please provide any additional information below.

Comment #1

Posted on Feb 10, 2013 by Quick Hippo

It's the internal rate that should change. Not the announced one.

The feature to actively adapt to actual available bitrate is not yet available in pjsip so this feature can't be added without big effort (and that will benefit all codecs -- not only silk). BTW, the real codec for which it makes big sense is Opus (that is somehow derivate of SILK).

However, the current implementation already supports if remote side decide to reduce the bitrate and the internal sample rate. You should have no problem with that (else let me know). As this part (treating remote packets at different input sampling rate) is made by the SILK implementation from skype.

The only thing that doesn't work is to detect available bandwidth and ask silk codec to change bandwidth usage. But if remote does, you will continue hearing something. Detecting this bandwidth is absolutely not obvious. It requires to get feedback from RTCP and involves complex algorithm to try to deduce from packet loss an accurate available bandwidth and to have it something convergent.

As this enhancement -to detect bandwidth- is already part of issue 2060, I merge the two issues.
