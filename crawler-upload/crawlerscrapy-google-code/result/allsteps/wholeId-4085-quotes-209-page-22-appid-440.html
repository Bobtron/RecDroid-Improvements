Comment #5

Posted on Feb 21, 2011 by Quick Hippo

As for your details I don't think it is a bug.

To use a codec, you must support it on your side BUT the OTHER side has to support the codec too.

What you describe is typical of the fact the other side does NOT support any of enabled codecs.

So the problem is not on CSipSimple side but on other side !!! The other side must support codec to decode / encode in this codec.... That's logic. I guess that you are trying to use a PSTN gateway somewhere that does not support the codec. If you try SIP to SIP with a server that does not do media gateway, it's almost sure that it will work properly if both SIP ends support selected codecs.

Usually most 'free' pstn gateway does only support PCMU/A (some support g729). But be really careful with G729. You have to had the right to place calls with it. It's a proprietary codec. And I do not provide the license to use it ! It's here just for those who already have a valid use license for it.

Comment #13

Posted on Feb 21, 2011 by Quick Hippo

On your device screen should not goes off on wifi. The keep awake while in call option as described in the FAQ should be on. Else you'll experiment bad quality on wifi.

On 3G, problem you experiment is different. First of all, check what is supported by the remote party. If you try to call with a codec not supported by the other party the SIP negociation will fallback to the first available codec supported by both party. If you disable all codecs excepted some codecs, you'll experiment direct hang up. That's normal, the negociation fail cause you announce the other part you support only codecs that it does not support.

So first of all get the list of what is supported by the other side. You can test, if you activate only speex 8kHz in csipsimple for example and left all other disabled and that the call directly hangup (red screen), it mean that the other part does NOT support this codec.

Once you have the entire list of supported codec by other side, you'll see what you can do with this remote side.

If, as I guess regarding all your comments, the remote party ONLY support PCMU/A, you can activate all codecs of the world in CSipSimple, if the other side does only support PCMU/A you will ONLY be able to communicate with him with PCMU/PCMA.

Any SIP client can do nothing with that. That's cause of the SIP server ! not cause of the client. If you'd like to use more codecs, use a SIP server that use more codecs. Some apps such as Fring and Skype use their own server as gateway between you and the rest of the world as consequence they can use codecs they want. To get the same thing in SIP, you have to use a SIP server that provide same level of feature. Again the SIP client can do nothing with that !!

As consquence, if you want to use SIP over 3G, you'll usually need a SIP SERVER that support more codecs than just PCMU/A which use a lot of bandwidth. I can't do nothing on my side if your SIP server does not support other codecs !
