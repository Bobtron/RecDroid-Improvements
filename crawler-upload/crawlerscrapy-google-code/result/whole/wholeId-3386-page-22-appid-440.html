csipsimple - issue #1410

Improper Handling of Negative SIP Reponses

Posted on Dec 2, 2011 by Helpful Cat

What steps will reproduce the problem?

  1. Register with your SIP provider and perform any action that will result is a 5xx SIP response. In my case, using OpenSIPS, I added my user account to our "PSTN-disabled" group. This prevents the user account from accessing the PSTN. Attempts to access the PSTN result in a SIP 503 Service Unavailable message.

  2. Trigger the 5xx response - in my case I simply dial any PSTN number to trigger the ACL in step #1.

What is the expected output? What do you see instead?

I would expect CSipSimple to provide some sort of indication that the SIP router returned a negative response. Currently, CSipSimple provides no response whatsoever, and gives the appearance that the application is broken.

In addition, these "failed" calls are [sometimes] written to the call log, but without any meaningful data since the call never set up properly. I would expect that an INVITE for which a 5XX response was generated would not be placed in the call logs.

What version of the product are you using? On what operating system?

CSipSimple r1099 - Linux / HTC Droid Eris

Please provide any additional information below.

I can give you SIP credentials to our SIP router with PSTN access disabled if you need to test against a SIP router that can return 5xx errors. If you want SIP credentials then send a private message to me at "phazlett at gmail dot com".

Comment #4

Posted on Feb 20, 2012 by Helpful Cat

I found another problem related to negative SIP responses. It turns out that CSipSimple is not adequately handling SIP messages with Q.850 reason codes.

If CSipSimple receives a BYE or CANCEL with Q.850 reason codes it treats the message as a normal BYE or CANCEL. But it really should look at the Q.850 reason and handle it properly.

A typical BYE message with Q.850 data is below and the header of interest is the "Reason" header. The example below is a result of dialing an invalid international number 01160123 where 011 is the international access code and 60 is the country code.

BYE sip:2008081234@216.XXX.XXX.XXX:38004;ob SIP/2.0. Record-Route: . Record-Route: . Via: SIP/2.0/UDP 72.XXX.XXX.XXX:5060;branch=z9hG4bK8e1a.87df8233.0. Via: SIP/2.0/UDP 172.9.4.1;received=172.9.4.1;rport=5060;branch=z9hG4bKFSacNZaFaH59S. Max-Forwards: 69. From: ;tag=KN4DUXt14Ueca. To: "BOB MOTTS" ;tag=nznIGZM9NqVXOF75lNaLmZGsbrLVBtSA. Call-ID: 5eVOR9mYNZDVgHsg-jEvbSlWlG-ZNRDh. CSeq: 24560582 BYE. Contact: . Allow: INVITE, ACK, BYE, CANCEL, OPTIONS, MESSAGE, UPDATE, INFO, REGISTER, REFER, NOTIFY, PUBLISH, SUBSCRIBE. Supported: timer, precondition, path, replaces. Reason: Q.850;cause=28;text="INVALID_NUMBER_FORMAT". Content-Length: 0. .

CSipSimple does not indicate to the user why the call was not completed. It simple returns to the dial pad activity. I'm guessing that not all Q.850 codes need to be handled but a few seem rather important; specifically:

cause=1; UNALLOCATED_NUMBER (maps to SIP 404) cause=2; NO_ROUTE_TRANSIT_NET (maps to SIP 404) cause=3; NO_ROUTE_DESTINATION (maps to SIP 404) cause=17; USER_BUSY (map to SIP 486) cause=18; NO_USER_RESPONSE (maps to SIP 408) cause=19; NO_ANSWER (maps to SIP 480) cause=28; INVALID_NUMBER_FORMAT (maps to SIP 484) cause=31; NORMAL_UNSPECIFIED (maps to SIP 480) cause=487; ORIGINATOR_CANCEL (maps to SIP 487)

A complete SIP-to-Q.850 mapping is available at http://wiki.freeswitch.org/wiki/Hangup_causes
