Comment #11

Posted on Dec 19, 2010 by Happy Bear

I have tested the revision 427 more extensively on WiFi now. The change to the audio software has definitely much improved things, but the problem has not completely gone away.

In CSipSimple_0.00-15-10 and earlier Market versions, the popping would most often occur as follows: normal audio for 5-10 sec, followed by a rapid sequence of 4-8 pops, followed by normal audio for 5-10 sec and then rapid pops again. The pops (as heard by the recipient of the call seem to be worse (or be triggered by) when the persion initiating the call (me with the Galaxy S and CSIPSimple) is talking - ie CSIPSimple is having to packetise audio.

In revision 427 the big improvement is that the pops occur much less frequently - only about once every 45 - 60 seconds, but the basic phenomenon is the same: normal audio for 45-60 sec, rapid sequence of pops, ... repeated.

Interestingly, this primarily occurs when the call is pure VoIP - VoIP, within the domain of one VoIP provider.

When the call is VoIP - PSTN, the effect is mitigated somewhat by the way the media gateway between the VoIP domain and the PSTN behaves. This appears to be different to the way two SIP mobiles interact. Instead of hearing popping sounds, the call recipient hears media interruptions every 45-60 seconds which is much less distracting. When the initiator is not actually talking, the call recipient does not hear the media interruptions at all. All in all it leads to a much better experience as the recipient doesnt notice the problem nearly as much. My guess is that the underlying fundamental problem is still there.

My guess is that packets (or buffers?) are getting lost from the CSIPSimple packetisation somewhere and that this is causing the popping sound (or media interuption). I can exclude internet connectivity problems, because if I use two Nokia E90s (with built in SIP stack) at both ends of the call, using the same ISPs, there is no popping or media interruption. The call is as clear as normal PSTN.

I have tried using both default 16khz and 8khz sampling. This does not have any effect on the popping issue. As you say, the 16khz sampling leads to a slightly better voice quality for the call recipient.

Hope that this helps to track down hte issue further.
