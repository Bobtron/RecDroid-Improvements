Posted on Jul 30, 2011 by Quick Hippo

Yes you can manually change the ptime in CSipSimple settings.

It's in ExpertSettingMode (see the wiki page about that) and then Settings > Media > ptime. By default as you noticed it's set to 20ms. 10ms should indeed be fine with many codecs. However it may consume more CPU and also maybe a bandwidth overhead (which can become significant). The good points are about packet loss cancellation and about latency. Bads are about CPU load and network load. 20ms is the default cause it's the default in pjsip. I never did advanced experiments on this parameter.

The only thing I noticed was that it would be interesting to have this option per network type (3G vs Wifi), like it's done for codecs.

About the nigtly to download : you could read this wiki page that explain what are purpose of each nightly build => http://code.google.com/p/csipsimple/wiki/HowToInstallDevVersion?wl=en

Comment #6

Posted on Jul 30, 2011 by Quick Hippo

About the native gingerbread SIP implementation, you should try in CSipSimple to change the audio settings. By default, (because most SIP manufacturer has just drop sip support in android 2.3), I disable the audio settings that are actually made for SIP calls.

You can manually re-activate these settings. (In expert setting mode) Settings > media > * mode for sip calls - choose IN_COMMUNICATION * micro source - choose COMMUNICATION.

These modes are known to try to enable the hardware echo canceller and put the audio driver in the state optimized for voice over IP calls.

I know that Cyanogen mods are very well implemented on this point and do respect what is in the Android OpenSource Project code. That's not the case of all manufacturer - and that's why it's disabled by default in CSipSimple.

So changing that you should have an implementation really closed to the stock SIP application. I did advanced tests about that and CSipSimple use same settings it's even a little bit better than the stock app because they rely on a less performant SIP stack. And besides they use something similar to what is implemented in trunk, they even didn't try to implement things by the new way I do in the gingerbread branch, which performance should not differe a lot cause as I said, no implementation (AFAIK) bind the new audio layer (OpenSL-ES) directly to the driver.

The important improvement in android 2.3 implementations is about the way the audio is opened when doing voice over IP. And this can be configured with audio mode and mic source.

What I call gingerbread in the gingerbread branch is something really different that may not have impact for now with known android implementations. It's about how audio frames are read/write to the audio driver, not about the way this one is open.

The source code of the stock sip application is available on android git that's why I can tell exactly what they do ;). However IN_COMMUNICATION mode and COMMUNICATION mic source are private api until android 3.0, that's why many manufacturers dropped the support of SIP and did crappy audio driver when they updated their devices to 2.3.
