csipsimple - issue #1539

Phone UI crashes after a couple of SIP calls; Motorola Flipout

Posted on Jan 25, 2012 by Grumpy Hippo

What steps will reproduce the problem? 1. Make a few SIP calls using CSipSimple 2. Hang up, return to home screen 3. Wait and do nothing for a while

What is the expected output? What do you see instead? - After some time, the phone's user interface crashes and the phone restarts it (blinking Motoblur logo). In one case, the phone even was stuck in this state, and I had to remove the battery to reset the phone.

What version of the product are you using? On what operating system? - CSipSimple 0.03-01 r1108, installed from market - on a Motorola Flipout (MB511), Android 2.1-update1, Blur_Version 1.19.8.MB511.Retail.en.DE

Please provide any additional information below. - Interestingly, the same problem occurs with Sipdroid (see http://code.google.com/p/sipdroid/issues/detail?id=678). Do the two projects share a common library, middleware stack, or similar?

Comment #1

Posted on Jan 29, 2012 by Quick Hippo

No the two sip stack are really very different between csipsimple and sipdroid. CSipSimple use pjsip (which is a native sip stack -- it is built in C codec and should run faster), while sipdroid runs a java stack.

So, it is more likely something with the device and its ROM. Many devices manufacturer does really crappy things and the audio layer is sometimes very buggy. CSipSimple and SipDroid sometimes reuse same kind workarounds to try to tear up the audio layer. We try to share this stuff because this is an headache to manage across devices.

On CSipSimple these workarounds can be configured in expert mode. So you could try to play with : http://code.google.com/p/csipsimple/wiki/FAQ?wl=en#Audio_routing_troubleshooting

Also what is known to impact audio routing is the activation of bluetooth (it's buggy in 2.1) -- and it can impact even if no device is connected, the simple fact to have BT turned on can lead to problems. Also, you should check if motorola didn't release any updates about that. They use to fix this kind of problem after the release of the phone (they did for the Atrix for example that was very very buggy on audio layer).

There is also settings you could try to play with such as "Focus audio" that could particularly leads to this kind of symptom. Regarding your description it occurs after several calls, so after several tear up of the audio. If the ROM is buggy and leak the audio focus for example, it can lead to have the phone in a state where there is too many holds on the audio focus. Normally both csipsimple and sipdroid release cleanly the audio focus, but if the rom is buggy...

Let me know how things goes. If there is some setting that help I can turn the default to be the setting you found for this specific device.
