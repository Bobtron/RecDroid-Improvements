csipsimple - issue #2109

Session timers (rfc4028) improvements

Posted on Dec 5, 2012 by Massive Kangaroo

What steps will reproduce the problem? 1. Make a SIP call using csipsimple over a cell network (4G, 3G, etc) 2. Disconnect the cell network (do not hang up) 3. csipsimple continues the call, until the Hang Up button is manually clicked.

What is the expected output? What do you see instead? When the 'Hang Up' button is clicked, because the cell network connection was already disconnected, it appears that the VoIP provider/SIP system did not receive a BYE for this call as the network connection was dropped by my cell phone. In this case, a 5 minute call was billed as a 6.5 hour call.

This resulted in the SIP system continuing to bill my account as the SIP/VoIP provider did not receive a BYE to transmit to the terminating end. They were billed for this call as well.

In cases in which our customers experience this, the SIP/VoIP provider highly recommends that csipsimple use session timers which avoid such conditions from impacting their account balances.

Session timers specify an interval (a minimum of 120 seconds) which the customer system and carrier checks to see if the current call is still active. If it is is not active, then the call is terminated immediately

What version of the product are you using? On what operating system? csipsimple 0.04-04 r1916 running on Android 4.1.1, Samsung SGH-T999 on T-Mobile

Please provide any additional information below.

Comment #1

Posted on Dec 5, 2012 by Quick Hippo

Normally session timer (rfc4028) is supported by the sip stack (pjsip) and left as "optional" (the default setting) by csipsimple. So, if the remote support session timers it should be active and working. If remote doesn't support rfc4028, it will obviously not work.

In CSipSimple, you can configure the session timers value (min-se/session expiration period) in expert setting mode (global settings > network > sip protocol). By default it's min-se = 90 and session expiration = 1800 (you can reduce it if you find it too high for your case -- especially if it's paid service)

If you need per account setting, I can give access to the option to force session timers in account setting (in expert wizard probably). But I fear that if it's not currently activated with your sip provider the only result you'll get is no call established at all.

(for the record; about the setting(s) I could add in the user interface : http://www.pjsip.org/pjsip/docs/html/group__PJSUA__LIB__BASE.htm#gaf352829cc383112c4ce4da58a4d0b37e --default is optional) So let me know if you are interested in these options, I can add this as enhancement. If I misunderstood something in the bug report and you are referring something different than rfc4028 or noticed it doesn't work properly let me know, it will be treated as defect but I would need some logs (see HowToCollectLogs wiki page)

Some side point : may I suggest that your sip server check RTP/RTCP streams and cuts the connection before when no activity detected on stream. The should be robust to sip clients that lost connection and doesn't support session timers (which is not the case of csipsimple). Session timers are mainly useful if you are not on media stream path, but sounds in your case you gateway to pstn network and are on media path. BTW, this suggestion is also made in introduction of the rfc4028 and standardized in rfc3550 as linked in rfc4028.

Hi,

Flowroute checked the logs on some calls and confirmed that the session timers worked:

"I was able to confirm that your system is properly sending the compact form of Session-Expires setting up call session timers."

So it should be fine to close this issue. The actual issue seems to be on Flowroute's side. They apparently can't implement RFS3550 because they use Direct Media Delivery (DMD) and they don't relay media (they only see signaling traffic). The call was properly terminated by the peering carrier, however, the billing system wasn't able to catch this. They still need to find a way to end an 'active' call that really isn't active (e.g., check if the SIP client logs out, etc).
