Sip Device Auto-configuration

Posted on Feb 19, 2012 by Swift Lion

Feature Request:

While not directly useful to me, It'd be really nice (especially if it were used in a corporate environment) to have a simple way of configuring up phones. QR Codes work quite well for some things (and maybe for csipsimple as well). Google Authenticator does this, but it has a small number of configuration varibales. But simply pointing someones phone at a qr code and it configures up the phone would be really nice in alot of cases.

Comment #3

Posted on Feb 19, 2012 by Quick Hippo

Great ! :)

It's now based on Content Providers (from nightly builds). So it is about insterting/querying/updating on a content provider.

You can do that by hand, or just copy the entiere com.csipsimple.api package in your app. There is definition of constants you'll need and some usefull wrapping method. You can have a look on this file : http://code.google.com/p/csipsimple/source/browse/trunk/CSipSimple/src/com/csipsimple/api/SipProfile.java

The content provider authority and useful uris are defined here. The class is also a wrapping class around content values and add some basic static method to call content provider to get account config http://code.google.com/p/csipsimple/source/browse/trunk/CSipSimple/src/com/csipsimple/api/SipProfile.java#746

Normally the api package is designed to be standalone and can be reused in another project. (almost) All the user interface of csipsimple relies on this interface. The only exception is video stuff where the surface is linked to the service directly. But else you can get inspired of all csipsimple user interface. I try to keep virtually possible to separate user interface from service which guarantee that the api allows to do almost everything (there is things I restrict for security reason like accessing to password however).

Another file you may be interested in is the json serializer/deserializer : http://code.google.com/p/csipsimple/source/browse/trunk/CSipSimple/src/com/csipsimple/utils/SipProfileJson.java in your case you will have to redefine projection (because as I said the password field is not publicly available).

Don't hesitate to join and ask on the csipsimple developer google group (http://groups.google.com/group/csipsimple-dev/)

If you want to commit your project directly in csipsimple svn (as a separate folder) I can grant you commit writes. However, I'd advise to use github instead. I already started some plugin projects for csipsimple on github because it should be easier to fork publicly.

Comment #7

Posted on Feb 27, 2012 by Swift Lion

I had the same thoughts originally as well but depending on the qrcode you can go quite high (original versions i think went up to about 300 chars, but i think these days the only limit is the resolution of your camera) I have a test qrcode i wrote sometime ago that has 1024 chars on it and even my old htc hero could cope with it.

Originally i was thinking you'd just have keywords encoded into the qrcode such as "server address" or a pickup url.

Also, i was thinking it need not necessarily be only qrcodes, content: urls could be shared in a variety of means such as email, nfc, etc etc.

I faced similar issues about how you secure that a little while ago when i was working on google authenticator project, cause the qrcode contains the secret key for the token - i.e. anyone who can read it gets your token.

I came up with a few ways of solving it, but for this particular project, i'd envisage either or both these could work (but in alot of ways its up to the provider of the config info): 1) encryption - encrypt the data in the code via a password (this could plausibly be your actual sip password in this situation or something sent oob) 2) one-time pickup via a url (this worked well for tokens as it was easy to generate new ones if the wrong person picked it up) but the url was along the lines of https://pickupaddress/gettoken?id= and adding authentication to that shouldnt cause problems.

What I ultimately mean though is the end product (configuration component) would be along the lines of: content://csipsimple/sipaddr1=address&sipaddr2=address&... or content://csipsimple/configurl=someurl or content://csipsimple/datatype=encrypted&enctype=aes512&data=

just to be on the safe side with encrypted data you could base64 encode it incase some mix of hardware/software had problems with non-ascii chars. As you say tho, that could be very efficiently done and 1024 chars (while its a large qrcode and a massive amount of data in reality) is plausibly manageable.

But, consider something like this: content://csipsimple/sipadddr1=reallylonghostname.sip.nsw.iinet.net.au:5060&username=somelongusername is only 100 chars Or, for the sip providers themselves it could be along the lines of content://csipsimple/provider=iinet&state=nsw&voipnumber=0299999999 is under 70 chars.

Sound feasible?

Comment #9

Posted on Feb 27, 2012 by Quick Hippo

What I have in my mind for now is something like that : * Sip server publish at some http url their global config as an xml file (dtd to be normalized). It should express everything required to configure a sip server : stun, registrar, proxy, presence of a sbc, realm, support of sip presence etc etc. It should also announce things that could vary from one user to another as "variable". I don't know yet if it should be expressed as "${username}", "${password}" in content of the xml tags or as attributes of the xml tags (the first approach would allow to share var names between two xml values entries). For example ${username}@${user_domain}. In this case the xml should also describe in one of the xml part what variable stands for and give a description for it and an example.

  • With that done, we can support on android some sipconfig(s)://sip_server_or_webserver/path_to_the_file.xml I think that we can consider registering for a scheme rather than from generic content provider with only an authority name. We be better if other apps than csipsimple wants to adopt that on android ;). Then we can consider that sipconfig should use http as app protocol and sipconfigs should use https. (or maybe as you propose to pass the url completely as an argument). So that when you reach this url on android you'll be prompted which app to configure with this account config. When the app will get this url it should be able at this point to automatically generate a simplified UI (a wizard actually) to allow user to configure easily his own account.

Now the tricky point is about credentials. Before your post I was simply thinking about adding that as args to the uri. For example : sipconfigs://myserver/generic_config.xml?username=toto&password=secret And to have the app automatically consider that in the xml file all entries of ${username} and ${password} has to be replaced. And if remaining vars are left in xml file to generate an user interface to allow user to enter remaining infos. But your idea of encrypting things could be really interesting. Maybe it's a good idea to tell that params should be encrypted as you propose.

BTW, we will have something that leave free sip provider to share these sipconfig uris as qrcode or just urls on their website. It will be also a good solution to have autogenerated wizard in csipsimple (and probably in all other sip apps even desktop).

I don't think it's too much work to do technically on csipsimple side. However, the big point is about the normalization of xml file. I really fear doing something that would apply only to csipsimple while it's something that is a problem for all sip server and all sip clients. That's the reason why I'd like to have some more "normative" process on this xml file. I saw some candidate formats (old windows mobile sip config files, some desk sip phone also have xml config files), but none are normalized and built by a community. So I'd like to see if we can get some feedback (for now here) from sip server implementers and if we can reach to some proposal about how to define a sip configuration. Maybe it already exists (I don't know how IMS solves that cleanly yet), but if not it would be a great enhancement for sip.
