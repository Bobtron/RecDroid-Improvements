Comment #4

Posted on Jan 15, 2011 by Quick Hippo

The first popup you get doesn't depend of configuration of apps. Actually no code from CSipSimple or from the native stack has been launched at this point.

The two apps registers for the intent "User want to make a call". As the two apps has the capability to manage that, it's asked to the user to decide which one to launch. It goes the same way with email application. When you want to write a mail, if you have a ROM with Gmail + a standard email client, you'll be asked if you want to continue with Gmail or the mail client, even if you have no google account configured or if you have no email account configured. At this point, it's android that talk directly to the user. The app doesn't know that and can't say 'hey I'm not yet configured, do not propose the user to use me'.

If you tried to install an alternative home application such as Laucher pro or AW Launcher, you observe the same behavior: pressing home key will launch android activity chooser.

If there is only one app that can manage the "intent", the user will not be prompted. (Was the case in previous android version if you had not Sipdroid installed or another sip application that also register the "outgoing call" intent).

In our case that's a little bit more tricky cause the intent can be "intercepted" by the application you choose, but the application can also decide to leave standard GSM telephony to treat the intent. So, once you select either stock SIP or CSipSimple, the SIP application will now execute its code and decide whether it should intercept the intent and propose the user to do something or let telephony app manage the call.

So, the missing thing for me once I'll have a gingerbread will be to integrate back the native SIP accounts so that I'll say to my users => "Always choose CSipSimple once in this android popup window" and then CSipSimple will dispatch the intent where it should.

Comment #11

Posted on Feb 13, 2011 by Quick Hippo

Yes probably. It was maybe an entry considered as a mix of internet address and phone tel number. So that it missed each one of my hooks. Actually, I could register for all CALL_PRIVILEGED intents, included tel: ones. But would mean to always have the android chooser even for gsm calls. You'll say that now with 2.3 it does not make a lot of difference cause anyway you'll have this popup because of the fact the stock sip app also register the same hook for normal gsm calls.

But would mean a worse integration for <= 2.2. (It's the way that was choosen by skype). Also it would lead to break the filtering rules feature. So I think that the good choice for now is to : * Register CALL_PRIVILEGED for sip: datas. * Register OUTGOING_CALLS for tel intents (that come later after the tel app treats the phone intent). As this one is a broadcast it can be registered silently by an app, without conflicting the stock app (cause obviously it's the way the phone app manage external apps hooks).
