csipsimple - issue #706

Calls don't go/come through after registered for a while

Posted on Feb 10, 2011 by Happy Giraffe

What steps will reproduce the problem? 1. Update or install version 0.01-01 r615 (latest market release) 2. If installed for the first time, use wizard to set up Gizmo5 and Sipsorcery accounts (or leave everything the same as before if you've just updated) 3. Register with either account and leave it for a while (I haven't done enough testing to figure out how long or if the timing is even consistent)

What is the expected output? What do you see instead? Csipsimple continues to say that it's registered with the account, but something on the back end appears to time out, since calls do not come through. In order to fix it, you must unregister with your provider, then re-register. It seems that this refreshes whatever had timed out, since calls now come through just fine.

What version of the product are you using? On what operating system? Version 0.01-01 r615 Nexus One Cyanogenmod 6.1.1 (Android 2.2.1)

Please provide any additional information below. I've been an avid csipsimple user for a long time. I have never seen this problem until updating to the latest market release. It seems like the problem may be caused by csipsimple not properly keeping the registration alive. Csipsimple shows that it is registered, but calls do not come through.

Comment #3

Posted on Feb 19, 2011 by Quick Hippo

There is several settings that impact the re-registration :

  • The actual registration timeout : By default if you use basic wizard it's 300 seconds. But when registration is done SIP server can decide to reduce it. You can also choose to use expert wizard and change the default value. See ExpertSettingMode wiki page to know how to change the wizard for your account.

  • The keep alive interval : As most of the time network equipment may close open UDP sockets that are open for long time without activity, when registration is done there is some small keep alive packets that are sent to the SIP server. The time network equipments close socket obviously depends on the equipment, as consequence this keep alive value has to be tuned to on one side keep you connected and on another side do not drain the battery. By default CSipSimple is configured to send keep alive packets on wifi each 100s and each 40s over 3G. As almost everything in CSipSimple, it can be tweaked :D. You should also read the ExpertSettingMode wiki page but about general setting this time (cause keep alive is not linked to account for performance reason -- previously it was, but now, not anymore). Then go in Network settings, and you'll see two entries for 3G and wifi keep alive values.

Maybe tuning it will help. Or maybe changing registration timeout using expert account wizard.

If you get interesting results let us know. But actually this kind of issue really depend on the network in front of you, of the sip server. So that's pure configuration. On networks I use, with default values I do not experiment any problems.

If users massively get me feedback from a country that for example keep alive value should be changed, I can change default settings for the country. Or if some sip server registration timeout is too big while network usually require shorter values I can also create a wizard for your SIP provider that automatically set registration timeout to a better value.

For example, previous default registration timeout was 1800 instead of 3000 but I changed to get lower battery usage. Maybe I should change for gizmo5 and sipsorcery for 1800 as it was previously?...
