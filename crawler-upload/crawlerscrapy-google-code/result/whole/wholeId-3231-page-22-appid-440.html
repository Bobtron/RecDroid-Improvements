Comment #1

Posted on Feb 18, 2012 by Quick Hippo

If outgoing calls works it means that the account is properly registered and that your authentication works. So that's a good starting point :)

Then about incoming calls it's possible that the application does not register with the good number. What wizard are you using to do your configuration? Basic wizard or Expert wizard?

There is two possible root cause : 1- it could be about the username that is different from the authentication name. It often happens on enterprise configuration where authentication is made against a corporate directory while sip username is the phone number. You are maybe in this case regarding since in your description identification name and username are different. I guess you have 3333 as username and something different for authentication. In this case here's the process to configure your account : * First create it with basic wizard. In identification put the authorization name, in server 192.xxx.xxx.xxx, and the password. The account name can be whatever you want it's just about the display inside the application. * Save the account * Long press the account row, select choose wizard and choose "expert". * Then in sip id field, change from to Save, now it will register as 3333 on your server and maybe it will help the server to find you back.

Another possible root cause is about the ip the application announce. In your case I guess that you are directly connected in the LAN network. So the application should annonce to your server the private IP. By default it should be the case, but it could be broken if - There is another public account registered (in this case the app may detect a different public ip and try to register your sip server with that ip) - You activated STUN or other NAT traversal options that may resolve your ip as the public ip. In this case, the app may announce to your server the public ip of the phone, which will not be resolved by your company sip server.

Let me know which case it is. If the second one and you have several accounts configured, there is possible workarounds :)

:) yes indeed. I've added in recent versions in the description of the field the fact that @domain should not be added ;) (I think that's after the stable release, but it's in nightly build).

I try to reduce uselesss user input as much as possible.

However, there is no extra check because some users actually has a "@" in the username itself. For example "user@dom1@sip_domain" (actually in this case the first "@" has to be uri escaped which is done by the app)
