Comment #3

Posted on Feb 12, 2013 by Quick Hippo

Hi guys,

Well the problem is with the public announced RTP address.

Basically how does csipsimple resolves this address (you can find that on other issues and on the csipsimple-users group too as has been said several times already but I'll repeat it here).

  • By default the app will resolve using the interface that should most likely to be used for incoming packet. This is NOT linked to the interface used to reach your sip server. In SIP the control layer and the media layer are DIFFERENT and that's the reason why resolving the "public" interface to announce to the remote part should be independant. As consequence, to make it short, it will more likely resolve on your interface used as default gateway. If all your internet traffic goes to direct connection instead of VPN in your current topology, it will be the internet interface that will be resolved. It's just absolutely perfect if in your topology your remote are also on internet. But become a problem if your remotes are on internet (or if your sip server is a media gateway).

So to solve that there is a VERY CLEAN way to do in SIP. Instead of forcing the annonce of one of the interface in the SDP and getting headaches trying to configure with static ip addresses and so on, there is something dedicated to multi-homed situations called "ICE". This is supported by csipsimple and must be supported by your remote part to work normally. Using ICE, the application will resolve all possible interface (private using interface, public using stuns servers configured and potentially turn relay if any configured). Then in the SDP you'll see a lot of candidates corresponding to all possible ways to reach this client. The remote will then consider these remotes candidates and tries each candidates until finding a valid one. This solution is the rock robust one you should prefer.

For some reason if your media gateway or remote peer doesn't support ICE, you can in csipsimple using ExpertSettingMode (the account part of the wiki page), force an RTP announced ip:port. This obviously means you can predictively known what is the IP address obtained by your device (this can be done by adding some static dhcp entries to your vpn dhcp server).
