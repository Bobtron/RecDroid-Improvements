csipsimple - issue #2044

[wizard] Optimum Lightpath Voice

Posted on Oct 29, 2012 by Happy Lion

What steps will reproduce the problem? 1. Optimum Lightpath Voice requires Expert wizard configuration 2. 3.

What is the expected output? What do you see instead?

Wanted to be able to configure client without blindly plowing through complex and irrelevant configuration trying to find the right one (probably impossible without being able to see the network dump)

What version of the product are you using? On what operating system?

Android HTC Sensation running 2.3.4, csipsimple version 0.04-04 r1916

Please provide any additional information below.

With basic or advanced generic wizards, the optimum lightpath voice SIP server returns 403 Too Many Contacts (there are two, one for the public (natted) IP address of the system and one for the private (real) IP address of the system. On the Expert wizard I was able to deselect "Allow contact rewrite" which allowed the registration to pass.

Note that there are apparently multiple server IP addresses in use at olpv and you have to use the correct one. I am aware of registerh.optimumlightpathvoice.com and registerp.optimumlightpathvoice.com. The second one is the one that works for me, but I am given to understand that other customers needs to use the former. I have no specific knowledge about whether more servers might exist.

This is a US provider for corporate voip installations (so, for example, I had to install a filter rule to prefix the call with "9") and the icon is available at http://www.optimumlightpathvoice.com

Note while I was trying to submit this ticket, it told me "Issue attachment storage quota exceeded". FYI.

Posted on Oct 29, 2012 by Quick Hippo

Thanks a lot for the feedback and very detailed information for the wizard.

About the issue attachment it's known problem. We have reached google's limit here. But not a big deal as I prefer users to not publish their logs publicly : it's better to send by mail using the dedicated tool inside the app.

About the "Allow contact rewrite". Maybe changing the "Contact rewrite method" can also solve your problem. If your sip registrar keep the two contacts it's more likely because it doesn't support the SIP valid method to unregister and register in a single register request (with old contact with expires=0). Some SIP registrar do not support that while it's SIP valid. If SIP registrar doesn't support that they might be interested to know. If you set the "Contact rewrite method" to 'legacy' it will first send a regular unregister (a register with expires=0 and only that in the request) and then a second register with the new IP (the valid one as seen by the server). In case of buggy registrar it helps because they usually support simple unregistration.

If it works I prefer to use this method rather than disabling totally 'allow contact rewrite'. Having the correct public IP address registered is often better in terms of reliability.
So if you can test that and let me know how it goes will be great :)

An alternative to all this weird dual registration process is to have a stun server configured. Do you know they provide a stun server on their network (or advise to use one). Some sip provider have SBC instead which also solves NAT problems, so it's not always good to activate it.
