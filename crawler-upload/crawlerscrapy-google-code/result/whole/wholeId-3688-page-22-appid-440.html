Comment #2

Posted on Jun 30, 2011 by Quick Hippo

Yes, thanks a lot for the feedback.

Indeed, that's a known problem in pjsip sip stack.

Actually here what's happen :

  • pjsip does not support multiple codec at a time for a media session once established.
  • As consequence, in order to fix that pjsip guys decided to do an re-invite or update (according to what is announced in capabilities) (which is absolutely compliant with SIP RFC !!!) in order to select only one codec when the remote party decide to establish communication with multiple codecs.
  • This only occurs on outgoing calls (cause obviously it's in this situation that it's up to the remote party to decide which codec to choose). For incoming call, since pjsip decide which codec to use, no problem at all!

About remote party that are not compliant with this part of the RFC, you probably use an old Asterisk version cause AFAIK, this is fixed in their latest version (obvisouly, if it's the asterisk server that decide of the content of the sdp and of the media negociation... if it's transparent for media negociation, the problem is on the UA ).

You should have a look to this comment : http://code.google.com/p/csipsimple/issues/detail?id=810#c13. and this one : http://trac.pjsip.org/repos/ticket/476 (for pjsip project).

So.... to fix that... unfortunately it would mean to change pjsip, and that's not relevant of csipsimple project. I try to keep the stack as provided by the pjsip project in order to make things easier to maintain. I can however raise the issue to pjsip guys and see what they think about that.

Besides, keep in mind that what pjsip does is compliant with RFC ! So the "bug" is not on pjsip side, but on the side that does not support the re-invite / update. Or with the configuration of the server in the middle that tell UPDATE is supported while it is not supported on the UA endpoint, and that the server is not the media gateway (I mean it announce something for the other endpoint while the other endpoint does not support that...).

So first I'd advise to check the configuration and see if there is not something wrong on the configuration. (For example if you can take some Tcpdump for your sip server <-> csipsimple + sip server <-> grandstream. And have a close look to capabilities (UPDATE capability), and what is sent from each endpoint about this re-invite.

Besides, last point, if you have this issue, it's not specific to csipsimple, pjsip is a widely used sip stack, and if you get the problem with csipsimple, you'll get that with many other sip clients (there is a lot of sip client for iphone based on pjsip, and even on android, 3CX softphone is also based on pjsip -- even if this one is not opensource).

Keep me in touch with that, I'll be pleased to help to debug if needed. (Maybe the problem is in pjsip however, and if so, with full tcpdump I could raise the issue to pjsip guys).

Hi,

I was the one who replied to the mailing list. Actually is not a hack, sending a re-INVITE is perfectly fine as per the RFC. Also, support for it is more wide spread that support for UPDATE with SDP. Some servers will not even fix the Contact header for UPDATEs because they assume they will never have an SDP...

I'd change it to re-INVITE always, for the sake of interoperability. The attached patch implements that.

Comment #9

Posted on Jul 4, 2011 by Quick Hippo

Thanks a lot for the patch.

Actually the problem, which does not respect RFC, is the fact that the device announce to support UPDATE in the supported field, while it actually does not !

If the remote party does not announce to support UPDATE, pjsip will send a re-INVITE. And this behavior is absolutely correct.

I mean, if the remote side that does not support UDPATE, and does not announce that it support UPDATE, everything will go fine !

If remote side support update, it would be better to keep a way to actually use UPDATE cause it use less bandwidth. That's why I'd prefer to add an option instead of something hardcoded.

But, your patch will help me a lot cause it point out where to add the option :) Thanks a lot. I'll do that right now.
