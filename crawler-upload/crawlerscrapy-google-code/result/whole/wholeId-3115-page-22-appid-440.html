csipsimple - issue #1691

Use handset microphone to generate entropy for ZRTP

Posted on Apr 18, 2012 by Happy Monkey

I looked in the following file

jni/pjsip/sources/third_party/srtp/crypto/rng/rand_source.c

to find the source of randomness during an encrypted phone call. It looks like there are two options, first /dev/urandom is opened. If that file does not exist, the C rand() function is used.

It looks like openssl doesn't care /how/ entropy is generated, only that enough is generated in a private pool before it outputs the contents as random.

jni/openssl/sources/crypto/rand/md_rand.c

AFAIK, the contents of /dev/urandom is seeded at OS boot time. It would be a novel approach to use the input from the user's microphone as a source to fill up the entropy pool to get enough for the random source. PrivateGSM claims to use this approach, as does another proprietary ZRTP client author I've talked with.

http://www.privatewave.com/security/security-protocols/zrtp.html

I'm looking at source that is a little out of date. Apologies if this is already in progress.

Posted on May 20, 2012 by Quick Hippo

Interesting point. Well it's not in progress on my side. Also, it should be managed either in zrtp4pj or in ZRTPCPP (https://github.com/wernerd/ZRTPCPP ) project not in csipsimple. If done in this project, CSipSimple will almost automatically benefit of it. So you should try to ping zrtp developers there with an issue.

However, you have to know that it's somehow pretty complicate on android to get a reliable way the micro capture - the api is complicate and there in native space it's pretty complicate to get buffers from mic. So it would be some injected values from the rest of the app and may require some design complication on both libraries (pjsip/pjmedia that manages media and zrtp that manage encryption).
