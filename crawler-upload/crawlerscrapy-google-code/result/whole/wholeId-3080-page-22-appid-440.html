Comment #16

Posted on May 2, 2012 by Quick Hippo

No, other projects are not necessarily affected. SipDroid for example is almost pure java project and have very few native stuff. CSipSimple is almost all done of native stuff (for performance purpose and because the underlying sip stack is native).

Besides, even for native project they could not be impacted as well. The problem seems to be some optimization added by gcc (with __udivdi3) for some kind of divisions. Not all projects contains this kind of situation and so, not everything build with the NDK may be impacted. But for pjsip, it's the case because there is many place where this optimization could apply. So I'm trying to dive into the mips gcc to see how (if possible) to disable this optimisation introduced by this gcc. The very amazing thing is that this optim symbol seems to be built in the NDK (it appears in the libc.a).

Comment #17

Posted on May 2, 2012 by Quick Hippo

Well, a new build is available here : http://nightlies.csipsimple.com/specific_builds/CSipSimple-Mips4.apk

I trust more in this one (at least about the __udivdi3 fix).

I did some hard fix in the Mips NDK to workaround what appears to me either a NDK bug or a miss on the distributed android ROM that doesn't conform the NDK requirements. (I think more about the second option since the emulator doesn't has the problem).

If you can test it and tell me how it goes; Also in case of failure, I will be very interested in logs you could collect. I've hard included one core library to have it working, but I could still miss another one (since errors are raised one by one).

For the more technical part of how I workaround that in case it works and interest anybody else : since udivdi3 seems not present on the real device, and is part of libgcc, I just included libgcc as a static library dependency of the project (which means adding a new prebuilt target in the mk files and referencing this target as static_lib of each .so you build). This is workaround should only apply to this mips arch/device, since it's the very first time a device is encountered without respecting what is provided by the ndk as libgcc library. So care to make this dependency only active when arch is mips and not in other cases (or your builds will be unnecessarily bigger).
