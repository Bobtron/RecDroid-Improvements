csipsimple - issue #468

No audio during calls on Samsung Moment with Android 1.5

Posted on Dec 8, 2010 by Helpful Ox

What steps will reproduce the problem? 1. Install stock CL14 (Android 1.5, M900.8.0S.DE03, Kernel 2.6.27, CUPCAKE.CL14) http://www.mediafire.com/file/oytdgjlj5zi/update_Sprint_Stock_CL14.zip 2. Install nightly CSipSimple-r416-trunk http://nightlies.csipsimple.com/trunk/CSipSimple-r416-trunk.apk 3. Setup a sipgate account. 4. Call echo service (10005@sipgate.com) 5. No audio is heard from either the ear piece or loud speaker.

What is the expected output? What do you see instead? Audio to be heard from the ear piece.

What version of the product are you using? On what operating system? CSipSimple v. 0.00-16 (Downloaded from the Market) Android 1.5, M900.8.0S.DE03, Kernel 2.6.27, CUPCAKE.CL14, Samsung Moment

Please provide any additional information below. Changing the audio settings appeared to make no difference. Use Routing API Use Mode API Audio Mode to use (All four options).

I did get the audio to work once. I had just gone out to the OS and turned off BuzzLock and set the ringer volume all the way up. After that I went into CSipSimple and enable the Mode API, and set the Mode to Ringtone. At that point the audio for the call came out of the loud speaker. The sound was choppy and slow. Responsiveness of the application was slow as well. It took several seconds for the Hung Up screen to appear after hitting "End."

I'll do some more testing when I get home. I have the logs again as well.

Comment #2

Posted on Dec 23, 2010 by Helpful Ox

I also tried this method here:

As consequence choosing IN_CALL mode or Mode API is not a good idea for galaxy S (i9000). * If I set audio mode to NORMAL, on the i9000 : - They don't change the routing for output - They route properly micro. - As soon as I try to use routing APIs or new API to set speakerphone mode, it route to speakerphone and it's impossible to get the routing back to the earpiece only unless I set again mode to IN_CALL.

As consequence my workaround that actually works for galaxy S is to first set mode to IN_CALL and then set to NORMAL. So on Galaxy S I get something working with Use routing api disabled, Use mode api disabled and in call Mode "NORMAL".

It resulted in no change in the behavior. Audio was still routed out the back. Also, I tried initiating a call with IN_CALL, and then initiating a call with NORMAL. All the while the Use Routing API, and Use Mode audio API were disabled.
