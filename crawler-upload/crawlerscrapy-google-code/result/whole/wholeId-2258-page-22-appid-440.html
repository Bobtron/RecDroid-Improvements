Codec settings per provider separately

I have two providers which offer different codecs. Sipgate and Easybell.

I'd like to save the codecs provider-dependant. Please :-)

Posted on Nov 30, 2013 by Quick Hippo

Thanks for the suggestion ! I mark this issue as enhancement with low priority because it should not be necessary for good functionality (as codec is based on negotiation between 2 parts [1]) and as it might confuse a lot users to have a different list and makes configuration a lot more error prone and hard to explain for mainstream users.

[1] : normally a sip endpoint announces it's capabilities when emitting a call and pick up a codec when receiving a call. As consequence, even if a sip provider does not support a codec, it's harmless to announce it/have it enabled in the list. Actually the list should be more focused on the endpoint capabilities and cover at least one of those supported by provider/remote. By endpoint capabilities, I mean CPU capabilities and network capabilities. That's why the current feature is to allow to select a codec list per network. It's way more important than per provider as network in use is a constraint and codec should be announced accordingly. As consequence, in your case, you should not need a separate list per provider. Instead just build a list based on capabilities of both providers. Place the one that consume less bandwidth and/or have better quality on the top of the list and let the fallback ones on the bottom of the list. The codec negotiation will do the job to pick the better (according to your list) supported by the provider in use.

In short, for your case, the best codec list I would pick is : For fast networks : PCMA (aka G.711) GSM

For sipgate PCMA (aka g711) will be used; For easybell PCMA (aka g711) will be used. It good because PCMA has better quality than GSM. For slow networks (3G/GPRS/EDGE) : GSM PCMA For sipgate g729 or GSM will be used (it will use less bandwidth which is necessary over mobile data); For easybell PCMA (in this case PCMA over 3G, you might experiment bad quality). If you have the g729 codec license (which I do not advise for freedom reason but could be the case for pragmatical reasons ;) ); you can put g729 on the top of both lists and it will be picked for sipgate in both cases.

You could also enable codecs not supported by both better (like SILK) on top of the list. This way, if, for example, easybell starts supporting SILK you'll get calls using silk immediately. (I take this as example because, if you didn't know, the easybell android app is a forked version of csipsimple and they might decide to enable on server side things allowed by csipsimple ;) ).

With this technical points known; do you think there is need for this feature enhancement? Is there another reason I missed that would require the need a list per provider?
