Comment #9

Posted on Feb 7, 2012 by Quick Hippo

What could also happen is that there is too many registrations on the server and that the sip server refuse any new connections. Logs could enlight about what actually happens (see HowToCollectLogs wiki page).

To add a little bit more details : The app register for about 15min by default. If the network connection is lost, it may be not able to cleanly unregister. In this case the server will still hold one pending registration for remaining 15 min. If this is repeated many time, you can reach the limit of sip server registrations. And in this case, some server are configured to refuse any new connections. What could made things going even worse is if the sip server is a little bit buggy and does not understand inline unregistration that the app may do when it is in a NATed environment.

Anyway with logs I'll be able to tell much more things about what really happens. :)

Comment #22

Posted on Feb 29, 2012 by Quick Hippo

It's possible. Some sip provider do this kind of filtering.

To validate and test in csipsimple that (because there is already a setting for that ;) ), you can : Go in csipsimple settings and activate ExpertSettingMode (see the wiki page). Then, go in "Network" section and here (among other expert settings), you'll see the "User agent" option. By default it's CSipSimple, you can change to Anything else. Tell me if it helps (that's also something that I can automatically change from wizard)

BTW, if you have wireshark traces, I can also try to investigate on my side to see what should be settings to apply in the wizard automatically.
