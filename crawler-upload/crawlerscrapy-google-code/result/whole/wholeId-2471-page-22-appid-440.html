Comment #1

Posted on May 14, 2013 by Quick Hippo

From the logs it appears the app switches to TCP (probably because you used transport Auto in expert mode). A very quick fix would be to change that to force transport to be either UDP or TCP at choice as all other wizard except the expert one does.

Now, about why you have that : pjsip has by default a behavior to solve the fact the UDP packets can be limited by network. Some routers has a limit for size of udp packets and do not transmit packets in case it's higher than the limit. As consequence pjsip (and the sip rfc) advises to switch to TCP if packets size becomes to big. That's what happen during your call as the packet size of the invite with the authentication is bigger than the limit. So the actual invite will be made over TCP. Now... what happens, it's first time pjsip uses TCP and was never able to resolve the NATed ip:port corresponding to your TCP socket. So it can only send a TCP address that might not be reachable from the server as contact. Then the server will reply... it has two choice.. if they are able to reuse socket it will go fine... but if they try to open a new socket, they will never be able to reach the application ! So the application will never get the OK ! So the application will never send the ACK ! So what you observe is absolutely logical and is something that can be expected from your configuration and your network topology. That's the reason why the default wizard are setup to automatically avoid this situation (but that's still a choice for expert users that are normally aware about such behaviors in sip).

To be sure my guess is correct, you can also collect and see the logs from the application (see http://code.google.com/p/csipsimple/wiki/HowToCollectLogs?wl=en ). You'll probably see that the app behavior is logical regarding what it receive and how it's configured. If it's not the case, can you send me the logs from the app so that I can compare and analyze the situation from the app point of view.
