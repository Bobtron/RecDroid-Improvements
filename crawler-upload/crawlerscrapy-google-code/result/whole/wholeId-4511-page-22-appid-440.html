Comment #3

Posted on Oct 31, 2010 by Quick Hippo

Ok, so your issue is about the fact that dtmf codes are not properly understood by the remote party.

You could try to activate DTMF INFO in settings (not sure was already included in 0.00-15 from the market, but is in latest dev builds (go in Settings > media and activate SIP INFO DTMF). It will change the way dtmf are send to the other party : There is 3 methods to send dtmf. By default I use the two most common ways (send using rtp messages or falling back to standard analogic sound production if it fails). But there is a third method, that you can try. It use SIP INFO messages that is another way to send DTMF codes to SIP servers.

Another reason could be that the remote server announce that it support first method but actually don't. And so I don't fall back on the crappy method (probably used by your sip pc clients). A workaround could be to add an option to force the crappy sound production on my side... Let me know how it goes with SIP INFO DTMF and if it doesn't help I'll add this extra setting.

Comment #5

Posted on Nov 1, 2010 by Happy Rhino

I know where the problem lies.

IPKall doesn't support SIP INFO or MESSAGE for DTMF. To get DTMF through it, the tones must be sent in-band.

A possible solution is to change the SIP INFO toggle to a radio button with {Auto|SIP INFO|SIP MESSAGE|InBand Analog Tone Generator}
