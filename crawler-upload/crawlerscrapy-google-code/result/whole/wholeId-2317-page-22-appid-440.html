csipsimple - issue #2460

CSipSimple can't receive calls from Jitsi

Posted on Aug 10, 2013 by Helpful Rabbit

What steps will reproduce the problem? 1. Get two free SIP accounts, e.g. on ippi.fr 2. Connect a computer and a phone to the same wifi router 3. Set up Jitsi to use one account on the computer, and CSipSimple to use the other account on the phone 4. Use Jitsi to call CSipSimple

What is the expected output? What do you see instead?

Expected: the call works.

Actual: Both CSipSimple and Jitsi display the call as connected, but there is no audio output on either end. The "info" dialog on both ends shows that both clients have received zero packets, although both clients have sent packets.

This is not a codec negotiation issue, because both clients report that they are using the SILK codec.

What version of the product are you using? On what device / operating system?

Samsung Galaxy S3, Android 4.1.2, latest nightly build of CSipSimple from http://code.google.com/p/csipsimple/

Please provide any additional information below.

If I use CSipSimple to call Jitsi, everything works fine.

If I enable ZRTP in CSipSimple, it also works fine for outgoing calls, but incoming calls have no ZRTP as well as no sound.

I tried enabling ICE, but this had no effect.

Comment #3

Posted on Aug 15, 2013 by Quick Hippo

Yes, that's usually problem with network configuration.

For reference, why you got this problem :

Situation : - you have 2 sip clients on the same network. - you have stun enabled but one of the two clients does not support ICE. - you are not using a sip server on public network that does media gateway.

What happens : The app will try to announce to remote part a valid IP. When STUN is enabled it asks the stun server to resolve the "public IP" (actually the IP as seen by the stun server). If both sip clients are in the same domain and the router is not able to loopback, this IP is invalid for the second sip client. The valid IP for this sip client is the "private IP".

Solutions : - Have both clients with ICE enabled. In this case, the client will send more than one IP and the remote will try all possible IP. In this case, the app will announce the public IP resolved by STUN server(s) and the private(s) IP of the client. Obviously both clients must support ICE to have this solution functional. Else, this solution will solve almost all topology. OR - You know that you'll always reach another client in the same LAN... so disable STUN or use a STUN server that you install in your own LAN so that the IP resolved by the stun server is something valid for your LAN. The important thing to understand is that STUN allows to "resolve" the IP from a point of view. This point of view is valid for other SIP clients/server that are in this same network and is not valid for clients. For example if you are in a country that have a sandboxed "public network", the default stun server might not resolve something valid even for a sip server in public network of your country. OR - Have a sip server that also gateways media and have a stun server on the network than this sip server. In this case, the media negociation will always be between sip client and sip server (one for you and one for the remote part); and if a stun server is available with the same point of view than this sip server... it's sure that the point of view will be correct and the IP proposed the good one.
