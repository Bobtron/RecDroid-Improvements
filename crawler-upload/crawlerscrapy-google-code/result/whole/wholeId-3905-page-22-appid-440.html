csipsimple - issue #929

[PATCH]: add autoanswer in background and openURL filters

Posted on May 6, 2011 by Grumpy Elephant

Hello,
attached there are two identical patches ( one generated by svn diff and one by diff -Nur ).

The main goal is an addition to the filters usefull for a project i'm developing
( a video-citophone system ). Basically they add two filters:

1- Autoanswer in background: Same feature as autoanswer, but it autoanswer with the speaker on by default, and without make the client coming in foreground ( so, you can continue to see, for example, a web page. )

The main goal is, for me, to have a web page with a video streaming in it and a button that, when pressed, make an asterisk pbx call the android device, and let the user talk with a remote voip phone continuing to see the video on the web page.

2- openURL patch
It create a filter that permit a specific callerid to send a special SIP MESSAGE containing only a valid URL. When received, the client will open this URL in a web browser.

The goal for me is to open the video page of my videocitophone when someone press the external porter button of the citophone, so, in the webpage, i will send a "ring" using <audio> html5 tag or something similar, and
i will let the user see the video streaming from the citophone plus the buttons to answer and open the door.

The patches are against trunk svn revision 830.They are complete with filter_choices.xml res files, but actually the only two translations are english and italian ( the other files are filled with the english version to be translated ), and both new filters are missing an icon.

Comment #4

Posted on May 29, 2012 by Quick Hippo

After review, and with new API in mind, it should be done as a plugin of the app in an alternate InCallScreen activity.

The new activity of your app may just have to register in the manifest like that :

            <category android:name="android.intent.category.DEFAULT" />
        </intent-filter>
    </activity>

By this way, you can in your activity open a webview. You can also activate the speaker when you want; Even from a javascript API presented to the webview.

The only remaining step to have that seamless from user point of view is to give a way from the plugin app to force CSipSimple the use the new InCall activity. For now it will be prompted to the user that may choose one app to control his communication and could choose to always use the same app to control his communication. But such an evolution is planned as API evolution and will take place as a single new entry in filtering rules as "Answer with activity"; if more than the default of CSipSimple is detected.

It allows more extensibility than introducing each possibility of action that could be done while managing a call. And will also be lighter to process for the application.

Also, for reference, the option to answer with speaker on has been introduced as a global option and will affect mediaManager. It doesn't prevent the InCall Activity to override that at any time.
