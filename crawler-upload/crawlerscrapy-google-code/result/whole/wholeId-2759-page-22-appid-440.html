csipsimple - issue #2043

Wrong account identification on incoming call

Posted on Oct 29, 2012 by Happy Panda

What steps will reproduce the problem? 1.Create two accounts on Sipgate (I used Sipgate UK) and allow both to register 2.Make a call from elsewhere (a real phone line) to each of the accounts in turn. It doesn't matter whether or not they are answered

What is the expected output? What do you see instead? I would expect to see the name and icon displayed for the appropriate account/line number. So, for example, if I had accounts A and B configured with Sipgate, a call to the line corresponding to account B should show the icon and name associated with account B. Instead, I always see the name and icon associated with account A. This means that it's not possible to identify the line on which the call is arriving.

What version of the product are you using? On what operating system? CSipSimple 0.04-04r1916 on Android 2.2.1

Please provide any additional information below.

Comment #4

Posted on Nov 1, 2012 by Quick Hippo

Hi, Thx for the details.

Can you collect some logs (see HowToCollectLogs wiki page)?

Normally the sip stack tries to detect the account but it could be wrong in some particular cases. For example if the server doesn't sends the correct 'To', the stack will be unable to detect using the callee information and will fallback on the server that is sending the request. In this case, as it's the same for the two accounts it may always fallback on the same account. With logs I'll be able to see more clearly what's sent by the sip server and why it could break the detection of the incoming account.

Comment #8

Posted on Nov 4, 2012 by Quick Hippo

Well, after private analysis of logs, appears that my guess is confirmed. Sipgate invite the application with a different phone number than the one the account is configured with. In this case, the previous behavior of the sip stack was to match first on the domain name and match the account with highest priority.

I've just added a patch to sip stack that make matching a little bit more fuzzy/clever. It nows try to rank account based on username and domain matching.

Ranking is computed with following rules : ( 0 + 3 * domain name length if the same than the one of configured account + 1 * max length of common substrings of username configured and username invited by server ) * 100 if account has registrar

By this way it should be more robust in your case (as apparently sipgate just adds prefix).

It will still not match exactly, so go in the new fuzzy case; and here both account will rank for the server name equally and one should rank more than the other for username.

This fix should also help in other cases where domain name invited by server doesn't match the one we have registered to. Apparently very common problem with asterisk that are by default configured very weirdly.

This fix will also help with all features derivative from account detection such as auto-answer filters. It will be produced in about 30min as r1999 on nightly build server.
