Hi Daniel,

Thanks for asking, good reflex instead of being confused without asking...

The expected output is *75385423741@sipbroker.com

sipbroker service is something different than localphone service... You seems to me very confused with what sipbroker service is useful for. So let me try to explain why you should absolutely not expect that from a standard SIP client.

It's NOT necessary to use sipbroker service to use SIP to SIP calls ! It's NOT STANDARD SERVICE and it's not the way SIP protocol works !

It's just a facility you can use in any sip client but that's NOT to be hardcoded as a crappy automatic filter to rewrite in a client that has nothing to do with a service.... I don't want users to be forced using a specific service. They are free to configure things as they want and who they trust for privacy of their information.

SIP address are just like email addresses. You theoretically should be able to reach any sip remote party on any sip domain from any sip provider. Not all sip providers allows that however, but it's how it should work in ideal world.

As consequence of that ....

#1 : *75385423741@localphone.com is ABSOLUTELY VALID in theory (I mean from SIP standard point of view)!! It would be absolutely possible that localphone can handle this number !!!! For example ekiga.net can ! It's NOT a BUG to be able to dial that !! And it's expected that if you dial any number in the digit field it will transmit it adding the domain of YOUR sip provider. And not link you to any other service you don't want.... If things follows your think it would be a little bit like when you send an email to somebody whatever your email provider is, the email would use for example the yahoo.com server to proxy the request and do some rewrite to select the domain of your remote party whatever it is ! Makes absolutely NO sense ;) #2 : if you dial 5423741@camundanet.com using your localphone.com account, it will ask localphone.com server to transmit the request to 5423741@camundanet.com. If localphone.com doesn't allow to transmit to other SIP domain... you should blame THEM... They are just blocking you to send a request to another domain. They do a little bit like if email providers would block outgoing emails to other domains.... Blame them and use another sip provider for your sip to sip calls. The error you get telling it's a wrong number comes from localphone.com ! #3 : For your sip to sip calls you can use sipbroker service if you want... No problem at all doing that... but it's ANOTHER service... so another account !! You can easily configure it by creating basic wizard with sipbroker.com as server. AFAIK same kind of feature is also available on ekiga.net. You could also use sip2sip.info or any sip provider that allows to call other domains. If you choose sipbroker you can use their shortcuts with *xx

Also, another side note : you could be interested by the filtering/rewriting feature of csipsimple. It's for integration with stock dialer/contact app and allow to automatically rewrite numbers and should allow to address same features than the one provided by sipbroker but locally (without using sipbroker servers).

I hope that lighten you a little bit more on how sip works. Sorry for my english, not my native tongue. Don't hesitate to ask if something appears not clear ;)
