Comment #1

Posted on Sep 17, 2013 by Quick Hippo

The purpose of "partial wake lock" option is to enable a partial wake lock ALWAYS. If you do not enable this option the app will behaves a correct way regarding wake lock and just enable the cpu when it's needed.

Enabling the "partial wake lock" option is an expert option and should be done only if you know what you are doing and if you are in a situation where the phone is buggy regarding incoming packets (and it's the case for some phones).

I'd like to add that waking up things every 100ms is not a good idea at all because when you know how cpu lock works, you understand that a short cycle of wakes is exactly equivalent to always wake as the phone waits some time before going in sleep mode.

So nothing to do here. The current standard behavior is the good one, and if you don't want battery drain, do not enable workaround option that explicitly drain battery faster...

Comment #3

Posted on Sep 17, 2013 by Quick Hippo

Nexus 4 has a bug from google (it's known and accepted by google, there is an open issue on their tracker on their 4.3 builds). So the bug is not to be tracked on csipsimple. It's something that google must fix.

Meanwhile, the only solutions are (at choice): use udp with high value of keep alive use a custom rom with a fix from community (If I don't mistake some guys did a fix on this problem) use a short ka interval (that simulate exactly what you describe, but will not let phone go to sleep mode, so should not introduce big gain in battery savings). use the partial wake lock and have battery drain faster.

Unfortunately, no other solution are possible. Reducing wake lock interval, will actually do exactly the same than 100% wake lock (due to the way android manages cpu locks). Having to big interval will result in loosing calls (because usually sip server doesn't wait indefinitely for client to reply with 100 trying).

You can simulate a wake lock interval shorter by simply reducing the KA interval. Each time a KA must be send, the phone is wake up. But as I said, a too short KA interval will result in phone always wake up (so same thing than keep awake). A too long, in case of buggy nexus4 will result in call not received at all.

For more details about one part of the problem opened on google side: https://code.google.com/p/android/issues/detail?id=42272 It doesn't only affect csipsimple but all apps receiving tcp event in background... the shame is that it also affect google apps such as gtalk ! When I receive a gtalk notification on my galaxy s3 it's instant, on nexus 4 it has a very very long delay. For chat it's not critical, but for phone calls it is. The nexus 4 also have some problem with sip due to some bad contrack rule they added that prevent too big tcp packet. I guess you are not affected, but if you see one day a 408 timeout error while using tcp on nexus4, it could be this problem. Well... N4 is the first time google worked with LG, so maybe we should be indulgent, but weird they do not react quickly and provide fixes on these critical issues.
