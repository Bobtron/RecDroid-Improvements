Comment #1

Posted on Nov 14, 2012 by Quick Hippo

Hi,

How do you expect DTMF to be sent?

There is 3 ways to send DTMF in SIP. You can force each way in settings of the app. * The RTP way will pass over media stream with a specified RTP payload type. If you observe the audio stream with wireshark you'll not see that. You'll only see some particular RTP packets with a dedicated payload type (usually it's 101 but will depend on SDP neg) * The analogic tone generation will generate inside the audio media the tones generated with old way. It should produce some sound on the other side. This way will play well with legacy devices/servers or if there is gateways in the middle that doesn't supports the two others ways properly. However, if codec choosen in neg cuts high frequencies, it could loss information in the tones and not being able to be decoded properly on the other side. * The SIP INFO way. This will pass your DTMF on signalization sip layer. You should see in wireshark a full clear SIP exchange about an INFO request send from the app to the server.

Not all sip servers supports all cases. And worse, some announce to support some of the ways but actually just ignore it (or do not gateway it). So it's sometimes necessary to force in csipsimple settings one of the option. By default it will try RTP if the other side announce to support it (which might be wrong), and fallback to analogic generation else.

Comment #3

Posted on Nov 15, 2012 by Quick Hippo

Ok, so RFC2833 is what I called in last post the "RTP" way. It normally the default but it can be good to force it.

Normally, the DTMF in this mode is not linked to the audio layer, so it should not be impacted by the audio routing (rear speaker/headset). Also, the app doesn't feedback anything when you press the tone (because sometimes due to echo it interfere with the actual tone you send).

The tone feedback, if comes from somewhere, comes from the server.

So maybe could be interesting to check network quality and ensure there is not too much packet loss. You can click the "info" button of the call card to see the loss rate is big. Using more efficient audio codecs could relax bandwidth usage.

You can also try to upgrade to latest version. Can be an interesting test too (there is no specific bug fix for this problem, but it should be globally better regarding audio management) : http://nightlies.csipsimple.com/trunk/CSipSimple-latest-trunk.apk

As you are able to make wireshark trace I would also be very interested it dump you can collect. So that I can see RTP stream and check there is well something sent by the app. Send it directly by mail (developers [at] csipsimple.com) to me (publishing on public place is not good for privacy).

I forgot that but I added something to have local feedback of the dtmf tone. Normally it should have been optional and disabled by default but as it's a setting that is shared between the dialpad and the dtmf pad... it was probably later re-enabled.

However, it should not be a problem for most devices, in your case, the problem is that it's a Samsung and Samsung devices usually requires audio workarounds to get audio working I think the audio stream where dtmf tones are and the stream where is actual audio are different in your case.
