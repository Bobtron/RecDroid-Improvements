csipsimple - issue #2155

ZRTP setting

Posted on Dec 28, 2012 by Quick Giraffe

What steps will reproduce the problem? 1. Go to "Settings | Network | Secure transport | ZRTP mode" and set "No ZRTP" 2. Watch for INVITE message, SDP and find "zrtp-hash:1.10" attribute there.

What is the expected output? What do you see instead? If "No ZRTP" mode is chosen, there should be no ZRTP related attributes in SDP. Instead, there is no difference if you set it on or off - attribute is always there.

What version of the product are you using? On what operating system? r2063, Android v4.1.2, HTC Aria

Please provide any additional information below. I've noticed, that CSS starts crashing OBi100 ATA (which doesn't support ZRTP). I presume, that it's because of the new ZRTP related attribute in SDP. I can't check this assumption without removing that attribute by turning ZRTP off in CSS.

BTW, there is no reason to send that attribute at all (containing a long hash, making packets bigger) when CSS is not in ZRTP mode.

Comment #1

Posted on Dec 30, 2012 by Quick Hippo

There is two place for ZRTP setting.

One in global settings is the "default" value. If a sip provider is known to have zrtp feature it's enabled at account level. It can also be modified in expert wizard mode.

So try the following : Go in account, long press your account row, select choose wizard and pick the expert one (in the generic section). Edit the wizard, you'll find an option for ZRTP (global setting, on, off).

BTW, I've tested CSS ZRTP with OBi100 ATA and SIP client PhonerLite.

  2. When PhonerLite calls to CSS, ZRTP is on and working as expected. But when I call in opposite direction (from CSS to PhonerLite) - there is no ZRTP mode. I have no idea why. I don't know does it matter or no and I can't analyze RTP traffic though.

Comment #3

Posted on Dec 31, 2012 by Quick Hippo

Mmm interesting. Maybe an alternate way to send zrtp capability is expected by phonerLite. As per http://tools.ietf.org/html/rfc6189#section-8.1 the zrtp-hash implemented by Werner is the normalized solution.

Normally this sdp attr is optional and if not sent the remote should still try to have ZRTP. (that's probably why CSS starts zrtp when receive invite from PhonerLite even if doesn't send zrtp-hash). Maybe on the other way phonerlite requires the ssrc. But I would say it's a bug on their side. For the OBi100, I see two possible root cause for that. Either the SDP is not parsed correctly when the a=zrtp-hash is present. (I could try to add an option to not have zrtp filling sdp with this optional attr if you want). Or it somehow get problems with zrtp hello packets sent by CSS. Maybe some feature of the zrtp it doesn't support ?
