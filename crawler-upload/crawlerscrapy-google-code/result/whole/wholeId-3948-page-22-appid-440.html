Comment #1

Posted on Apr 2, 2011 by Quick Hippo

Your right on something, but finally wrong :)

If you have a look to the history of the code, you'll see that previously it was not threaded.

However, it lead to an Enormous problem... there where dead locks. Indeed pjsua does not allow to play too much with sip call state when you are in call_state. As consquence if another thread ask for call info at the same time there is an event (which could happen on slow devices).... then ... dead lock !!

So threading thing is usually the solution. The on_call_state and on_media_state should be treaten as events only. Then the rest of the app will try to get the call_info. That's absolutely not hurting to get a call info that does not match the call info of the update provided the fact this call info is more recent !

The rest of the app should be safe regarding getting a call info more recent. If call has been disconnected you'll see that get call info will just in pjsua and that csipsimple will consider call has been disconnected. So that's just fine.

If you find somewhere in the code where it's not safe regarding a deferred call info do not hesitate to tell us. But, actually there is a very good reason to do things threaded. You can try with non thread, you'll see that if you stress the app or use it on a low cpu device, you'll get dead locks.
