csipsimple - issue #1712

csipsimple reports missed call when answered elsewhere

Posted on Apr 30, 2012 by Swift Elephant

RFC3326 §1:

  1. Introduction

    The same SIP [1] request can be issued for a variety of reasons. For example, a SIP CANCEL request can be issued if the call has completed on another branch or was abandoned before answer. While the protocol and system behavior is the same in both cases, namely, alerting will cease, the user interface may well differ. In the second case, the call may be logged as a missed call, while this would not be appropriate if the call was picked up elsewhere.

CSipSimple does log a call as having been missed, which RFC3326 describes as inappropriate, when the call has been picked up elsewhere.

The CANCEL message will contain a Reason header such as: Reason: SIP ;cause=200 ;text="Call completed elsewhere"

We should act on it accordingly.

Comment #6

Posted on Jul 2, 2013 by Grumpy Giraffe

I've made a patch to address this issue, creating behaviour identical to what I'm used to from hardware (Snom, AVM Fritz, AFAICR Siemens) SIP phones.

I.e. when a call is completed elsewhere, it is neither logged as a missed call nor is a notification added for it.

The patch touches two areas of CSipSimple: [I'm posting pastebin URLs for the patch content, since I cannot seem to attach to this issue - "Issue attachment storage quota exceeded".]

A) jni/csipsimple-wrapper is extended to provide get_event_reason_code(pjsip_event*), which allows the java code to extract the SIP cause code from a 'Reason:' header.

http://pastebin.com/zyDazMpD

B) SipHome is extended as follows: B.1) SipCallSession / SipCallSessionImpl get a new property lastReasonCode tracking the cause code from the 'Reason:' header B.2) PjSipCalls maintains this property from its updateSessionFromPj() method B.3) UAStateReceiver omits all of the missed call logic when there is a 'Reason: ... cause=200 ...' in the CANCEL message.

http://pastebin.com/jmhzSGKz

Comment #10

Posted on Jul 23, 2013 by Grumpy Giraffe

Please reopen the issue, it is not fixed in current code.

I've tracked down the problems, there appear to be the following issues:

a) get_reason_code() is now run multiple times against the same buffer; this doesn't work since strtok() modifies the string it's supposed to tokenize. I.e. the second invokation of get_reason_code() for the SIP code I'm interested in cannot tokenize the reasen header and therefore does not find the reason code.

A fix is to use strdup()/free() for the header field in get_reason_code(): https://gist.github.com/jmbreuer/6061204#file-fix-get_reason_header-strtok-patch

b) Something about the ordering of messages relative to the UAStateReceiver.on_* callbacks appears to have changed (or I was just lucky the first time around); updateSessionFromPj() is now only called with an event that is a response to the CANCEL message; not the received CANCEL request itself. In the response, it is too late to extract the reason code from the request.

A fix is to also call updateSessionFromPj() from on_call_tsx_state(), and only update the reason code when there actually was one in the message: https://gist.github.com/jmbreuer/6061204#file-fix-update-rx-reason-patch

Ah, and with your slight change in UAStateReceiver.WorkerHandler.handleMessage() the "picked up elsewhere"-calls get logged as "received"; that's very nice.
