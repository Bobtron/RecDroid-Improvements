Comment #1

Posted on Mar 6, 2012 by Quick Hippo

It's probably a matter of available bandwidth on mobile data regarding codec in use.

The first thing to be aware of is that usually mobile data connection is not symmetric in terms of bandwidth. Download is usually privileged on upload. So incoming stream has more bandwidth than outgoing stream. This could explain delay on one way.

Second point, is that call quality and delay is really dependant on the codec you use. For example PCMU/PCMA (aka G711 or uLaw) is a non compressing codec. It uses 64kBit/s. For mobile data connection it's not suitable. Unfortunately that the only codec supported by some sip providers.

So if you want ot investigate, you have to find first what are the codecs supported by your sip provider. If only g711, you'll get hard time to get things working with them. And independantly of the sip client. There is no miracle, if sip provider does not support low bandwidth codecs, you'll not be able to use them on low bandwidth.

If they actually supports some better codec for low bandwidth, check that it's activated in csipsimple and with a high priority (under low bandwidth tab). If the codec is not supported by csipsimple report an enhancement request here, but normally csipsimple supports most of the codecs possible. To be sure that one codec is selected (or to test it's available on sip provider), you can try to disable all codecs but the selected one. If the call directly fails, it means that the sip providers doesn't support this codec.

Comment #4

Posted on Mar 7, 2012 by Quick Hippo

G711 is PCMU (it's the same, it's also sometimes known as uLaw/aLaw). It's probably what's used when you get bad quality. Since it's not compressing codec it use a very big bandwidth (64kbps).

g726 is available in nightly builds + the codec pack (http://nightlies.csipsimple.com/trunk/ + http://nightlies.csipsimple.com/plugins/)

g729 is supported but not activated by default. It also requires an user license. Some sip providers are buggy however with the ptime of this codec. I have to add a workaround option -- however I deeply advise to not use it since it's a proprietary codec.

g723 is indeed not supported but it's an obsolete codec that doesn't produce a good audio quality and has been completely replaced by G726. (see http://en.wikipedia.org/wiki/G.723 -- it tells that this codec is obsolete). However, I think that what's supported by easyvoip is G723.1 (it's different of G723). This one is also not supported for now, but it is subject to license so I can't implement that inside CSipSimple until the patent expires (in 2014).

GSMFR is also known as GSM. You should try this one (disable all codecs but gsm in the low bandwidh tab), it's a good compromise for mobile data communication. 13kbps is low enough to pass through mobile data connection.

One other point, in easyvoip wizard it automatically activate stun. It's not only useful if you get problem with G711, but for all codecs (it's about the ip/port announced by the device). So even changing codec, don't deactivate the STUN option (else you'll get no incoming sound).
