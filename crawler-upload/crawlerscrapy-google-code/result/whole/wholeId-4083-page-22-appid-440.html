Comment #6

Posted on Mar 5, 2011 by Quick Hippo

Could you test with latests dev version : http://nightlies.csipsimple.com/trunk/

I made a lot of changes regarding startup and network changes use cases. I guess it will help in this kind of case.

As for what you expect in the initial issue : well in fact csipsimple should detect it only when your device will detect wifi is down. Else obviously there is no ways for csipsimple to know about the fact wifi link is actually broken until next re-registration. In fact here it's matter of TCP/UDP protocol. TCP ensure packet delivery. So if you are using TCP to register broken link should be detected after TCP timeout. If you use UDP (as most sip provider do), the sip client is not sure about the fact link is still up. So you can observer strange things during some little time. Usually this time correspond to the time the network can recover from its failure.

Last point when you say : "but infact it is already disconnected in the server" > that's wrong : it's actually still registered. Registration is active until either client de-register, registration expires. Actually if server try to contact client it will fail cause client is not available from the server. But that's due to the fact the link between client and server is "temporarily" (or not) broken. There is retry mechanisms in SIP to try to get rid of this failure. But that's done at level not managed by csipsimple.

As consequence CSipSimple should try to reflect the information it actually has and say it is deregister only when it knowns the network state has changed on the device.

The latest version try to improve this kind of use cases (where start and stopping are done in the same time so that it may improve what you observe). Let me know how it goes. As there is a lot of refactoring, there is maybe also regression so I'm interested in feedback ;)
