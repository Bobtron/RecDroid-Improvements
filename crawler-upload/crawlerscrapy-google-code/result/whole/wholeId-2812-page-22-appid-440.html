About release commits it appears in the manifest of the stable versions. I guess there is some tools to extract it.

Posted on Oct 25, 2012 by Quick Hippo

Oh, about the version confusion, sorry I forgot to tell you about that :

As you know, android can be deployed on several CPU arch. For now officially supported are armeabi, armeabi-v7a, mips and x86. To publish on some market such as android playstore there is two choices. Release one universal app that is 4 times bigger than necessary and contains all arch. Or release versions per arch. During builds 0.04-xx release process I searched a little bit to support all arch on the play store. So, here's how it works now. The stable release process do the following

  • Step 1, build with all targets armeabi, armeabi-v7a, x86 and mips using make
  • Step 2 : run following shell : fake_version=getCsipsimpleRevision case $1 in x86) let fake_version=fake_version ;; mips) let fake_version=fake_version+1 ;; armeabi|trunk) let fake_version=fake_version+2 ;; armeabi-v7a) let fake_version=fake_version+3 ;; esac echo "Version to build $fake_version" cd $CSIPSIMPLE_DIR ant -f release.build.xml $1 && \ sed -i 's/android:versionCode="[^"]"/android:versionCode="'$fake_version'"/' AndroidManifest.xml && \ sed -i 's/android:versionName="[^"]"/android:versionName="'$STABLE_VERSION'"/' AndroidManifest.xml && \ ant release

It basically : * use the ant task to flag the manifest as a release one * replace versionCode with the revision adapted to cpu arch so that we can release 3 versions on play store (and other store that supports multi-arch -- I don't know any other for now however). * replace the versionName with the version being released.

The ant task will take in charge to flag the manifest as needed and the sed will replace things that needed to be changed for the release.

About your problem building the other version (0.04-03 and 0.04-05) it sounds to me you did a svn update ... while it doens't work as simply : To make easy to rely on third party project without having some crappy duplicate of their code, there is a quilt patches folder that applies in the make process. This implies that you must take care about that when upgrading and that running svn update without unpatching before will break things.

That's why, as explained on the HowToBuild wiki page, you should use "make update" instead of svn update. Alternatively if you understand how it works it's possible to use makefiles to unpatch. But will require to understand the build toolchain. So in your case the "<<" sounds to me a svn conflict that you missed and that is not resolved. I know that's not very clean. If you have suggestion it's of course welcome. I didn't spent a lot of time on that and prefer focus on features and having some env that I understand and can easily manage.

So, in short for you : you should restart from clean checkout (or do that manually if you understand what I'm talking about ;) ).

However, take care, version 0.04-00 to 0.04-03 are buggy versions (the -xx means that's a build of the same thing). So these version have known critical problems with some devices/sip providers and should not be distributed anymore. Next stable version with new features will be a 0.05-00. And 0.04-05 is the dev versionName, but for nightly builds I prefer using svn revisions instead. So 0.04-05 will probably never land as a stable versionName.
