Comment #1

Posted on Jun 25, 2012 by Quick Hippo

It's "fixed" in nightly builds.

Actually sipdroid doesn't work better. You think it does because it doesn't register in background, but it goes the same way. In fact the underlying problem is that android 4.0 doesn't broadcast anymore changes in vpn connection (anway was private api, so not android guys to blame for removing it). The only workaround possible, is to check VPN connection state frequently. The option is available as a "fix" in nightly build. This option is available in ExpertSettingMode, but normally thanks to the new warning views, you should have a pretty warning explaining the problem and giving a button to fix it in one click.

Download and install nightly from : http://nightlies.csipsimple.com/trunk/

Comment #3

Posted on Jun 26, 2012 by Quick Hippo

Can you try to collect logs. Your problem is maybe not the one I was thinking about. In fact I spent past two days to read android source code to find out how to workaround their last change.

But in your case sounds it's absolutely not linked to the fact it changes, but rather to the fact the route configuration make things going wrong.

So if you can collect log, maybe there is some interesting clue : http://code.google.com/p/csipsimple/wiki/HowToCollectLogs?wl=en

Also, if you can precise your network topology : where is the DNS, the route set by the VPN and where is the SIP server. sipdroid and csipsimple has different way to manage network interface. The one of pjsip (the sip stack behind csipsimple) is normally more logical -- actually that's the root reason why I did CSipSimple at the beggining, because I was unpleased with the way it worked on sipdroid and their dev team didn't replied to my patch ... so I switch to a reliable native sip stack that is used in many other platform and manage network interface a logical and clear way --. It makes that it doesn't magically work in some case when it shouldn't work and doesn't in other cases (without any possible workaround) There is also some known limitation of the sip stack with multiple account and public IP address too, logs will help me to determine in which case we are here and what are the possible way to connect :).
