Comment #15

Posted on Jan 29, 2011 by Happy Rhino

Thanks! The network problem is also interesting and I'll explain it here because it might be worth adding to the FAQ or something. It's something I fixed before but it reappeared because of an IP address change on the phone.

What happens is:

  1. Phone does DNS SRV lookup on SIP server.
  2. Phone registers with server.
  3. Later a re-registration starts and another DNS SRV lookup is done.
  4. This time the DNS SRV returns a different server address.
  5. Re-register is done to the new address, but because of this, the original NAT state on the router for the session does not apply, and a new NAT state is created with a different outside port number.
  6. This confuses the registration process for a little while until it sorts itself out.

This happens if your SIP provider returns different addresses in response to DNS SRV queries, which it may do for load-balancing.

My fix was to add a static NAT mapping on the router so that it always uses the same outside port number.

I don't know if the instability caused by this problem is related to this issue - it may have triggered it because of the extra messages. I'll wait to see if there's another crash now that this is fixed to confirm, and if so will try r602.

I think that your problem is in your JNI wrapper. Usually this kind of problem :

""" W/dalvikvm( 398): JNI WARNING: JNI method called with exception raised W/dalvikvm( 398): in Ldalvik/system/NativeStart;.run ()V (CallStaticVoidMethod) W/dalvikvm( 398): Pending exception is: I/dalvikvm( 398): Ljava/lang/NoClassDefFoundError;: [generic] """

Occurs cause your native code tries to callback java using JNI.

This part of the jni glue is usually really complicated to do. To make things much more easier in csipsimple I did use SWIG (which automatically generate things and has a director approach). However I did some patch cause of the specificity of the Dalvik JVM : there is weird things in this JVM and we have to workaround it if we want to have things going well on all threads.

As logs your get and your problem is really really different from the one of this thread - the only common point is that there is ~a~ crash in the native lib ;) -, we I could continue to help you if you send me a email on my private email (r3gis.3r at gmail_dot_com).
