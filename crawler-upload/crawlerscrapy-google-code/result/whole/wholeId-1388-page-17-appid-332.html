Comment #2

Posted on Sep 2, 2010 by Grumpy Giraffe

I just tried a vanilla qi with the 2010-08-26 cupcake daily which hung during the first boot logo. The only qi which managed to boot into a working system was the qi.img delivered with the installer-archive (Qi Bootloader s3c2442 Beastor no_branch_4e6b3d5b84eca172 Tue Apr 14 17:54:45 CST 2009 Copyright (C) 2008 Openmoko, Inc.)

I logged the boot process of a vanilla and that koolu? qi to find the differences. Here is what I found and how I fixed the boot:

Cmdline(android booting): loglevel=4 console=tty0 console=ttySAC2,115200 init=/sbin/init ro mtdparts=physmap-flash:-(nor);neo1973-nand:0x00040000(qi),0x00040000(depr-ub-env),0x00800000(kernel),0x000a0000(depr),0x00040000(identity-ext2),0x04000000(system),0x0b6a0000(userdata) rootfstype=jffs2 root=/dev/mtdblock6

Cmdline(android not booting): loglevel=8 console=tty0 console=ttySAC2,115200 init=/sbin/init ro mtdparts=physmap-flash:-(nor);neo1973-nand:0x00040000(qi),0x00040000(depr-ub-env),0x00800000(kernel),0x000a0000(depr),0x00040000(identity-ext2),0x0f6a0000(rootfs) rootfstype=jffs2 root=/dev/mtdblock6

It seems the partition layout of the android filesystem requires the "system" and "userdata" partitions while other distributions just use the "rootfs"-partition.

One should also note, that the aof-repository has a default partition layout (only rootfs) in the qi sourcecode, so simply compiling it will not give you a working android bootloader (at least for cupcake?).
