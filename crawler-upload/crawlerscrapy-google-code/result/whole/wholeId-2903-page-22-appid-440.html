csipsimple - issue #1927

EVO 4G Voice Very Choppy on WiFi with Screen On

Posted on Aug 30, 2012 by Swift Monkey

When on a WiFi call (PCMU codec), the incoming voice is good for several seconds, and then becomes very choppy and lagging for several seconds, and so on. The phone becomes not very responsive during the call. But System Panel shows the CPU clock mostly swings between 300~400MHz, CPU usage mostly swings between 80~90%, but the CPU clock won't go higher. Information page shows no packet loss. Screen is on. "Keep awake on a call" checked. "Lock Wifi", "partial CPU wake" checked or not, the same. No "high perf wifi" option.

When on 3G, sound is good. A few static or distortions maybe like Issue 301 but much better. Information page shows no packet loss.

It is an EVO 4G on Android 2.2.

Comment #1

Posted on Aug 31, 2012 by Swift Monkey

I noticed voice was fine if USB is connected. Looks like Csipsimple puts the CPU in some restricted mode incorrectly if the phone is not charging. Other apps and VoIP programs work fine and CPU clock changes on demand and can reach top speed 1GHz freely when the phone is not charging.

Comment #2

Posted on Aug 31, 2012 by Quick Hippo

Thanks for the very interesting point.

Strange that it changes when charging or not and that other VoIP apps works fine. There is not so many things in android API to change lock the CPU and csipsimple use http://developer.android.com/reference/android/os/PowerManager.html#PARTIAL_WAKE_LOCK that is normally the one to lock CPU.

However in your case seems more linked to wifi no? If works (almost) correctly on 3G (btw, in 3G it's not recommanded to use pcmu that is 64kBit/s codec which is very high).

So maybe something with CPU/wifi topic. On that there is a little bit more options available to developers. The weird thing is that it comes more complex from android 2.3 and you are using 2.2 from your description so should not be impacted.

If you have time to run two test I would be very interested in the results you get :

#1 : go in android settings, in wifi section, in advanced parameters menu. There is an option to "Keep wifi active in sleep mode when"... here normally 3 choices : never, always or when charging. I suspect you are in the charging mode. Can you confirm? Also, can you change that temporarily to "Always" just for the test. Then test a call and see if better.

#2 : while in call, try to switch from speaker on/off to see if it changes something.

I would also be interested in packet loss you get in RX stream while in communication to determine if something with network or with media.

Thanks in advance

Comment #7

Posted on Sep 1, 2012 by Quick Hippo

Ok, so probably linked to issue 601. I merge with it.

I guess there is one profile in setCPU to change policy when phone is charging (there's one by default). What's still strange is that other apps can get rid of that.

In fact SetCPU uses root priviledges to be able to change CPU policy. So it would be a security problem if another third party app such as csipsimple without root priviledges would be able to change cpu governor policy.

As you seems skilled in tech, maybe would be interesting to run in adb when in call with csipsimple and when in call with other app that works : cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor To see if there is difference.

I don't think other apps will change the cpu governor mode directly but maybe they match a rule in setCPU that changes the governor... the question is which one ;). (Or maybe it's csipsimple that make the state matching another rule that change to worse cpu governor mode of setCPU).
