rogue-op - issue #2

Need a more platform-neutral version of framework classes

Posted on Apr 23, 2011 by Massive Dog

Many framework classes are inevitably bound to the target platform. This is expected, but this dependency propagates throughout the framework class hierarchy.

For example, Soundy synthesis itself is platform-independent, pure Java code. But the package depends on the Android audio classes in order to play its output. This issue stalled out work on the SWT SFXR tool, which fails to compile as soon as I started adding the synthesis code. There is a certain level of coupling which is necessary to actually get anything done, but a more careful design should allow us to manage this with minimal overhead.

Since Java doesn't allow for conditional compilation, we are left with a few possible courses of action: 1. Use the standalone C preprocessor to generate platform-ready java files. 2. Maintain separate platform-specific branches of the framework 3. Make platform-specific versions of packages, and import the one we need for a particular build. 4. Make all such classes into interfaces or abstract classes, and have separate platform-specific implementations.

(1) has the obvious drawback that anyone not building with our toolchain would not be able to successfully compile the framework; additionally, this technique won't play nice with Eclipse without writing a plugin.

(2) just sounds like a nightmare, to keep two or more versions of everything in sync.

(3) compromises here by allowing most code to remain integrated, with only those classes that are platform-specific showing up in more than one place. But it still necessitates duplication and complicates the package structure. On the other hand, a major pro for this option is that all platform knowledge in the client classes (that only use the platform-specific ones) is sequestered in the import block; none of the actual Java code has to change. This could also be implemented by having separate folders to hold the sub-packages in question, and use the project settings to choose the search path for the classes we want; this would mean none of the code has to change.

4) Is probably cleanest, with only the parts of those classes that are platform-specific being reimplemented, but takes more effort up front than option (3) to rewrite the framework, and also requires rewriting the code where these classes are allocated.

There may be other alternatives I have not thought of. Please leave suggestions as comments below.
