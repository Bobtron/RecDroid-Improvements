Comment #3

Posted on Nov 20, 2011 by Quick Hippo

The root of the problem was probably due to zrtp builds. That's now integrated to trunk and should be better.

As for second issue, that's a little bit different problem. I thought about this possibility to ignore the sip stack and just manage things in the UI. However, it's not very robust. In fact when you says "interact with SIP server" it's not actually what csipsimple does... it interacts with the sip stack which can (will) interact with sip server. The sip stack also manage timeout if the server does not reply and will inform anyway the UI about the call is terminated. And by terminated it means terminated as a SIP dialog defined in SIP RFC. If something were done on UI side ignoring SIP protocol, it would mean that there is an inconsistent state between what is the actual state of the communication in the sip stack layer and the UI. That's why I said this would be not very robust. I envisage other solution to have something on UI side that look like faster for end call action but that would keep the view somewhere so that the user is still informed about the fact the transaction is not yet totally finished. But that's another issue :).
