csipsimple - issue #2301

Feature request: Smart local IP detection when SIP server is on LAN and does not support ICE

Posted on Mar 29, 2013 by Swift Bear

Problem

I am using CSipSimple to register to my FritzBox on the same LAN (192.168.6.0/24) which works fine. When I am in a different LAN (say, 192.168.2.0/24) and I am connected to my home LAN via VPN, CSipSimple announces my 192.168.2.x address which causes RTP to be sent into nirvana, i.e. audio won't work.

I can temporarily work around the problem by configuring the bound RTP address in the settings, however when I'm back in my home LAN I receive a different IP than via VPN, causing the workaround to break everything again.

Feature Request

When connecting to a SIP server without STUN, advertise the local IP which is actually used for the "connection" (=source IP of UDP datagram) instead of the IP with highest priority.

Additional info

  • The VPN is only used for local connections. The default route (i.e. to the internet) is not via VPN
  • Installing a STUN server on the LAN is not an option as it would require another computer to be constantly running.

Comment #2

Posted on Mar 30, 2013 by Quick Hippo

Well why not adding it in csipsimple, but the spirit would be more to have it done by a third party application that would detect that you are connected using VPN (there is no API in android to know the device is connected through vpn).

As temporary workaround you can : - configure two accounts (one for lan use, one for wan+vpn use) - use locale/tasker plugin for csipsimple to switch from one account to the other /// or use the csipsimple widget on home screen to quickly enable/disable the account you wish.

For the record for other reading it : the approach described here are only if no STUN server can be put on the sip server and that the sip server doesn't support ICE. (these case should be marginal and considered as not real sip providers/or sip providers that will by nature not play well with nat traversal and multi-homed situations).

But, anyway, got your point, and indeed, why not having this as a tricky option hidden in csipsimple dedicated for FritzBox (or other enterprise sip service that is not properly configured and with an IT that doesn't want to support things correctly).

Posted on Mar 30, 2013 by Swift Bear

Hi, Thanks already for at least considering it!

A few more thoughts about this: I just find it somewhat strange, that a UDP packet from 192.168.6.211 to 192.168.6.254 would claim to originate from 192.168.2.123 in its payload. Also, as Android devices are becoming more complex, it might become likely that they also will use more than one active IP uplink. Things that come to mind are dual-SIM handsets with load balancing (or other routing decisions) or similar on both SIMs. Or LTE devices with a dedicated PDP context for VoLTE in addition to a "regular" IP uplink. Both are cases where the routing decision is made based on more than just priority. On either case it would just make sense to me if the UE would advertise its correct IP address in the SDP.
