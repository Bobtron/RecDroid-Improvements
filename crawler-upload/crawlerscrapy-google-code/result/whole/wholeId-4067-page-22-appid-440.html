csipsimple - issue #756

support outbound calls w/o registration

Posted on Feb 28, 2011 by Helpful Kangaroo

> What steps will reproduce the problem? 1. add account for Euroset provider which doesn't permit REGISTER (details below)

> What is the expected output?

Account is only used for outbound calls (INVITE) via provider's SIP proxy; no attempt to register with provider's SIP registrar is made

> What do you see instead?

Account is disabled due to failed registration (the server responds with "SIP/2.0 401 incorrect password or account name")

> What version of the product are you using? On what operating system? 0.01-01 r615 LG P500 Optimus One, Android 2.2.1

> Please provide any additional information below.

Here are provider's instructions for use with one of the SIP UAs on Mac OS X: http://card.euroset.ru/index.php?act=article&id=91

(google did a decent job translating it: http://translate.google.com/translate?js=n&prev=_t&ie=UTF-8&layout=2&eotf=1&sl=ru&tl=en&u=http%3A%2F%2Fcard.euroset.ru%2Findex.php%3Fact%3Darticle%26id%3D91)

Note the last bullet in item 5. says to uncheck 'Register with domain and receive incoming calls'. Users in various forums also claim that REGISTER is rejected by this provider, however placing calls works fine (e.g. people managed to forward calls from Asterisk PBX).

Quick skimming through specs suggests that this is perfectly allowed in SIP. Dunno if pjsip supports this; if it does, it would be nice to have this kind of accounts supported in CSipSimple

Comment #3

Posted on Mar 1, 2011 by Quick Hippo

That's not exactly the same. What you ask for is no registrar at all. Not registration on demand. In your case as far as I understood registration does not work.

On demand registration of issue 329 is already possible using either the existing widgets or the "only for outgoing" mode. Besides latest dev versions now have full "outgoing" only mode support (after call the application deregister to not drain battery).

It's a best idea to ensure we can register before making a call for several reason (for example it helps to find a correct public IP). So that the approach to register each time we need to make a call and de-register after.

However your use case is interesting. And it make sense to be able to configure an account without registrar at all. But it mean modifying some behaviors (which has already be done for "local" account but not cleanly enough ;) ).

I guess I've found the problem:

com.csipsimple.api.SipProfile.getDefaultDomain() returns null early if reg_uri is null, which is exactly the case for registrar-less accounts.

The following patch should fix that (sorry I don't have the toolchain handy to even compile-test it).
