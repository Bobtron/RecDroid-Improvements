Comment #2

Posted on Jan 19, 2012 by Quick Hippo

What codec is in use for incoming calls? (press the little arrow under the call card and select the info button). And what codec is in use for outgoing calls?

In fact since for outgoing the choose is made by remote side it could result on a different codec in use. And some codecs (such as PCMU/A aka G711) use a lot of bandwidth. In this case, it's a good idea to see what codec is used for incoming calls and to disable all codec but this one for narrowband codecs list (settings > media > codec, long press on a row to disable).

If incoming call are fine it's probably not linked to NAT (but I still recommand to enable STUN at least, ICE is not necessarily needed).

If the codec changes does not help, could you collect log (see HowToCollectLogs wiki page).

Thanks

Comment #4

Posted on Jan 19, 2012 by Quick Hippo

The one way audio when the second part can hear you and you can't hear them is usually the consequence of a NAT issue. So if you get that for both incoming and outgoing, the diagnostic is more about NAT stuff :)

So in a first time, try to enable just STUN (not ICE). In the STUN server you have probably counterpath as the default one. Sometimes this one is not responsive (many sip clients use it ;) ). Also depending of the country you live in it may be a wrong stun server : some countries are isolated from the rest of the world and public stun server of counterpath or others, does not resolve the public address of the phone but the one of the firewall of the country... and it's annoying if the sip server is in the same country than you ;) (because the "public" address given by the stun server is not relevant in this case).

Logs may help me to see what actually happens and how stun server resolves your phone.
