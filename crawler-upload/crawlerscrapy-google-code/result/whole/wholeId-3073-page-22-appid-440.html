Comment #7

Posted on May 19, 2012 by Quick Hippo

@amessina => in your case you should create a rewriting rule for your provider.

Go in settings > filters > Select provider > add filtering/rewriting rule. Here create a rule "starts with" "+" > "replace with" "00" (for example if that's what is expected by your provider). This rewriting rule will apply on all phone numbers considered as coming from outside of csipsimple : * integration with stock dialer, * list of contacts autocompleted when you click a number

it doesn't apply if you write things by hand inside csipsimple. If you explicitly type +123456789 in csipsimple, I consider that you know what you are trying to do since you have a way to control which account will be used. And so no rewriting rules applies. (It also allows to bypass rewriting rules).

But normally that's the feature you are looking for ;).

Comment #9

Posted on May 19, 2012 by Quick Hippo

Change is not supposed to remove the leading +. Change is supposed to not encode leading + into uri encoded equivalent which was leading to report bad uri from csipsimple (not from sip provider).

So this issue is actually fixed.

Now, maybe there's something with the filtering rule that doesn't apply when you click an auto-completed result, but that's another thing. If you can confirm that : 1) you have such a rule for the account currently selected in chooser button. 2) when you click on a contacts with a leading + in their phone number row, it doesn't rewrite the phone number when adding it to the edit text box. 3) your scenario is not adding with keyboard the leading "+" in the text box (because in this case no rewriting rules applies).

=> that's another bug, and you could open a new issue about that. Also if you can collect logs (see HowToCollectLogs), it give me some clue :) Thanks.
