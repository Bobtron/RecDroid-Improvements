one way voice with cisco anyconnect(and probably all VPN clients)

Posted on Aug 21, 2014 by Grumpy Lion

What steps will reproduce the problem? 1. SIP server in internal network 2. Set up VPN(I use cisco anyconnect) via 3G network 3. Make a call 4. Check voice path

Comment #1

Posted on Aug 23, 2014 by Quick Hippo

It's a common problem using VPN. It's a topology problem and it has been debated several times in this issue list, but the way pjsip (the sip stack behind csipsimple) works is to select the main interface as the one for RTP announcement by default. It's not a bug as it's a very valid way SIP topology (signaling can go a different path than media which for example might allow direct client to client media while negociation is made on the web).

Anyway, it's something that can be very annoying in your situation.

But, several solutions are possible. The usually good solution in case of multi-homing is to enable ICE and have ICE supported on both sides. This solution (ICE) will simply solve most topologies. The clients will announce all possible interfaces it can use and the other side will try all possible interfaces. This way, the faster and valid path will be used for the call.

If your topology/server does not support ICE, there is an option in recent csipsimple (http://nightlies.csipsimple.com/trunk/) that request the app to use the interface found during registration. To enable this option, transform your account into an expert one (see : https://code.google.com/p/csipsimple/wiki/ExpertSettingMode?wl=en#Account_settings << the Account settings section). And scroll down to Allow SDP NAT rewrite option.

A third solution is to use the "announced rtp address" to fix one ip address to announce in SDP. (you could also use the binding rtp address if you don't want the phone to listen on public interface). This solution unfortunately requires to have a static (or dhcp fixed assigned) ip address on your vpn for your device.

Let me know how it goes and which solutions was the best for your case.
