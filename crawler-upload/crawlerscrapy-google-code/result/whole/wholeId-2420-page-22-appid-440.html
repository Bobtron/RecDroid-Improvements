Posted on May 14, 2013 by Quick Hippo

If I do not mistake, from logs 1, what sends the low AS value is the xlite side. If you want to announce bandwidth info from csipsimple you have to enable that in settings (it's by default disabled because some sip providers doesn't parse SDP correctly and gets broken when sent). To enable it : go in menu > settings > menu > 'Expert mode' once enabled, go in network section > sip protocol > Enable the add tias fields ... (rfc 3890).

However, I'm not sure it's the problem. Apparently you are not using same network conditions in both tests. To be able to help I would also be interested in logs from the app (see HowToCollectLogs) it comes with what the app see, the rtcp stats, the model of the device etc. I would also be interested how you mesure jitter? Comment #4

Posted on May 16, 2013 by Quick Hippo

Before further things, from your logs you are not using the up to date csipsimple version. Please upgrade it to latest stable : http://nightlies.csipsimple.com/stable/1.00.00/. If you cn produce same logs but with the stable version it will be interesting.

Then about rtp timestamps. They are absolutely right for the Opus codec implementation. This implementation is currently under dev and I don't advise people to use it yet unless they are under some constrained conditions. There is maybe another codec that has been recently modified to use webRTC codec implementation and might be buggy, it's iLBC. So if you are using one of these codecs it could be the case and could be the root cause.

However, it's not the case for all other codecs and it should behaves exactly the same way other pjsip based sip client does (microsip is a pjsip based sip client). An AFAIK it's not buggy with freeswitch (I use to do my test with this server on the other side which is the easiest to configure and the most stable ;) ).
