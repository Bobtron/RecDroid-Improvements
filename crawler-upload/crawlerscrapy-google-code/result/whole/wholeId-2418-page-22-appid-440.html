csipsimple - issue #2360

ZRTP does not work between Android 2.2 devices (Galaxy S) and devices above Android 2.3

Posted on May 14, 2013 by Happy Kangaroo

ZTRP works correctly on calls between Android > 2.3 devices, and between two Android 2.2 devices, but NOT between Android > 2.3 (inclusive) and a 2.2 device.

This has been tested with: - Galaxy S, CyanogenMod 10.1 M1, M2, RC1 (Android 4.2.2) - Galaxy SII, Official ROM (Android 4.0) and later with ParanoidAndroid ROM (Android 4.1) - Nexus S (Android 4.1) - Chinese device, Android 2.3 - 2x Galaxy S, original ROM (Android 2.2)

I know that 2.2 is residual now, but it does not makes sense to me that ZTRP doesn't work.

I've tested over several csipcsimple nightlies. Attached logs are with the Google Play just released version 1.0, using the Galaxy S with CyanogenMod 10.1 RC1 and Galaxy S with Android 2.2

Note I have no other 2.2 devices to test than the Galaxy S.

PS: I can't attach the logs, it reads "Issue attachment storage quota exceeded." so I'm sending it to developers@csipsimple.com and referencing this issue.

Comment #1

Posted on May 14, 2013 by Quick Hippo

HI

Thanks for the logs. It appears there is a freeswitch in the middle that probably handle the ZRTP in the middle. So it's not really direct sip client to sip client call and there were freeswitch version that didn't handled properly the zrtp when media proxy.

What you can try to be sure it's a problem with the freeswitch version is to place calls that completely bypass the server and are direct calls.

To do so : Create an account on each sip client with the wizard 'Local'. Write down the ip address of client (let call this one number 2). Go in settings and make sure the STUN option is disabled on both clients.

Return in dialer of and press the 'txt' option in bottom menu bar of client number 1. Dial test@IP_ADDRESS_OF_NUMBER_2

In this case the sip call is a peer to peer call and will not introduce extra complexity. Also, as far as I understood from author of libzrtpcpp (included in pjsip) there where a major change in zrtp specifications recently to introduce a new version. So the negotiation of version number was added (RFC6189) and if not managed properly by freeswitch (as was the case previously for key algo negotiation it might introduce weird things).
