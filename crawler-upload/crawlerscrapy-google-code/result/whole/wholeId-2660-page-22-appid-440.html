csipsimple - issue #2179

Incoming call from PBXes with gtalk trunk using iLBC results in one-way audio

Posted on Jan 17, 2013 by Happy Cat

What steps will reproduce the problem? 1. Register an account with PBXes (and possibly other SIP registrars) and create a gtalk trunk 2. Select iLBC as the preferred codec in CSipSimple 3. Accept an incoming call via the account created in 1.

What is the expected output? What do you see instead? Expected: two-way audio; observed: Rx audio is fine, but no (well, garbage) Tx audio

What version of the product are you using? On what operating system? Any (market or nightly)

Please provide any additional information below. Calls initiated with CSipSimple using iLBC result in two-way audio, so only incoming calls are a problem. Interestingly, when a call comes in with one-way audio using iLBC, things can be fixed by placing the call on hold followed by "resume'. I believe that codec negotiation is to blame here. So, I ran a few wireshark traces:

  1. Inbound call using iLBC as the preferred codec: pbxes offers payload type pt=97 and MIME type "iLBC" (note the lower-case "i"). CSipSimple accepts the negotiation with pt=97 and MIME type "ILBC" (note the capital "I"), resulting in no (bad) Tx audio. When I put the call on hold and resume, the call gets re-negotiated (initiated by CSipSimple) using pt=102 and MIME type "ILBC", resulting in fine two-way audio

  2. Outbound call using iLBC: CSipSimple offers pt=102 and MIME type "ILBC", pbxes accepts with pt=102 and "iLBC". Audio is fine in both directions

I believe CSipSimple does not actually register the correct Tx codec when offered a payload type that differs from pt=102. I know that there is an issue with the webRTC implementation where iLBC is defined as a codec with a fixed payload type pt=102 (which is non-standard). However, pt=102 is in the dynamic pt range and should not be treated as a fixed pt.

If webRTC is being used then I think the problem is in (at least) two files:

  • function find_codec(...) http://code.google.com/p/csipsimple/source/browse/trunk/CSipSimple/jni/webrtc/pj_sources/webrtc_codec.cpp

and, more importantly, the iLBC codec definition in webRTC itself, i.e. - [...]/modules/audio_coding/main/source/acm_codec_database.cc iLBC is (wrongly) defined as if it were a fixed payload-type codec .

My first stab would be to have pjmedia ignore the payload type and try to match the codec via MIME type only (or in addition), i.e. either "ILBC" or "iLBC".
