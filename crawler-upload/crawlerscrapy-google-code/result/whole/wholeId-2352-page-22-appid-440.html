Comment #1

Posted on Sep 25, 2013 by Quick Hippo

"The Settings > Media > Clock rate is set to 44.1kHz. Should that be set to 16kHz?" Yes you should use 16kHz instead of 44.1kHz (which introduce extra resampling which is CPU consuming)

"If Opus is fixed at 16kHz, can you make a range of fixed ones like SILK that are selectable (until auto changing is implemented)" Actually the clock rate in media > Clock rate is not directly linked to the clock rate used by the codec to transmit over the network. The clock rate value here should be the one at which the audio layer of the device is opened. In short : it's already dynamic as opus specifies on what is transmitted over network; the missing improvement is to fix it to value setup for android audio layer (in media > clock rate) (which should be by default 16kHz). That's because of this default, that it's hardcoded to 16kHz currently -- but this 'internal' clock rate will always be something fixed by audio layer settings and not something variable. What is variable is what opus codec decide to use as clock rate to send over the network (and the opus codec does the resampling internally).

About the problem of metallic sound I'll try to investigate. Just more background info about the topic : Actually the change to 2 channel is mainly "cosmetic" (it changes what the app announces). So, it should not change how the app handle packets but just what is announced. Announcing 2 channel is mandatory in RFC spec of opus. It does not mean the audio layer on the device is stereo. BTW, it's currently still mono on csipsimple side. The previous implem were not finalized, and actually it was something that another sip client should not have been able to handle at all. So I'm surprised that jitsi was able to work with the previous implem.

Thanks a lot for the tests and the feedback. It's very valuable to get results of this new feature !
