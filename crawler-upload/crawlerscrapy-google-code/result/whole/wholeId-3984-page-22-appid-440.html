Comment #11

Posted on Mar 24, 2011 by Quick Hippo

@Luke : I'm now analyzing your logs. (Sorry for the delay I was busy on the video task ;) ).

It sounds the choosen codec is G722 for your calls. Problem is that what is sent by the server seems not to be G722. In fact, the server you use reply with multiple codecs. Unfortunately, pjsip does not like that. It only allow to choose one codec and then talk with this codec. As consequence pjsip resend an invite by choosing only g722 but your server on the other side ignore that and talk using another codec. As consequence pjsip is lost. (You'll probably experiment ANR and no sound at all). Usually server that send replies with multiple codecs do that cause your are now in call with an auto attendee. (automatic machine such as voice mail and so on, support multiple codecs in reception but only send using one codec). So it could be interesting to disable g722 and only test with for example pcmu.

I'll have more closer look cause there is also interesting point about how I manage network changes.

I'll keep you in touch

Posted on Mar 24, 2011 by Quick Hippo

As for the nightly build website, that's fixed now. I was trying to fix www.csipsimple.com and csipsimple.com but I broke nightlies.csipsimple.com while doing that lol. You'll probably have to clear your browsing cache. Else it will probably try to redirect again and again to code.google.com... or if you don't want to remove your browsing history just go to http://nightlies.csipsimple.com/trunk/?a=o (a=o param will fake the browser so that it will try to reload without using its cache ;) ).

For the SIP process the standard echange is should be like that :

  • CSipSimple INVITE sending SDP (SDP contains supported codecs)
  • The other side after optional 100 trying and 180 ringing send a 200 OK with a SDP containing chosen codec.
  • Communication is established based on the chosen codec by the other side.

The problem here is that in the 200 OK from the asterisk contains the list of supported codec. So after the 200 OK, while call is 'established' <> codec is still not choosen. As far as I know this is not forbidden by SIP RFCs. However it annoy pjsip cause they do not support multiple codecs at the same time. As consequence, pjsip re-send invite with only one codec. As the other side announce it support all this codecs, pjsip decide to choose only g722 in our case. However your server seems to ignore this reinvite with only one codec and send to pjsip RTP packets encoded with one of the other codecs of the first invite (I guess that's using pcmu). Problem is that at this point, pjsip expect only g722 RTP packets (or telephony events RTP packets). So then you asterisk thinks it can talk using "any of it" the announced codec in first INVITE while pjsip expect only g722 (since it has explicitly asked for it in re-invite).

I'm not an expert at asterisk configuration but hopefully there is a way to workaround that on the server side to either allow re-invite or to choose definitely a codec when replying invite and then transcode codec if needed.

So in your scenario there is two problems, one on pjsip and one on the server : On pjsip side it could be cool to support multiple codecs. For reference the issue in pjsip : http://trac.pjsip.org/repos/ticket/476 -- it was marked as fixed time ago cause at pjsip side they consider sending re-INVITE from pjsip to force only one codec should fix it. On asterisk : it probably does not actually take into account the re-INVITE codec change. However what's strange is that asterisk seems to acknowledge the re-INVITE and to correctly consider the G722 as only codec. But regarding logs about bad RTP format it seems to actually continue to send using another codec.
