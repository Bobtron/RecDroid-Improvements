Comment #5

Posted on Nov 17, 2011 by Quick Hippo

Just last point about that, the setting of TCP keep alive is actually not something available in settings (the one in settings is only about UDP keep alive).

In previous version, since keep alive was managed by the application (csipsimple) and not the sip stack (pjsip), it was also impacting TCP transport. Now that it's back to pjsip control (most reliable in background cases and disconnect/reconnect cases), this value (for TCP) is defined at build time. So for TCP, independantly of what you'll set in csipsimple settings it will always be 90. Maybe pjsip guys will give an option in future to have this value modifying all transport types. But not really a problem relevant of csipsimple project, but more something on pjsip project (the sip stack is widely used by many other sip client, opensource or not, and not only on android).

Comment #6

Posted on Nov 17, 2011 by Massive Monkey

Not sure I understand. Are you saying that TCP keep-alives are hard-coded to 90 seconds and that value is unchangeable from the UI?

If so, that unfortunately makes the feature almost useless. The point of using TCP instead of UDP is that in networks, TCP NAT entries are typically timed out much, much slower than UDP entries. This allows for TCP keep-alives with much longer intervals (e.g. 10-15 minutes) than could be used with UDP keep-alives. And the end-result is thus much better device battery life. A TCP keep-alive value of 90 seconds thus defeats the whole purpose of using TCP.

Or am I misunderstanding what's going on?
