csipsimple - issue #496

csipsimple causing occasional Android OS crashes

Posted on Dec 14, 2010 by Happy Rhino

Hello! Trying to find the root cause of intermittent Android OS crashes which happen only during an active call with csipsimple. The platform is HTC Wildfire running Android 2.2.1 (CyanogenMod 6.1).

Problem: During some SIP calls, the call will freeze for several seconds, audio dies, and a moment later the Android boot screen appears and the system restarts. The logcat shows that at a certain point the Android system_server starts sending SIGQUITs to a bunch of processes and this eventually leads to a crash/restart. This happens on about 1 out of every 3-4 calls, longer calls making it more likely.

This did not seem to hapen in the csipsimple 00.13 market version. But since moving to 00.15-18, 00.16-00, and now revision 427, it has happened consistently.

I'm not familiar with Android internals but the SIGQUIT-sending suggests some kind of purposeful action, like trying to recover from critical resource exhaustion. Several log messages appear just before this happens:

My current theory is that csipsimple is using up a large amount of resources, which at times gets bad enough to cause the OS to kill itself. I have also noticed that in-call UI things are much slower in the recent csipsimple versions than they were in 00.13. For example there is often a pause of several seconds before the screen changes state when answering a call, and after ending a call, the screen goes black for as long as 10-15 seconds before returning to the dialer/homescreen.

The Wildfire has a relatively low-powered CPU (528MHz) so high resource usage would be more likely to hit it than faster devices.

Any idea what could have changed after 00.13 that would cause significantly higher system load?

Comment #1

Posted on Dec 14, 2010 by Quick Hippo

Yes CSipSimple "UI" use resources. However, the native SIP stack if correctly configured use really few CPU (if audio 'driver' behaves correctly).

Actually there is two method to read audio. One which was used in prior versions to 0.00-13 were based on a straight implementation, very basic and hopping that the audio driver is correctly coded. It worked very well... until Samsung (and their awful audio drivers) comes ....

Unfortunately, with these drivers the simple implementation was buggy. And I had to change my implementation to get it working on the Galaxy S (and some other devices based on Alsa driver). Unfortunately that's not optimized for HTC devices (that prefers the straight implementation and overrun a little bit).

So what I would advise first of all : * Change the audio frequency : from 16kHz (which is the default on 2.2.x devices to 8kHz which is the default for old generation 1.5/6 devices... but in your case as you are autodetected as a new device... but with a low CPU.... you have automatically 16kHz choosen). 8kHz will help a lot : * No more need of a resampler (cause all codecs are 8kHz so better to have 8kHz as output if you prefer performances to call quality) * The audio driver will less overload it's buffers * Ensure that codecs activated are only low CPU consuption codecs (for example no iLBC, no g722 etc) * Disable Echo cancellation * Disable Voice Auto Detection

I could also do a custom build with the Alsa compatibility turned off, but I'd be interested first by results of theses tests, cause I think it's more likely to be the real cause of the problem (special audio frequency, cause in 0.00-13 was by default to 8kHz for all devices).

Comment #2

Posted on Dec 19, 2010 by Happy Rhino

Thanks for the suggestions, I've reduced the frequency to 8kHz and am still testing.

In the meantime, I've been looking more closely at CPU usage during "idle" periods, while no call is active and csipsimple is in the background. (I suppose any such background usage would be due to pjsip.) I was prompted to do this because at one point the phone had slowed to a crawl. When I opened a shell and ran a "top", I saw that system_server would often rise to the top of the list and consume more than 90% CPU for 10-15 seconds. Then it would go back to normal for a few seconds, then shoot up again.

I couldn't figure this out until I opened csipsimple and hit Menu->Disconnect. After this the idle CPU usage immediately dropped back to 1-2% and stayed there. After I told csipsimple to reconnect, CPU usage went up a little but was quite reasonable: periodically (once every 30-60 seconds) I would see csipsimple and system_server simultaneously rise up to about 7-10% each, then drop back after a couple of seconds. I assume this is due to processing keepalives/registrations (I have keepalive set to the default of 40s and re-registration happens with callcentric every 60s). There is no STUN etc. enabled.

The strange thing is that the csipsimple/system_server CPU usage seems to creep back up over time. I got in the habit of running a "top -m 3" in the shell now and then, and eventually I saw that the periodic "peaks" were getting both taller and wider: csipsimple hitting 15-20% and system_server hitting 90+%, and staying that way for 5-10 seconds before dropping. Once again, telling csipsimple to disconnect/reconnect knocked the CPU usage back down.

So there is something odd happening with how csipsimple/pjsip uses background CPU over time. Any idea why the background usage would increase this way?
