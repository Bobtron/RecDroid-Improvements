Comment #3

Posted on Jan 24, 2011 by Quick Hippo

Does the other side accept speex @8kHz. And does it use speex 8kHz when it works on 585?

Cause in this version due to the new feature of codecs per network I reorder codecs and it may change the default codec used with a server. All the more so as now codec priority list taken into account is the one on our side (even if it should not choose something that is not on the remote party list).

And the error code "Unable to create media session: Invalid media payload type (PJMEDIA_EINVALIDPT) [status=220102]" Usually mean that the other side has accepted our codec but actually send something different to us. (Something with a different RTP payload type and so usually a different codec).

(589 is broken it should be valid from 590 and upper), but does not seems to be your problem.

Comment #4

Posted on Jan 24, 2011 by Happy Monkey

Yes, it now looks to me like a problem in Asterisk. I am attaching an excerpt from the log, the key point is when Asterisk sends ACK, there is this line in the SDP packet:

a=rtpmap:103 speex/8000

So Asterisk apparently agrees to use codec 103 which is Speex/16000, but "thinks" that the number corresponds to Speex/8, and probably uses Speex/8000 in the media session. With version 585, Speex/8000 is used and correctly identified as 102 by both parties.

I think this can be closed (but good to know the nature of the problem anyway).
