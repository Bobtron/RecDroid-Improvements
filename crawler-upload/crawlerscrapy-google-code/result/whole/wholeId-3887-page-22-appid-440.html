csipsimple - issue #911

Keep-alive interval not being honored with TCP

Posted on Apr 27, 2011 by Massive Monkey

What steps will reproduce the problem? 1.Set keep-alive interval (wifi or mobile) to, say, 600 2.Set account protocol to TCP (not sure if this is relevant) 3.Let account register

What is the expected output? What do you see instead? I see \r\n\r\n being emitted every 90 seconds (precisely 90 seconds if phone is awake, much more erratic if phone is asleep) I expect to see \r\n\r\n every 600 or so seconds.

What version of the product are you using? On what operating system? This was observed with the current market version (0.01-01) and with nightly build r807. This is on a Nexus S running Android 2.3.3

Please provide any additional information below. I also tried setting the keep-alive interval to 0 in the hope that this may disable keep-alives altogether). It made no difference. I have to use TCP because my carrier times out UDP NAT entries very, very quickly. However, 90 seconds is way smaller than necessary (more like 900 is sufficient) and kills my battery.

Also would like to point out that by setting TCP keep-alives on the server/proxy side (as I do) it is possible to do away with the need for application level keep-alives. So it would be useful if it was possible to either totally disable keep-alives or set the value to something very high (e.g. 3600).

Comment #1

Posted on Apr 27, 2011 by Quick Hippo

With latest nightly build, can you try to collect logs. (see the wiki page about how to do). I'd be interested to see if it could comes from CSipSimple or directly from pjsip.

Actually, when using TCP, the pjsip sip stack should even not use this keep alive since it's useless to send keep alive when using TCP (TCP already ensure itself reliability of the connection unlike UDP where it's needed to send blank packet to wake up routers). However in android, in order to ensure keep alive is used I did some hacks to ensure CPU is up and running when sending the blank packets (there is things in the SDK to do so). So if it comes from CSipSimple (with nightly build), it should not be erratic even with phone asleep. If it comes from the pjsip stack, it could indeed be erratic and would mean there is something I missed in my keep alive hack for the TCP case. I highly tested the UDP case to ensure it really work regardless cpu status, but never really checked the TCP case. So if you can collect some logs I'd be interested to see whether it could come from the android part of the native sip stack .

I do not need logs, I know why you get that. It's cause pjsip (the sip stack that pjsip use), do keep alive for TCP transport differently.

The nightly build will include a fix for this problem. It disabled the way it was done inside pjsip and use the way it is done in CSipSimple for TCP.

I'm not really sure about potential regressions it could introduce. So if while testing you observe strange things, do not hesitate to report. I'll try to do more test to ensure it does not introduce unwanted side effect.

It was safe for UDP because already do that externally to the transport but for TCP my fear is that it could conflict with the implementation of the TCP transport.

For reference, it should also fix the same problem for TLS.
