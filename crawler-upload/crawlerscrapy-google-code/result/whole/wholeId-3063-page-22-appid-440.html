Comment #5

Posted on Jun 8, 2012 by Quick Hippo

While trying to reproduce your problem on my side without logs, I noticed that there is some possible issue if the sip provider uses RTP dtmf events only. My current implementation only works with analogic dtmf generation in band (because it's easier to manage extra pauses inside the generation process).

I've something that could do for RTP too, but means that the extra pauses after the first pause will be ignored.

Comment #7

Posted on Jun 9, 2012 by Quick Hippo

Ok, sip info was also affected by the problem. I guess that voip.ms supports in band generation too, but only when a lossless codec (such as pcmu/a) is negotiated. Which could explain difference between devices. The changes of revision 1598 will help with this particular problem, but that's not enough to close this issue because the case where there is "," after the first comma is still not managed properly.

About log collection, if you use the instructions of HowToCollectLogs wiki page, it produces an email with the to already set to the support email address. This way is better than publishing your logs on the issue list because it keep your information private. That's why I didn't asked google to increase the storage quote of the issue list : I prefer to prevent users to put their logs directly on the website available to anyone. Even if logs don't contains any password, but it contains SIP addresses and I consider it's a potentially private information you don't want to share with anyone. Sharing with the support email address is safe cause I don't reuse these addresses for anything, I just try to analyse logs ;).
