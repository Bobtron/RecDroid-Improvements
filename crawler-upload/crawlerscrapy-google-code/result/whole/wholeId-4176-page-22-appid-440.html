Comment #9

Posted on Feb 12, 2011 by Quick Hippo

No unfortunately this kind of issue is very hard to debug and to say just for logs.

In this kind of problems, that's due to some ~bad~ (not standard regarding the android platform) implementation from the manufacturer. In this case, usually the first thing I do is testing all existing acks. In your case Motorola consider clearly something wrong, say that audio mode "Normal" should always route to speaker. Which is a very bad idea for voip apps.

Did you try "Use MODE API" setting? I guess that you'll observe a different behavior. When manufacturer consider that mode "Normal" should route to speaker, they also consider that mode "In_call" should route to earpiece. In this case the Option in csipsimple "Use Mode API" use mode normal and in_call to route or not to the speaker. However it could become worse. Some manufacturer consider that in "Mode IN_CALL" applications should not receive sound from micro... In this case they directly plug micro to GSM chipset... which is fine for telephony app... but make things impossible for VoIP app.

So if you could give me feedback trying to play with mode API and other audio hacks (see the audio troubleshooting section in the FAQ), it could help to understand what is the approach of motorola for this issue.

If "mode API" does not help, test with other SIP clients (specially opensource ones). If you get one working with it let me know I could have a look. But if none works with it... you should fear that none of the SIP application will succeed in supporting this device... and cry to motorola...

(Crackling sound could be due to garbage collector - which I'll eliminate in Gingerbread release ;) ... but not sure xt502 will be updated :( )

Comment #11

Posted on Feb 14, 2011 by Quick Hippo

1 -> cool :) I'll add autodetection of this one azap.

2 -> I guess they use the same thing than me ;)... Unfortunately that's due to mistakes (or misunderstood) from manufacturers... coders are really disappointed about that. Fortunately 2.3 will lead to more unity cause manufacturer will be obliged to do something good, else 2.3 stock sip app will not work.

3 -> Garbage collector is some java feature. As memory is released dynamically in java, frequently, a Garbage collection is launched. During this process it blocks the entire processing of other stuff and collect unused allocated memory. It's linked to the fact memory is managed by the JVM. Music player and voice calls is managed entirely on the native stack. My new driver using OpenSL is not affected by that cause it is entirely done in the native stack. My old driver I had, based on a private API of android (which is quietly exactly the same than the one used by the stock android 2.3 app) was also not affected. But unfortunately as it was based on private API it was not portable at all. As consequence : 4 -> you need to run a 2.3 device (so updating your device) and the ginger build of csipsimple available on nightly build website in the folder "ginger".

5 -> Be careful about "hacks" some introduce less stability. For example some could increase Garbage collector time and as consequence affect performances of VoIP apps, but helps for other performance stuff (benchmarking, battery maybe...)
