csipsimple - issue #1769

High audio latency for sound output

Posted on Jun 1, 2012 by Swift Panda

What steps will reproduce the problem? 1. Make a VoIP call or run a local audio echo test 2. 3.

What is the expected output? What do you see instead?

The latency for audio output is very high (on my device about 300-500ms), which makes it quite useless for interactive voice communication. BTW, the input from the microphone gets captured and processed without any noticable delay!

From what if read, the problem mainly is on the OS/kernel side, because audio buffers are too large. But csipsimple should try to deal with this OS issue as good as possible. One suggested workaround is to use stereo output instead of mono output, which should decrease the latency by 50% (since audio buffers have the same size for mono or stereo output).

What version of the product are you using? On what operating system?

Latest trunk with OpenSL-ES

Comment #1

Posted on Jun 1, 2012 by Quick Hippo

Yes android is one of the biggest problem.

In fact both OpenSL-ES and java/jni implementation relies finally on the "android media server" layer that by design introduce latency. Google engineers promised that the OpenSL-ES layer (implement in libwilhelm) will rely on libstagefright in the future. But for now that just rely on native part of "android media server".

So until google actually do this job, we have to deal with the fact there is a big latency on android by design for third party applications.

Your tip to use stereo sounds interesting, but I don't really get it. Why it should reduce latency. Is it not the same to increase clock rate ?

BTW, I never tried to do advanced tweaking on pjmedia jitter buffers. Maybe there is something to find there. Maybe my audio glue is not perfect too; well if I look at the one of webRTC made by google it's not so different so I'm probably not completely wrong, but I may miss something.
