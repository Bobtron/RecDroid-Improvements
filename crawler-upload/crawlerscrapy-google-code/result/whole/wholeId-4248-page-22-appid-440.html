Comment #8

Posted on Jan 23, 2011 by Quick Hippo

In revision 590, I introduce support of keyUp for end call button. Seems to be the only thing that can be caught by an application.

Tested on my galaxy i550 : when you press the first time screen goes off. when you press second time and release screen goes on and hang up. Behavior is weird but seems nothing better can be done. Tested on sipdroid, they do exactly the same thing and result is the same behavior (not surprising : same causes same effects ;) ).

Not absolutely sure that's better... cause the bad side of having that is that if screen is actually off on a phone without proximity sensor and user that own a phone with only a poweroff/endcall button will press on this button (for example to see elapsed comm time).. and hang up... not sure that is what is expected.....? Is it?

Regarding what I've read on the stock telephony application what they do is always done on keydown event (but the phone application has more rights than a third party application so that they can do that). As consequence I think that the "official" behavior is to => hangup if screen is on and hangup/power button pressed and => wake up screen if screen is off and hangup/power button pressed. However that's not technically possible for a third party app. So what is better. Using the workaround that may lead some users to hangup calls when they actually don't want... Or not using the workaround that will disable the hangup key button.... Or add an option to let users choose what they prefer?
