Comment #1

Posted on Mar 31, 2014 by Quick Hippo

Actually realm could be "*" (which means = ok with whatever server challenge with). Realm is more a concept of authentication than of access. Maybe default domain should maybe be the same than your sip AoR one rather than the one of the proxy. Would this work for your case ?

P.S. : for short term, as a workaround, csipsimple allows you to specify for each call the target domain. To do so: go in "txt" dialer mode (press txt button on dialer screen) and enter the @nsn.com at the end.

Comment #2

Posted on Mar 31, 2014 by Quick Hippo

Reviewing code default should be registrar first and proxy as fallback. It's more logical regarding what I expected at the time of coding : I was considering that you'd like to contact somebody on same domain than the one you register to.

However, using AoR (sip id) domain as very first choice would be even better : rather than using the sip server user register to, it will use id the user register as. I still consider that realm is not a valid option as linked to credentials. To get convinced : you might have several credentials in sip, even multiple layer of credentials that would lead to multiple realm -- for example it's theoretically possible to have one challenge from the proxy with one realm, and one challenge from the registrar with another realm. It never happened in current real deployment I found -- and that's why csipsimple allows you to enter only one credential and realm for now; but it's possible in SIP protocol and might be allowed in csipsimple in the future if somebody needs it. That's why I consider this as a non valid solution, but if you have good reason why you thought I'm wrong, please tell. I'm not totally sure of what should be the good one : normally, just like with email, SIP users should normally always enter a domain in the address they call. The way we dial in phone dialers, is more something that was standardized in SIP later as the "tel:" scheme. BTW, the best geek solution would be to have a dedicated option as you suggested to set the default call domain. But that would be only something available in expert mode and few mainstream users will care about it. So I would rather try to find the best "default" domain and then, if still necessary to add some extra option for very special cases, lets add it. But if we succeed in something that pleased everybody automatically, it's better ;)
