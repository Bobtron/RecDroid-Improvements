csipsimple - issue #1239

Reorder filters priorities by accounts / Handle filter conflicts/intersections

Posted on Aug 29, 2011 by Quick Elephant

What steps will reproduce the problem? This scheme allows transparent calling with no account selector UI appearing at all: 1) one ambiguous filter "Dial directly"/ALL for the dialer handling most of the calls. 2) several "Dial directly"/RegExp? exception filters for multiple SIP accounts for specific calls.

What is the expected output? What do you see instead? What currently happens: the dialer generic filter overrides all the SIP specific filters. I have to create a bogus Dialer/SIP "Can't call" filter for EACH specific SIP "Dial directly rule" making the whole filters logic ineffective and duplicated.

What I expect: more specific filters should take priority over more generic filters OR I must have an option to reorder the way filters are processed by accounts (e.g. making Dialer filters affect the last).

What version of the product are you using? On what operating system? CSipSimple 0.02-03 r944 (Market) Android 2.3.3

Comment #5

Posted on Aug 29, 2011 by Quick Hippo

Filters will apply according to account priority (for account re-ordering it is in account list, press menu (this time yes ;) ), and reorder).

BTW, making something more complicated would be overengenieering and will lost mainstream users. I usually try to focus on mainstream users and try to add hidden features for expert users. But when there is a conflict between what is better for mainstream users and what would be cool for 0.1% of users, I priviledge mainstream users ;).

About 3-rd party dialers, yes, indeed, that now managed through plugins. http://nightlies.csipsimple.com/plugins/ (Preivously you was probably unable to use SIPNET entry cause nothing where done in csipsimple to manage that). Now SIPNET app (or a third party app) can do the integration which is much more clean.

About echo mode, yes indeed, now default is WebRTC echo canceller. It is much more better than speex and simple and can run low cpu devices. (WebRTC is a project from google). So yes, indeed it change the default value, but it should be by far better. Let me know if not. I'm pretty interested in feedback about that.

Comment #7

Posted on Aug 29, 2011 by Quick Hippo

Ok, I guess the root of the problem is that you are setting filters directly from sip account view.

Try to go in settings > filters. You'll see the "Use GSM" option at the bottom. In your case just add a "Force call" rule on it.

About implementation, it is not as easy as you think. If you propose a patch for that it will be welcome ;). About the logic, it would mean having an order of accounts in filters and another one in account list... not sure that will be easy to understand for a mainstream user. You are a power user, so you even understand how things works behind, but you'd be amazed of how a mainstream user use an application.

Anyway, you are absolutely true, this feature is not documented enough. There is a wiki page about that but that's not enough yet. Unfortunately I have very very few time for developping, and less for documentation ;). This is an opensource project I do only on my free time (nights and holidays), so if you want to contribute to the wiki it will be welcome :D

Ho, and about mic and echo canceller, very weird. Did you try the new micro source IN_COMMUNICATION?

There is also an option to boost (by software) the micro level. It can also help. The weird thing is that I have not this behavior on my SGS1 (well with a cyanogenMod -- which is by far better than samsung roms ;) ). Echo canceller can indeed reduce a lot micro source, but only in cases the echo is high. Samsung device produce a lot of echo by form factor (and materials).
