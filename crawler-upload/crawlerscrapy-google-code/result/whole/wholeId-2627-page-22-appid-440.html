csipsimple - issue #2198

Exiting settings should unregister active SIP accounts before re-registering them

Posted on Jan 25, 2013 by Helpful Cat

What steps will reproduce the problem? 1. Register a SIP account 2. Go into the settings <menu> | settings 3. Exit settings

What is the expected output? What do you see instead?

When the preferences activity finishes the SIP stack is restarted, however any active accounts are not unregistered before they are re-registered. This can result in the SIP UA not being allowed to register when the SIP provider restricts the number of active SIP registrations.

What version of the product are you using? On what operating system?

CSS r2094 CyanogenMod 10

Please provide any additional information below.

This patch may suffice as a solution:

Comment #2

Posted on Jan 26, 2013 by Helpful Cat

You are correct that CSS tries to unregister active SIP accounts, but it fails to handle unregistration when the SIP router challenges the unREGISTER message.

So the sequence that I see is this;

1) CSS sends one (1) REGISTER message with EXPIRES==0

2) The SIP router replies with 401 Unauthorized

3) CSS sends a new REGISTER message with EXPIRES != 0

I've emailed the logs to [developers at csipsimple dot com]. The CSS log starts at the very moment I exited the preferences activity as well as the SIP router log captured on our SIP router.

Comment #6

Posted on Jan 26, 2013 by Quick Hippo

Oh about the AUTH_HEADER_CACHING you're right it's already enabled. I missed that your server send qop="auth" (which explicitely ask the sip stack to not cache).

For the destroy method, it's indeed the right place where things are managed. In fact the flag comes from : http://www.pjsip.org/docs/latest/pjsip/docs/html/group__PJSUA__LIB__BASE.htm#gafbe98339390b11b2e248d9420a2ed499 (pjsua_destroy2).

If we want RX to be managed, 0 instead of 1 is indeed a good value. If think we can keep 3 (aka no network usage) in case of connection no valid for outgoing as we don't want any network usage in this case. So having "0" will probably still delay if network is connected (from android point of view) but sip server not reachable, but maybe it's something affordable. Actually I didn't noticed it doesn't take into account the case where qop requires authentication to be done each time, but it sounds somehow more critical than delaying a little bit the destroy of the stack. BTW, actually the problem is not really the restart (for example when going in settings), thanks to the latest changes that automatically disconnects on next REGISTER by adding a Contact header with expires=0. But more about the real disconnections that should occurs when switching from wifi to 3g (with 3g available as a sip network) or if users manually disconnect using the "disconnect" dialer option.
