csipsimple - issue #2679

One way voice

Posted on Apr 4, 2014 by Swift Dog

First off all I would like to thank you for great SIP app. Nevertheless in some cases behind WiFI Router - NAT, we detected problem with one way voice. At the end we analyse that the problem is because of Router behaviour but there could be a workaround within SIP session handling.

Short description of a problem: When call is made behind NAT network with CSIPSIMPLE app. in some cases we have one way voice. (A user (calling party) does not hear B user (called party). Call flow description: A user --- Invite (SDP RTP port50022)----------> SIP proxy --> B user A user <---100 Trying ------------------------ SIP proxy A user <---183 Session Progres (SDP RTP port 7444)-- SIP proxy A user DST port 50022<-------RTP-------SRC port 7444 A user SRC port 1024 -------RTP------>SRC port 7444 ---A user does not hear ringing! A user <---200 OK (SDP RTP port7444)-- SIP proxy A user SRC port 1024 -------RTP------>SRC port 7444 A user DST port 50022<-------RTP-------SRC port 7444 ---A user does not hear B user, B user hears A user .....

Described problem is because of Router! Call flow of SIP application is ok! Nevertheless to overcome this problem SIP application should disable early media. For example pure "SIPapplication" integrated within android V 4.2.2 works also behind such routers because early media is not supported. Ringing is played internally (as wav file) after 183 Session progress SIP message, the same behaviour as after 180 Ringing SIP message. With this solution we would have in most cases voice in both direction, and we lose early media what we do not see as a problem.

Call flow description to solve one way voice problem: A user --- Invite (SDP RTP port50022)----------> SIP proxy --> B user A user <---100 Trying ------------------------ SIP proxy A user <---183 Session Progres (SDP RTP port 7444)-- SIP proxy ---A user hear ringing- plaied internaly within SIP app

A user <---200 OK (SDP RTP port7444)------------ SIP proxy A user DST port 50022<-------RTP-----------------SRC port 7444 A user SRC port 50022 -------RTP---------------->SRC port 7444

---VOICE in BOTH Dirrection--- ...

Tests made on several devices,...one of them; Android version 4.2.2, Model Number: GT-19082

In case there is any configuration option where we could do some corresponding changes to overcome this problem please let me know. I hope we could find a solution.

Hi,

Can you copy paste the SDP message. When debugging this issues, SDP exchanged between apps are very very helpful. From your description I guess you have good skills in tracking packets and will find sdp easily in sip flow, but if you don't know how to you can simply follow guide : http://code.google.com/p/csipsimple/wiki/HowToCollectLogs?wl=en

Else about the problem it's usually just because of SDP that does send a private IP and nothing in the middle able to handle it. By default csipsimple does not enable STUN (which tries to resolve your public ip port). Many other clients enable this by default. In csipsimple I think it's not good to enable it by default for several reason (does not work on all countries, and when you use a private network only). So, might be helpful to enable it. If your two endpoints supports it, ICE feature is also the killer one for these situation. In the SDP, instead of having only one ip:port, it allows to send several candidates : those resolved with stun server and those on private network. The other endpoint will try all these candidates. You can read more on this FAQ entry about that : https://code.google.com/p/csipsimple/wiki/FAQ?wl=en#The_other_party_can_hear_me_but_I_can_not_hear_them
