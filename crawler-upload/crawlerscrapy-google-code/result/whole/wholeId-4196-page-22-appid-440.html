csipsimple - issue #628

armeabi (not v7a) and echo cancellation support

Posted on Jan 26, 2011 by Happy Rhino

Although r590 is much better for battery life, it's unfortunately unusable for me due to severe problems with audio during calls. Sound quality is poor and occasionally the CPU gets pegged and the phone stops responding until the call hangs up.

The main thing I notice that's different is the codec selection. Previously my calls used either PCMU/PCMA 8KHz and now they all seem to be using GSM, even though I disabled it in the codec list (both narrow and wideband lists, but only wide is relevant since all calls are over wifi). It seems the codec list setting has no effect on the codec that actually gets used.

I don't know if this is only a codec selection problem or if something else is causing audio problems - I would have thought this phone could handle the GSM codec but even with the CPU clock at 633MHz it dies on r590 (audio clock rate set to 8KHz).

Comment #6

Posted on Jan 26, 2011 by Happy Rhino

Just did some simple tests with the ekiga echo test number and PCMU. Disabling echo cancellation makes a huge difference in CPU use: with it enabled, the csipsimple process runs at 50-55% CPU, and with it disabled, it runs at 15-25% CPU. Even increasing the audio clock rate from 8KHz to 16KHz only adds about 5% average CPU.

Also, although calling the echo-tester isn't conclusive, it seems like call quality is actually better without the echo cancellation. There is a bit more echo with it turned off, but with it enabled there's a nasty "screech" audio artifact that happens when you start speaking or say something suddenly at a higher volume. This seemed to be the same with both the default "simple" and the "speex" algorithm.

With the questionable impact on quality and the major impact on CPU, I think I agree with disabling it by default. :) I will do some more testing though.

Comment #14

Posted on Jan 28, 2011 by Quick Hippo

Ok got it. The weird thing is in pjsip. It does not take into account the echo mode when starting the stack. As consequence it always use the default (for pjsip), which is in latest dev version speex cause I've enabled it in pjsip. The default for CSipSimple is Simple echo mode but since was not taken into account by pjsip .... it used speex AEC...

So now r597 (which will be on nightly builds server soon, I've launched a manual build), will force the choice of the echo canceller option when call is established (this method seems to be valid :) ).

I thought previously that it was fixed cause I used the option in media while in call to test to deactivate/reactivate echo cancellation which use the good method. And once it's done one time it's keep in mind by pjsip... reason why I thought it was fixed ... even if what I changed was absolutely not related.... :/

So for armv6 devices, should now be fine with simple echo canceller (but you should never activate speex echo canceller cause it will be efficient only with armv7 devices).

Could you confirm? Thanks in advance for tests.
