csipsimple - issue #777

Feature request: Only try connection on specified wifi aps

Posted on Mar 8, 2011 by Grumpy Elephant

Hi,

for company use, it might be nice to be able to tell csipsimple to only try and login to a pbx, if the wireless is associated with certain specified APs (like your company wlan which is not reachable from "outside").

This might save lots of battery...

Comment #1

Posted on Mar 8, 2011 by Quick Hippo

Please read my comments on issue 410.

The idea is very good, but as I said the best solution to address this issue is not to do it in CSipSimple itself. I really believe in the android approach where all apps acts together to reply needs of users. Each apps should allow other apps to extends its features and what you ask for is exactly on case where it should be done outside from the app cause : * it keeps the app simple for those how does not need it * it's SO MUCH powerful than the same feature inside the app.... just imagine... infinity of plugins could acts on many other specific use cases.

Each time somebody ask for a feature like this one I'm really sad with the fact there is not yet an open platform for addressing such needs with input plugins and output plugins. Such a platform would be an excellent way to save battery and not only for a SIP application... for almost everything on the android OS.

Just also for info, read issue 267, issue 586.

Comment #5

Posted on Jan 10, 2013 by Quick Hippo

Openintent actually doesn't address the Locale/tasker like features. It's a repository for intents only. Not an engine with "plugin conditions" and with "plugin settings". But such an app could take place inside OpenIntent projects.

About the why limiting to wlan, you're right. It should normally be also addressed by a third party app. But in it's simple form it's some simple feature that mainstream users can understand and that 100% users use. So makes sense inside the app.

Something that address "time range", "wifi ssid", "3G antenna id", "location", "have I some google calendar event", "what the weather like today",... conditions it's something that less users will need and that more users will get confused about if inside the main app. I voluntarily keep the lane between core app and what could be plugin very high because I see a lot of possible features if done in an external app.
