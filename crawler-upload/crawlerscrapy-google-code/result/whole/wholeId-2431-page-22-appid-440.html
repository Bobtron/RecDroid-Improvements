csipsimple - issue #2373

Make User-Agent header optional (privacy and RFC conformance)

Posted on May 21, 2013 by Happy Horse

Right now, the general purpose User-Agent header sent (i.e. not overridden by a provider) gives off a pretty intense fingerprint by listing the application, build number, and even the device used. This allows SIP and network providers etc. to link anonymous accounts.

Please make sending any User-Agent header at all a user-configurable option (RFC 3261 says it's optional and that "[i]mplementers SHOULD make the User-Agent header field a configurable option"). For OSTEL accounts, where users tend to care very much about privacy issues, maybe disable the header by default?

I don't have the build infrastructure at the moment to test if cfg.setUser_agent(NULL) in PjSipService.java would work...

Comment #1

Posted on May 21, 2013 by Quick Hippo

It's already the case from long long time.

Switch to expert mode (http://code.google.com/p/csipsimple/wiki/ExpertSettingMode?wl=en) for general settings. Then in Network > User Agent you can change the value to anything you want. If it's default (csipsimple) it also add device model + android version. If it's custom it takes exactly what you set.

BTW, try to not think in terms of internal API (pjsip) if you are working with the code. There is a documented csipsimple API that should be used as soon as it match expectation. And in your case it is : http://r3gis3r.github.com/SampleCSipSimpleApp/javadoc/com/csipsimple/api/SipConfigManager.html#USER_AGENT If you try to hardcode something in the implementation for only one sip stack (even if for now there is only one supported) it's less easy to maintain than something that uses the upper API of csipsimple

Comment #3

Posted on May 21, 2013 by Quick Hippo

:)

It's a little bit hidden for mainstream users that might get quickly lost when they want very common features. But geeks (me the first) like to tweak and test more features and like to have a full control on what the sip stack does behind the scenes. That's the reason why I adopted this approach to have a different mode for who wants to tweak more, and try to keep the rest automatically configured from wizard they use + device the app detects + conditions that seems to be used. The goal is to have something that works out of box with minimal configuration for most users. But leave others users free to tweak :)

About your first remark to have the user-agent blank by default for OSTEL account. Well, it's technically possible, but I'm not sure it's necessary to change that by default. As soon as you use OSTEL account your path between you and the sip server is supposed to be secured and you trust the OSTEL servers. Besides, knowing the user agent version can be very helpful to debug an issue on server side. So, I'm not sure it should be the default for this wizard, but I keep my mind open to good arguments ;).
