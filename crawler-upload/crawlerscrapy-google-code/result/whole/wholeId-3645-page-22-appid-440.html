csipsimple - issue #1200

Takes too long (~20seconds) in sending the INVITE with "Proxy-Authorization" header after 407 (because it tries TCP !!!)

Posted on Aug 10, 2011 by Happy Cat
  • Samsung Galaxy Mini S5570.
  • Android 2.2.1
  • CSipSimple r944
  • Asterisk 1.4.24.1

The CSipSimple account is configured using "Basic" profile in which "Use TCP" is dissabled.

Scenario:

Mobile conected via WiFi. Call sent to an Asterisk server in Internet (good Internet connection). Using SIP over UDP.

Asterisk replies 407 and CSipSimple sends the ACK.

Then CSipSimple takes ~20 seconds (always!) in sending the new INVITE with credentials. I attach CSipSimple_takes_21_seconds_for_second_INVITE.txt showing a real trace.

Also, during these ~20 seconds, pressing "Hangup" button does nothing at GUI level (bad user experience). This is a separate issue but important IMHO. The user GUI should not be affected due to lower layers issues. When the user presses "Hangup" to cancel/terminate a call, the GUI should just work as expected (no matter if internally CSipSimple remains waiting for a provisional response before sending the CANCEL or trying to make the TCP connection or whatever).

I've realized that the problem occurs because after the 407, CSipSimple tries to open a TCP connection with the Asterisk server! (even if "Use TCP" is dissabled). The iptables in the server do "DROP" rather than "REJECT" so CSipSimple makes 3 TCP connection attemps and after N seconds tries again using UDP.

The question is: why is it trying TCP for the second INVITE?

NOTE: Repeating same test but using a Kamailio server listening into UDP and TCP, I can confirm that first INVITE uses UDP and second one (with credentials) uses TCP. In this case the CSqipSimple account is made via "Expert" profile and "Transport" is set to "auto". There is no NAPTR record for the proxy domain but there is SRV for both SIP over UDP and TCP.

Comment #1

Posted on Aug 10, 2011 by Happy Cat

By reconfiguring the account using the "Expert" profile and setting "Transport: UDP" (rather than "auto") CSipSimple just tries UDP and things work correctly.

So the problem seems to be that, using the "Basic" profile and setting "Use TCP" to dissabled value, it still tries TCP (and there is no SRV record with TCP and so, even less taking into account that I set an IP as proxy hostname rather than a domain).

Comment #2

Posted on Aug 10, 2011 by Quick Hippo

Yes indeed. "Use TCP" option should never be disabled. It disable just the creation of the transport. (it's an option in expert settings mode and it only impacts some stuff in pjsip stack).

The good way to do if some wants to prevent an account from using TCP is to switch the account to expert mode and use Force UDP. If set to auto, it will try to use TCP if UDP packets size is too big. I guess you previously changed that on this account, cause by default a basic account use default settings (Force UDP). If you did that in expert wizard and then switch back to basic wizard, the basic wizard will not change this setting. It will keep settings as set in expert mode and only modify those that are relevant of basic configuration if you save again with basic wizard.

So what you observe is expected. Great power that comes with expert wizard comes with great responsibility ;).
