Posted on Jan 19, 2012 by Quick Hippo

Hi Luke,

This kind of problem is often obtained when the route is not maintained by the sip server.

In fact, in sip once the remote sip side has been found by the registrar server, the sip signal can pass directly as a peer to peer signal. This is the case of the "confirm" signal sent by the application when it wants to confirm that the communication has been established. The remote side should send a "ack" in this case that the communication has been established. From this point the application change the background from grey to green.

I think that in your case the remote side ip can't be accessed from the mobile. In consequence it sends the confirm to the remote side, but the remote side never replies the "ack". As consequence, as specified in sip protocol, the application will consider that the remote side is dead. And will close the connection after 30 sec if no replies.

My guess is that is what you get.

Then how to solve the problem? Well, usually that's about configuration. It could be because of the fact the server is not really fine with NATed topology or because the configuration of the server on csipsimple does not match what is expected by the server.

Anyway, what could be helpful to start to understand where to look at is to collect some logs. See HowToCollectLogs wiki page and send me the logs you get when you place an outgoing call. Also if you can indicate me how you configured the account : basic wizard, advanced wizard, expert wizard?, did you entered a sip proxy -- this could particularily help in your case -- ? Did you activated the stun or ice options (this usually helps to getting voice path, but could also sometimes help with communication signalization)

Also if your pc softphone is able to collect logs it could be interesting to compare with their settings.

Comment #12

Posted on Jul 16, 2012 by Quick Hippo

Well, thanks to topcat logs, we were able to find the root cause (at least of his issue)

The following is a little bit technical -- it's just for me and for the record ---

It seems that the SIP server used in this case, reply the ACK with lower case tag. This is not a good idea from server because it leads to be incompatible with some sip client implementation that compares and check with case sensitivity. And it's the case of pjsip stack.

However, here seems that regarding the RFC this should be possible (cf https://lists.cs.columbia.edu/pipermail/sip-implementors/2005-August/009972.html ). So it seems to be something to fix in the SIP stack for robustness even if the idea of the sip server (in this case Sippy) to transform the tag is not very good.

I was able to produce a build that doesn't have the problem by generating only lower case tag and was tested by topcat. However not suitable for production fix. So I asked to pjsip maintainers what would be a correct fix for that (and if should be fixed or not... maybe the statement in the RFC is obsolete per another RFC or maybe there is a good reason to not compare case insensitive).

Depending on the reply we will have a fix or an argument to send to SIP provider to not transform case of the tag. (if needed I'll directly contact sippy developers so that the fix is done at the root thing that produce this case change).

Comment #16

Posted on Jul 31, 2012 by Quick Hippo

@igor : regarding your logs it seems a different problem. The server (for some reason), doesn't accept the re-invite that tries to fixes the codec. And instead of ignoring it ... the server hangup with a BYE "Protocol error, unspecified"

It's a problem on server side that doesn't support correclty what's made by the application to select the codec.

A workaround would be to enable only one codec in the application settings (so that no re-invite will be needed to fix the codec). To do so, go in settings > media > codecs and deselect all but one codec. Regarding the reply of the server, my advise on the codec you should select : If you are over wifi use PCMU as the only codec (in fast network tab). If over 3G use only GSM as the only codec (in slow network tab).
